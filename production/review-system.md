# Review System

Use this file when a plan, feature, config model, architecture change, release candidate, or operational decision must be reviewed against studio standards before acceptance.

## Review law

- Review is a gate, not encouragement theater.
- Findings come before summary.
- If a blocker exists, do not hide it behind style feedback.

## Review axes

- Stage fit
- Ownership fit
- Classification fit
- MVP fit
- Architecture fit
- Platform and release risk
- Operations and business impact
- Risk escalation need

## Review output rules

- List findings first.
- Name severity.
- Name what must change before approval.
- Separate blockers from follow-up improvements.
- If no findings exist, state residual risk honestly.

## Review record

```md
# Review Record

## Item Under Review

## Review Type

## Findings

## Blockers

## Residual Risks

## Approval Status
- approved
- approved with conditions
- blocked
```
