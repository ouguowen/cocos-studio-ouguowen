# MVP Prototype Rules

Use this file when the team is defining the first playable scope, cutting features, or deciding what must be proven before vertical slice.

## MVP law

- An MVP proves marketable play, not complete ambition.
- An MVP must ship one trustworthy core loop before supporting loops.
- An MVP without kill conditions becomes a slow-motion production trap.

## Required decisions

- Name the one session fantasy in one sentence.
- Name the one loop the player repeats in the first ten minutes.
- Name what the MVP must prove:
  - fun
  - clarity
  - retention hypothesis
  - monetization hypothesis
  - production feasibility
- Name what is explicitly out of scope for MVP.
- Name what quality bar is intentionally temporary.
- Name the kill conditions that stop expansion.

## MVP scope rules

- Keep only one primary progression track.
- Keep only one main resource economy unless the second resource is core to the loop.
- Keep only one content production model at first.
- Keep only one primary failure state and one success state per session.
- Delay social, guild, ranking, events, and live-ops layers unless they are the product.
- Delay cosmetic polish systems that do not change the learning value of the MVP.

## Prototype vs MVP

- Prototype answers "is this loop worth pursuing?"
- MVP answers "is this first-version game worth building and testing with real users?"
- Vertical slice answers "can this team execute the product at production standard?"

## Red flags

- "We need five modes to see the real fun."
- "The shop will make the loop work later."
- "We will balance it after building all content."
- "The tutorial needs the whole meta system first."
- "This feature is small" repeated across many edge systems.
