# Prompt Library

Use this file when the team wants consistent, high-signal prompts for recurring AI-assisted work.

## Prompt law

- Prompts should encode stage, owner, constraint, and expected output.
- A prompt library standardizes recurring asks; it must not replace review.
- If a prompt hides authority or scope, it will create convincing garbage faster.

## Prompt patterns

### 1. Game classification

```text
Use $cocos-studio-ouguowen. Classify this game by dominant player loop, main content unit, progression shape, and monetization pressure point. Then route it to the correct game type template, MVP scope, and architecture template. Do not use theme as the primary classifier.
```

### 2. MVP definition

```text
Use $cocos-studio-ouguowen. Define the MVP for this Cocos Creator 3.x game. Name the one-sentence fantasy, repeated loop, must-prove items, explicit non-goals, and kill conditions. Cut support systems aggressively.
```

### 3. Risk escalation

```text
Use $cocos-studio-ouguowen. Evaluate whether this issue is a product, technical, production, release, or live risk. Assign severity, owner, escalation authority, and recommended response. Do not soften the conclusion.
```

### 4. Review request

```text
Use $cocos-studio-ouguowen. Review this plan or implementation against stage, ownership, architecture, release risk, and quality gates. Findings first. Name blockers explicitly.
```

### 5. Memory update

```text
Use $cocos-studio-ouguowen. Update the project memory with any newly locked facts, changed assumptions, accepted risks, blocked items, and current milestone truth. Separate confirmed facts from open questions.
```
