# Version Roadmap System

Use this file when the team is defining milestone layers, version purpose, release sequence, or the path from concept to soft launch and live operations.

## Roadmap law

- A version exists to prove or deliver one meaningful outcome, not to hold a random pile of features.
- Every milestone must answer "why this version exists now" before answering "what is inside it".
- If two adjacent milestones prove the same thing, one of them is probably fake work.

## Standard milestone ladder

1. Idea Framing
2. Gameplay Prototype
3. MVP Definition
4. Vertical Slice
5. Production Alpha
6. Production Beta
7. Soft Launch
8. Global Or Main Launch
9. Live Operations Iteration

## Milestone purpose

### 1. Idea Framing

- Purpose: decide whether the game should exist at all.
- Must prove: audience, loop promise, scope boundary, top risks.
- Must not contain: broad implementation.

### 2. Gameplay Prototype

- Purpose: prove fun and feasibility.
- Must prove: core loop readability, failure state, input-to-feedback rhythm.
- Must not contain: production polish theater.

### 3. MVP Definition

- Purpose: lock the smallest real product worth building.
- Must prove: what first-version game exists, what is deliberately excluded.
- Must not contain: uncontrolled support systems.

### 4. Vertical Slice

- Purpose: prove the team can build one product-grade slice.
- Must prove: structure, quality bar, representative player flow, production standards.
- Must not contain: half the full game hidden inside one milestone.

### 5. Production Alpha

- Purpose: achieve system and content completeness for the intended core version.
- Must prove: the game is broadly assembled and internally playable.
- Must not contain: fake completeness with missing backbone systems.

### 6. Production Beta

- Purpose: stabilize, tune, and validate release risk.
- Must prove: key flows are robust enough for serious release preparation.
- Must not contain: major late feature growth.

### 7. Soft Launch

- Purpose: validate retention, monetization, economy, and operations under limited real traffic.
- Must prove: the product is viable beyond internal optimism.
- Must not contain: untracked business hypotheses.

### 8. Global Or Main Launch

- Purpose: scale release under controlled confidence.
- Must prove: release readiness, support readiness, rollback readiness, operational watch readiness.
- Must not contain: unresolved launch-critical risk hidden behind deadlines.

### 9. Live Operations Iteration

- Purpose: improve product health through measured updates.
- Must prove: version-to-version learning, not random patching.
- Must not contain: metrics with no owner or no hypothesis.

## Version naming rules

- Name the milestone type.
- Name the business or production purpose.
- Name the commit point for entry and exit.
- Name the kill condition or promotion condition.

## Promotion rules

- Do not advance because the team is tired of the current stage.
- Do not advance because the backlog is large.
- Advance only when the current milestone's proof obligation is satisfied.
- If proof is partial, either extend the milestone explicitly or cut the milestone claim.

## Roadmap red flags

- "We are almost in beta" with unstable first-session flow.
- "Soft launch" with no retention or monetization hypothesis.
- "Vertical slice" that still rewrites core direction weekly.
- "Alpha" that is really three disconnected prototypes.
- "Live ops" that is just panic patching.
