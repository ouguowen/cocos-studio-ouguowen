# AI Game Studio v2.3 MCP Connection Proof Architecture

This document defines the first controlled MCP connection proof architecture for
AI Game Studio. It describes how AI Game Studio will validate communication with
external MCP Servers. It is documentation only and does not create runtime code,
implement an MCP client, implement Cocos automation, modify core files, or
modify real Cocos project files.

## 1. Goal

Define the first controlled proof of:

```text
AI Game Studio
      |
      v
MCP Adapter
      |
      v
External MCP Server
      |
      v
Cocos Creator 3.8.8
```

The goal is communication validation, not full automation.

## 2. Proof Scope

Allowed proof actions:

- connect to MCP server
- discover available tools
- read tool metadata
- send controlled request
- receive response
- record execution result

Not included:

- automatic game creation
- unrestricted Cocos editing
- autonomous Agent execution

The proof must stay narrow and stop before unapproved project modification.

## 3. Connection Architecture

Architecture:

```text
User
  |
  v
AI Game Studio
  |
  v
Agent Runtime
  |
  v
MCP Adapter
  |
  v
MCP Client
  |
  v
Cocos MCP Server
  |
  v
Cocos Creator 3.8.8
```

Responsibilities:

- User: approves scope, project path, and proof target.
- AI Game Studio: interprets the request, applies boundaries, and prepares the
  proof plan.
- Agent Runtime: future coordination layer for approved Agent state.
- MCP Adapter: translates approved intent into provider-neutral MCP requests.
- MCP Client: future transport layer for MCP communication.
- Cocos MCP Server: external execution provider.
- Cocos Creator 3.8.8: engine/editor surface that returns project or operation
  results.

## 4. First Proof Scenario

Minimal scenario:

```text
User request:
"Create an empty Cocos scene"
```

Flow:

```text
Planner Agent
      |
      v
Capability lookup
      |
      v
Permission check
      |
      v
MCP Adapter
      |
      v
Cocos MCP tool call
      |
      v
Cocos Creator response
      |
      v
Validation result
```

The scenario is a future proof shape only. Actual scene creation requires
separate approval and runtime implementation.

## 5. MCP Tool Discovery

Tool discovery should include:

- server discovery
- tool list retrieval
- capability matching
- version compatibility check

Example:

MCP exposes:

- `create_scene`
- `query_scene`
- `create_node`

Adapter registers:

- `scene.create`
- `scene.query`
- `node.create`

The adapter should normalize tool names without hiding provider-specific
limitations.

## 6. Permission Control

Approval checkpoints:

No approval, inside an already approved proof scope:

- read project information
- query available tools

Approval required:

- create assets
- modify scenes
- change project settings

Forbidden:

- destructive operations without confirmation
- unrestricted project editing
- autonomous continuation
- cross-project inspection

Permission must be checked before tool execution and before retry.

## 7. Execution Record

Audit information:

- request id
- agent id
- timestamp
- MCP server
- tool name
- parameters
- result
- approval status

The execution record must distinguish discovery, read-only query, write action,
validation, warning, and failure.

## 8. Failure Handling

Connection failure:

```text
Connection failure
      |
      v
Retry
      |
      v
Fallback
      |
      v
Human notification
```

Tool failure:

```text
Capture error
      |
      v
Preserve state
      |
      v
Generate recovery proposal
```

Retries must be narrow. Write or destructive retries require human approval.

## 9. Cocos Creator 3.8.8 Validation

After MCP operation, check:

- scene exists
- hierarchy correct
- components valid
- no unexpected project changes

Validation must report PASS, NEEDS_REPAIR, BLOCKED, or FAIL with evidence. A
successful MCP response alone is not enough proof when Cocos project state or
runtime behavior matters.

## 10. Security Boundary

No autonomous execution.

No unrestricted MCP access.

No automatic project modification.

Human approval remains required.

The connection proof validates communication only. It does not authorize broad
Cocos automation.

## 11. Future Roadmap

v2.4: First MCP controlled execution prototype

- connect to one approved MCP server
- run read-only discovery first
- execute one approved safe operation

v2.5: Agent generated Cocos scene workflow

- connect Planner output to a controlled Cocos scene proposal
- require approval before scene creation
- validate scene state after execution

v2.6: Game feature generation pipeline

- combine design, implementation, Cocos automation, QA, and runtime proof under
  strict approval and audit logging
