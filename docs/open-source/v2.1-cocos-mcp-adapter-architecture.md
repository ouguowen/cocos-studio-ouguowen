# AI Game Studio v2.1 Cocos MCP Adapter Architecture

This document defines the architecture of the AI Game Studio Cocos MCP Adapter
layer. It is documentation only. It does not create an MCP server, runtime
execution, autonomous modification, Cocos automation code, or real Cocos project
changes.

## 1. Architecture Position

AI Game Studio does not replace Cocos MCP Server.

Definitions:

- AI Game Studio = orchestration and intelligence layer
- Cocos MCP Server = execution layer

Architecture:

```text
User
  |
  v
AI Game Studio
  |
  v
Agent System
  |
  v
MCP Adapter
  |
  v
External MCP Server
  |
  v
Cocos Creator 3.8.8
```

The adapter should translate approved AI Game Studio intent into provider-neutral
tool requests. The external MCP server remains responsible for actual tool
execution.

## 2. Adapter Responsibilities

The Cocos MCP Adapter is responsible for:

- MCP server discovery
- tool capability mapping
- tool abstraction
- permission filtering
- request translation
- response normalization
- error handling

The adapter must not bypass human approval, project path scope, tool permission,
or rollback requirements.

## 3. Cocos Creator 3.8.8 Capability Binding

Capability categories:

- Scene
- Node
- Component
- Prefab
- Asset
- Animation
- UI
- Physics
- Build

Example mapping:

```text
Cocos MCP tool:
create_node()

Adapter:
game.scene.create_entity()

Agent:
Create enemy spawn point
```

The adapter maps Agent intent to a safe capability name before selecting a
provider-specific MCP tool.

## 4. Agent To MCP Flow

Future flow:

```text
Planner Agent
      |
      v
Task Router
      |
      v
Capability Registry
      |
      v
MCP Adapter
      |
      v
Cocos MCP Tool
      |
      v
Cocos Creator
```

Every step must preserve the approved task scope and stop when permission,
project path, tool capability, or proof target is missing.

## 5. Permission Governance

Allowed after approval:

- create scene objects
- query project state
- modify approved resources

Requires approval:

- delete assets
- change project settings
- build release
- modify scenes, prefabs, animations, metadata, or scripts
- access browser preview or runtime console

Forbidden:

- uncontrolled file deletion
- modifying core engine files
- bypassing Cocos asset metadata
- executing unapproved tools
- inspecting unrelated projects
- autonomous continuation

## 6. Multi Engine Extension

Future MCP support may include:

- Cocos MCP
- Unity MCP
- Godot MCP
- Blender MCP

Architecture:

```text
Game Engine Adapter Layer
```

Each engine adapter must keep engine-specific files, rules, validation, proof
paths, and runtime assumptions separate. Godot, Unity, or Blender support must
not enter the Cocos Skill core by default.

## 7. Context Strategy

Context passed to MCP should be minimal:

- Core context: safety, approval, path scope, and operation mode
- Task context: approved task, files, scene paths, asset paths, and expected
  output
- Project context: approved project metadata and current state
- Engine context: Cocos Creator 3.8.8 capability constraints and tool mapping

Do not pass proof history, archive docs, full project memory, or unrelated
examples to MCP by default.

## 8. Failure Recovery

Failure types:

- MCP unavailable
- tool execution failed
- invalid parameters
- permission mismatch
- unexpected generated files
- timeout

Recovery rules:

- stop the workflow
- report the failing MCP server or tool
- include logs and normalized error output
- list modified or generated files
- avoid retrying destructive operations automatically
- propose the smallest safe repair
- request human approval before retry when scope, files, or risk changes

Rollback strategy:

- rollback only approved changes
- preserve user changes
- avoid dangerous Git commands
- report rollback limitations honestly

Audit record:

- user request
- selected capability
- adapter mapping
- MCP tool
- approved scope
- execution result
- logs
- warnings
- rollback status

## 9. Explicit Limitations

This document creates:

- no MCP server implementation
- no runtime execution
- no autonomous modification
- no Cocos automation code

It also creates no runtime files, no Agent implementation, no protocol changes,
and no real Cocos project changes.
