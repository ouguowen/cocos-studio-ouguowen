# v1.8 Agent Runtime Architecture

This document prepares the future Agent Runtime architecture for AI Game Studio.
It is documentation only. It does not create runtime code, Agent implementation,
MCP execution, Cocos automation, autonomous execution, `agents/*` files,
protocol changes, examples, or real Cocos project changes.

## 1. Runtime Layer Positioning

The future Runtime Layer would coordinate approved Agent work after planning,
permission, context, memory, and validation systems are mature.

Relationship map:

- Planner Agent prepares task scope, capability match, context request, and
  execution proposal.
- Agent Workflow defines handoff, approval, failure recovery, and audit trail.
- Capability Registry identifies which capability can support a task.
- Context Engine selects the smallest useful context for the current task.
- Memory System provides approved long-term and task-specific project knowledge.
- QA Agent reviews validation, proof, diff scope, and blocker state.
- Runtime Layer would coordinate approved state transitions and task execution.

The Runtime Layer must sit behind human approval and must not become default
execution.

## 2. Agent Lifecycle Model

Future Agent lifecycle states:

- Agent creation request: a human requests or approves a proposed Agent role.
- Agent initialization: identity, capability, permission boundary, and stop
  conditions are loaded.
- Agent ready state: the Agent is available for an approved task.
- Agent working state: the Agent is performing the approved task.
- Agent waiting state: the Agent is waiting for approval, context, validation,
  memory, or another handoff.
- Agent completed state: the Agent produced the approved output and stopped.
- Agent failure state: the Agent hit a blocker, validation failure, timeout, or
  forbidden scope.
- Agent shutdown state: the Agent is retired from the current task and cannot
  continue automatically.

Lifecycle changes require audit logging and human approval when scope or
authority changes.

## 3. Agent State Management

Runtime state would track:

- identity: agent id, name, role, owner, and lifecycle status
- capability: selected capability and version
- permissions: allowed reads, writes, commands, memory, and proof channels
- current task: task id, intent, scope, expected output, and stop conditions
- context binding: context id, loaded sources, compression state, and conflicts
- memory access: approved memory categories, source, and write permission
- execution status: ready, working, waiting, completed, failed, or shutdown

Agent state must be task-local unless explicitly archived.

## 4. Task Execution Model

Future execution flow:

```text
User request
      |
      v
Planner
      |
      v
Task queue
      |
      v
Agent assignment
      |
      v
Execution proposal
      |
      v
Human approval
      |
      v
Execution
```

The task queue must not execute automatically. It may hold proposed work, but
human approval remains required before any high-impact read, write, runtime,
Cocos/MCP, Git, memory, or release action.


Messages must be explicit, task-local, and auditable. A message must not grant
new permission by itself.

## 6. Multi-Agent Coordination

Planner coordination:

- owns task decomposition and approval proposal
- prepares handoff packages
- stops before execution unless approved

Designer handoff:

- receives approved task scope
- returns design constraints and acceptance criteria
- stops if the request expands beyond scope

Coder handoff:

- receives approved implementation scope
- returns changed files, validation results, and blockers
- stops on unexpected files or forbidden paths

QA feedback:

- reviews proof, validation, diff scope, and safety boundaries
- returns PASS, NEEDS_REPAIR, BLOCKED, or FAIL
- does not approve new work

Retry mechanism:

- retry only the failed step
- require approval for scope changes
- keep prior outputs as audit history
- never continue automatically after failure

## 7. Runtime Permission System

Allowed actions depend on approved permissions:

- read named files
- summarize approved context
- propose plans
- prepare bounded changes
- run approved validation commands
- report proof or blockers

Forbidden actions by default:

- autonomous execution
- real Cocos project inspection
- Cocos/MCP use
- browser preview or runtime console use
- project memory writes
- file writes outside approved scope
- package, CI, release, protocol, runtime, or Agent file changes
- staging, committing, pushing, tagging, or releasing

Approval requirements:

- human approval before high-impact actions
- explicit project path approval before project work
- explicit file scope before writes
- explicit proof target before runtime validation

Audit logging must record approvals, actions, validation results, failures, and
rollback.

## 8. Failure Recovery

Failure types:

- failed agent
- timeout
- validation failure
- forbidden file access
- scope expansion
- missing approval
- context conflict
- memory conflict

Recovery rules:

- stop the Agent
- mark failure state
- report expected vs actual result
- identify smallest repair
- request human intervention when approval or scope changes are needed
- rollback only approved changes
- preserve user changes
- avoid dangerous Git commands

## 9. Runtime Memory Interaction

Agent reads memory only when:

- memory access is directly triggered
- active project path is approved
- memory category is named
- read scope is approved

Agent writes memory only when:

- write scope is approved
- source and owner are recorded
- conflict check passes
- rollback or retirement path exists

Temporary context is task-local and expires after the workflow closes.

Permanent project knowledge must be reviewed, versioned, and scoped to an
approved project.

## 10. Performance Strategy

Parallel execution:

- possible only for independent approved tasks
- must not share mutable state without approval
- must preserve audit order

Context loading optimization:

- load core context first
- load task context narrowly
- load optional context by trigger
- load memory context by approval

Token usage control:

- summarize old handoffs
- avoid full proof logs by default
- compress context while preserving safety and stop conditions

Resource management:

- limit active Agents
- timebox waiting states
- retire completed task state
- avoid background autonomous work

## 11. Future Implementation Roadmap

Roadmap:

- v1.9 MCP execution layer: define controlled provider execution boundaries,
  proof channels, and approval gates.
- v2.0 Cocos automation: add approved Cocos project operations only after
  runtime and safety architecture mature.
- v2.x production Agent runtime: introduce actual Agent runtime only after CLI,
  permission, context, memory, QA, and audit systems have validation coverage.

## 12. Explicit Limitations

This document creates:

- no runtime code
- no Agent implementation
- no MCP execution
- no Cocos automation
- no autonomous execution

It also creates no `agents/*` files, no runtime files, no protocol changes, no
examples, and no real Cocos project changes.

Human approval remains required for execution, scope expansion, Cocos/MCP use,
memory writes, runtime proof, Git actions, and release actions.
