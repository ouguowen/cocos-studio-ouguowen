# AI Game Studio v3.5 Capability Marketplace Architecture

This document defines the Capability Marketplace architecture for AI Game
Studio. It is architecture documentation only. It does not implement code,
modify MCP, modify Cocos Creator projects, create Agent runtime behavior, modify
`agents/*`, or change runtime execution logic.

## 1. Capability Marketplace Goal

AI Game Studio needs a Capability Marketplace so future game-development skills
can be described, discovered, versioned, reviewed, and connected to Agents
without bloating the core system.

The marketplace exists to:

- describe reusable game-development capabilities
- map tasks to capabilities before selecting an Agent
- keep advanced capabilities trigger-only and non-default
- let Agents discover specialist support without loading the whole repository
- keep capability dependencies explicit
- preserve human approval before installation, loading, or project modification
- separate architecture design from runtime execution

The Capability Marketplace is a catalog and governance layer. It is not an
automatic installer, runtime executor, or permission bypass.

## 2. Capability Model

Each capability must have a clear metadata record.

Required fields:

- Capability ID
- Capability Name
- Version
- Description
- Required Agent
- Dependencies
- Category
- Status
- Inputs
- Outputs
- Permission Boundary
- Validation Requirement

Example:

```text
Capability ID:
  cocos-character-controller

Capability Name:
  Cocos Character Controller Planning

Version:
  1.0.0

Description:
  Designs a scoped Cocos Creator 3.8.8 character controller task plan.

Required Agent:
  Code Agent

Dependencies:
  cocos-typescript-component-rules
  game-design-context
  validation-framework
```

Capability rules:

- Capability ID must be stable and unique.
- Version must change when inputs, outputs, status, or permission boundaries
  change.
- Required Agent must refer to an Agent category or Agent Registry identity.
- Dependencies must be explicit.
- Capabilities do not grant execution permission.
- Capabilities must state validation expectations before implementation use.

## 4. Capability Discovery

Agents use capability discovery to find the smallest useful capability for the
current task.

Discovery flow:

```text
Agent Request
      |
      v
Capability Query
      |
      v
Capability Marketplace
      |
      v
Candidate Capabilities
      |
      v
Dependency Check
      |
      v
Permission Check
      |
      v
Selected Capability
```

Example: Code Agent searches for a character controller capability.

```text
Code Agent
  |
  v
Search:
  cocos-character-controller
```

Possible capability result:

- Capability ID: `cocos-character-controller`
- Category: Gameplay System
- Required Agent: Code Agent
- Dependencies: Cocos TypeScript rules, Project Memory, QA validation
- Status: beta or stable

Example: Asset Agent searches for a 2D character pipeline.

```text
Asset Agent
  |
  v
Search:
  2D-character-pipeline
```

Possible capability result:

- Capability ID: `2D-character-pipeline`
- Category: Asset Pipeline
- Required Agent: Asset Agent
- Dependencies: style guide, naming convention, validation framework
- Status: stable

Discovery rules:

- prefer stable capabilities when multiple matches exist
- prefer narrow capabilities over broad ones
- reject deprecated capabilities unless explicitly approved
- do not load capability details unless the task directly requires them
- do not install or execute a capability automatically
- return a blocker if dependencies or approval are missing

## 5. Capability Version Management

Capability status values:

- stable
- beta
- deprecated

Stable:

- reviewed
- documented
- has clear inputs and outputs
- has validation expectations
- can be recommended for normal approved workflows

Beta:

- experimental
- usable only with explicit approval
- may have incomplete validation coverage
- must not be loaded by default in Fast Build Mode

Deprecated:

- replaced or no longer recommended
- must not be selected automatically
- may remain for historical reference
- requires human approval if used for compatibility reasons

Version rules:

- patch version: wording, examples, or non-behavioral metadata update
- minor version: added capability output, new dependency, or expanded category
- major version: changed permission boundary, changed required Agent, or changed
  task behavior

Deprecation record:

- deprecated version
- replacement capability
- reason
- migration note
- approval requirement if still used

## 6. Capability Installation and Loading Flow

The marketplace separates capability discovery from installation and loading.

```text
Capability Search
      |
      v
Capability Proposal
      |
      v
Human Approval
      |
      v
Install or Enable
      |
      v
Load for Task
      |
      v
Use in Approved Scope
```

Installation proposal must include:

- Capability ID
- version
- category
- dependencies
- required Agent
- files or docs to load
- permission boundary
- validation requirement
- rollback plan

Loading rules:

- capabilities are not loaded by default
- Fast Build Mode loads only directly triggered capabilities
- beta capabilities require explicit human approval
- deprecated capabilities require explicit compatibility approval
- no capability may modify project files without approved scope
- no capability may call MCP by itself

Unloading rules:

- after task completion, capability context should be released
- capability results may be summarized into approved Project Memory only when
  memory write is explicitly approved
- failed capability use must produce a blocker report

## 7. Relationship With Agent Registry

v3.5 connects with v3.4 Agent Registry by mapping capabilities to registered
Agents.

```text
Task Requirement
      |
      v
Capability Marketplace
      |
      v
Capability Match
      |
      v
Agent Registry
      |
      v
Agent Selection
      |
      v
Multi-Agent Protocol
```

v3.4 Agent Registry provides:

- Agent identity
- Agent status
- Agent capability boundary
- Agent version
- Agent availability

v3.5 Capability Marketplace provides:

- capability metadata
- capability category
- version status
- dependencies
- required Agent
- loading rules

v3.3 Multi-Agent Protocol uses both:

- Agent Registry selects who can receive the task.
- Capability Marketplace selects what capability the Agent should use.
- Message Protocol carries task, context, approval state, and validation
  requirements.

v2.5 Capability Mapping provides:

- MCP capability mapping categories
- Cocos capability intent mapping
- expected result shape for engine-facing actions

Together:

- v2.5 maps engine capabilities.
- v3.4 maps Agents.
- v3.5 maps reusable AI Game Studio capabilities.

## 8. Human Approval Mechanism

Human approval remains required before any high-impact capability action.

Approval required before:

- installing or enabling a new capability
- loading beta or deprecated capabilities
- using a capability to modify project files
- using a capability to call MCP
- using a capability to write Project Memory
- expanding a task through capability dependencies
- committing, pushing, tagging, or releasing capability-related changes

Human review packet:

- Capability ID
- version
- status
- required Agent
- dependencies
- task reason
- expected output
- files or project areas affected
- validation plan
- rollback plan
- stop conditions

Safety rules:

- marketplace discovery is not approval
- capability availability is not approval
- dependency installation is not automatic
- no capability may bypass Agent Registry, Multi-Agent Protocol, or human
  confirmation
