# v1.4 Context Engine Architecture

This document prepares the future Context Engine architecture for AI Game
Studio. It is documentation only. It does not create runtime code, databases,
Agent implementation, MCP integration, Cocos automation, or real Cocos project
changes.

## 1. Context Engine Architecture

The Context Engine is the future layer that selects the smallest useful context
for a task. It should help AI Game Studio avoid loading the whole Skill,
unrelated memory, proof history, examples, or runtime templates by default.

The Context Engine should:

- classify the user request
- identify required context categories
- load core context first
- load task context only when needed
- load optional context only when directly triggered
- load memory context only after approval
- detect conflicts between context sources
- summarize context compactly for Fast Build Mode

It is not a runtime implementation in this document.

## 2. Relationship Between Memory and Context

Memory stores approved project facts, decisions, and histories.

Context is the selected subset of information used for a current task.

Memory is persistent. Context is temporary.

The Context Engine may retrieve memory, but memory must not load by default.
Project-specific memory requires explicit project path and scope approval.

## 3. Context Retrieval Pipeline

Future retrieval pipeline:

```text
Human Request
      |
      v
Task Classification
      |
      v
Core Context Load
      |
      v
Task Context Selection
      |
      v
Optional Context Gate
      |
      v
Memory Context Gate
      |
      v
Conflict Check
      |
      v
Compact Context Summary
```

The pipeline stops at any gate that requires approval, missing scope, or conflict
resolution.

## 4. Core Context

Core context includes the smallest always-safe routing and safety material:

- scope boundary
- operation mode
- Fast Build Mode rules
- pre-write approval
- forbidden default Cocos/MCP access
- human approval gates
- proof honesty
- validation expectations

Core context should stay compact and stable.

## 5. Task Context

Task context is loaded only for the current request.

Examples:

- documentation task files
- approved implementation files
- relevant design docs
- relevant architecture docs
- validation commands
- current Git status and diff

Task context expires when the task closes.

## 6. Asset Context

Asset context contains information about approved assets:

- asset names
- paths
- ownership
- import status
- placeholder or production status
- usage constraints

Asset context must not replace Cocos asset validation or source-control truth.

## 7. Character Context

Character context contains approved character-related information:

- character identity
- role
- gameplay purpose
- visual representation
- animation state list
- UI linkage

Character context should remain separate from runtime implementation unless the
user approves a project task.

## 8. Animation Context

Animation context contains approved presentation rules:

- animation state
- trigger
- exit rule
- fallback behavior
- proof requirement

Animation context must not own gameplay truth. Runtime state and gameplay logic
must remain authoritative.

## 9. UI Context

UI context contains approved interface decisions:

- screen or panel purpose
- visible controls
- input behavior
- feedback states
- data binding expectations
- accessibility notes

UI context should support implementation review without forcing UI work by
default.

## 10. Level Context

Level context contains approved level and configuration information:

- level id
- map model
- objectives
- waves or encounters
- rewards
- validation rules
- config schema

Level context should be scoped by selected game type. Do not force one universal
level model onto every game.

## 11. Context Priority System

Priority order:

1. current user instruction
2. explicit approval and scope
3. current repository state
4. validation results
5. runtime/browser proof when applicable
6. active task context
7. approved project memory
8. archive or proof history

When priorities conflict, stop and report the conflict instead of silently
choosing a source.

## 12. Context Compression Strategy

Context compression should:

- keep decisions and facts
- remove duplicated wording
- summarize old history
- cite source files when needed
- keep stop conditions visible
- keep forbidden scope visible
- avoid loading full proof logs by default

Compression must never remove safety, approval boundaries, proof honesty, or
rollback requirements.

## 13. Context Conflict Handling

Conflict examples:

- memory contradicts current repository state
- proof history contradicts current validation
- user request conflicts with forbidden files
- archive docs suggest obsolete behavior
- optional context tries to become default context

Conflict handling:

1. stop the task
2. name the conflicting sources
3. prefer current user instruction when safe
4. prefer repository state over stale memory
5. prefer validation and runtime proof over remembered claims
6. request approval before updating memory or scope

## 14. Future Runtime Implementation Roadmap

Possible future roadmap:

1. keep Context Engine architecture documentation-only
2. define context categories and metadata
3. add read-only context inspection commands
4. add memory retrieval gates
5. add context conflict detection
6. add compact context summaries
7. integrate with CLI only after permission and memory policies are validated
8. integrate with Cocos/MCP only after a separate runtime architecture review

## Safety Boundary

This document does not create:

- runtime code
- database storage
- Agent implementation
- MCP execution
- Cocos automation
- real project context files
- real Cocos project changes

The Context Engine remains future architecture only.
