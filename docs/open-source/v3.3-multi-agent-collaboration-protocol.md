# AI Game Studio v3.3 Multi-Agent Collaboration Protocol

This document defines the Multi-Agent Collaboration Protocol for AI Game Studio.
It is architecture documentation only. It does not implement code, modify MCP,
modify Cocos Creator projects, create Agent runtime behavior, modify `agents/*`,
or change runtime logic.

## 1. Multi-Agent System Goal

The Multi-Agent system lets AI Game Studio operate like a controlled production
team. Each Agent has a clear identity, responsibility, capability boundary, and
handoff contract.

Goals:

- coordinate planning, code generation, asset planning, validation, review, and
  release work
- keep task ownership explicit
- prevent one Agent from silently expanding scope
- keep human approval between planning and real project modification
- preserve Fast Build Mode by loading only the Agents directly needed by a task
- create an auditable trail from request to result

The collaboration protocol makes Agents cooperate as a team, but does not grant
autonomous project modification.

## 2. Agent Identity Model

Each Agent must have a stable identity record before participating in a workflow.

Identity fields:

- Agent ID
- Agent Role
- Agent Capability
- Agent Status
- Agent Responsibility
- Permission Boundary
- Current Task
- Handoff Target

Example Agents:

### Planner Agent

Responsibilities:

- game planning
- task decomposition
- dependency ordering
- scope boundary definition
- approval packet preparation

Capabilities:

- read approved project context
- create task plans
- route work to specialist Agents
- summarize blockers

### Code Agent

Responsibilities:

- Cocos Creator code generation planning
- approved file-scope implementation proposal
- code task result reporting

Capabilities:

- receive approved code tasks
- propose implementation changes
- return changed-file and validation expectations

Boundary:

- no code modification without human-approved scope
- no autonomous repair

### Asset Agent

Responsibilities:

- art resource requirement management
- asset naming and consistency planning
- prefab and import expectation planning

Capabilities:

- convert gameplay requirements into asset needs
- identify missing asset categories
- prepare validation expectations for assets

Boundary:

- no asset import or generation without approval
- no Cocos project modification

### QA Agent

Responsibilities:

- testing validation
- quality review
- runtime proof review when approved
- bug and risk reporting

Capabilities:

- review validation results
- classify defects
- propose repair tasks
- produce PASS, WARNING, FAILED, or BLOCKED reports

Boundary:

- no automatic repair
- no automatic merge approval

## 3. Agent Message Protocol

Agents communicate through structured messages. Messages should be small,
task-focused, and auditable.

Message format:

```json
{
  "sender": "Planner Agent",
  "receiver": "Code Agent",
  "task": {
    "task_id": "T001",
    "task_type": "code_generation",
    "title": "Create GameManager"
  },
  "context": {
    "project_type": "Tower Defense",
    "approved_files": ["assets/scripts/tower-defense/GameManager.ts"],
    "forbidden_files": ["package.json", "project.json", "settings/*"]
  },
  "priority": "normal",
  "status": "waiting"
}
```

Required message fields:

- sender
- receiver
- task
- context
- priority
- status
- approval_state
- expected_output
- validation_requirement
- stop_conditions

Optional message fields:

- dependency
- deadline
- risk_level
- memory_reference
- feedback_reference
- rollback_note

Message rules:

- A message is not approval.
- A message cannot expand allowed files by itself.
- A receiving Agent must reject unclear scope.
- A failed task must return a blocker message instead of silently continuing.

## 4. Agent Task Handoff Mechanism

The default production handoff path is:

```text
Planner
    |
    v
Code Agent
    |
    v
Asset Agent
    |
    v
QA Agent
```

Typical handoff:

1. Planner Agent converts user intent into a task plan.
2. Human approves scope when project modification is required.
3. Code Agent receives approved implementation task.
4. Asset Agent receives approved asset requirements or asset validation tasks.
5. QA Agent receives output, validation requirements, and proof targets.
6. QA Agent returns PASS, WARNING, FAILED, or BLOCKED.
7. Planner Agent prepares the next recommendation, which remains optional advice.

Handoff packet:

- task id
- source Agent
- target Agent
- task summary
- dependency status
- approved scope
- expected output
- validation requirement
- known risks
- stop conditions

Handoff must stop when:

- approved scope is missing
- project path is missing for real project work
- forbidden files would be touched
- runtime or MCP action is requested without approval
- the receiving Agent lacks required capability
- validation fails in a way that needs repair planning

## 5. Agent State Synchronization

Agent state must be explicit so the system does not lose ownership or continue
work without a valid owner.

Supported states:

- waiting
- working
- reviewing
- completed
- failed

State meanings:

- waiting: Agent has a task but cannot start until dependencies or approval are
  ready.
- working: Agent is performing approved analysis, planning, or execution.
- reviewing: Agent output is under QA, code review, or human review.
- completed: Agent output passed its expected result and handoff is closed.
- failed: Agent hit a blocker, validation failure, unclear scope, or forbidden
  action boundary.

State synchronization fields:

- agent id
- task id
- current state
- previous state
- owner
- last message id
- blocker id
- validation status
- human approval status

State transition rules:

- waiting -> working requires clear scope.
- working -> reviewing requires produced output.
- reviewing -> completed requires validation or human acceptance.
- any state -> failed when a stop condition appears.
- failed -> working requires a new approved repair task.

## 6. Agent Collaboration Rules

Calling rules:

- Planner Agent may route tasks to Design, Code, Asset, QA, and Release Agents.
- Code Agent may request QA review and Code Review feedback.
- Asset Agent may request QA review for asset consistency and binding plans.
- QA Agent may return issues to Planner Agent or Code Generation Orchestrator.
- Feedback Learning may receive approved postmortem records, but does not call
  project modification tasks by itself.

Review ownership:

- QA Agent owns validation review.
- Code Review Agent owns code review findings.
- Planner Agent owns task repair planning.
- Human reviewer owns final approval for project modification, merge, release,
  and scope expansion.

Final confirmation:

- Human approval has final authority.
- Agent consensus is not a substitute for human approval.
- `next recommended command` is optional advice only.
- No Agent may approve its own high-impact output.

Forbidden collaboration behavior:

- no fully autonomous project modification
- no silent scope expansion
- no unapproved Cocos/MCP execution
- no unapproved memory writes
- no automatic commit, push, tag, or release
- no bypass of validation or rollback discipline

## 7. Human Approval Mechanism

Human approval remains the control boundary between Agent planning and project
modification.

Approval flow:

```text
Agent Proposal
      |
      v
Human Review
      |
      v
Approved Scope
      |
      v
Agent Execution or Validation
      |
      v
Report
```

Approval is required before:

- modifying code, scenes, prefabs, assets, metadata, package files, settings, or
  build configuration
- calling Cocos MCP for project-changing operations
- running browser preview when tied to real project proof
- writing project memory
- applying review suggestions
- retrying with broader scope
- committing, pushing, tagging, or releasing

The approval packet must include:

- project path when real project work is involved
- allowed files
- forbidden files
- expected output
- validation plan
- rollback plan
- stop conditions
- responsible Agent

AI must not fully autonomously modify a project. Agents may propose, execute only
inside approved scope, report, and stop on blockers.

## 8. Integration With Existing Architecture

v2.x MCP Adapter:

- receives approved execution requests after permission checks.
- exposes Cocos capability mapping without becoming default context.
- remains explicit and non-default for real project changes.

v2.6 Task Planning:

- provides task ids, dependencies, required capabilities, MCP mappings, and
  validation requirements.
- feeds the Planner Agent handoff packet.

v2.10 Project Memory:

- provides approved project identity, decision history, architecture state, and
  validation history when explicitly triggered.
- may receive Agent history only after human-confirmed memory write.

v3.0 Code Generation:

- receives approved code tasks from Planner Agent.
- routes implementation work to Codex through a scoped execution proposal.

v3.1 Code Review:

- reviews generated code after implementation.
- returns structured findings to Code Generation Orchestrator and Planner Agent.

v3.2 Feedback Learning:

- receives approved review, QA, Cocos, and human feedback.
- extracts reusable knowledge only after human confirmation.

Combined collaboration loop:

```text
Planner Agent
      |
      v
Task Planning
      |
      v
Code / Asset / QA Agents
      |
      v
Review and Validation
      |
      v
Human Approval
      |
      v
Feedback Learning
```

The protocol links Agents into a controlled studio workflow while preserving
approval, validation, and safety boundaries.
