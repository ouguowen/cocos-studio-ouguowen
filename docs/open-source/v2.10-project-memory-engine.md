# AI Game Studio v2.10 Project Memory Engine Architecture

This document defines the Project Memory Engine architecture for AI Game Studio.
It is documentation only. It does not create runtime files, implement Agents,
implement MCP, modify Cocos projects, or authorize autonomous execution.

## 1. Positioning

The Project Memory Engine is the long-term project knowledge layer for AI Game
Studio.

It sits between current task execution and durable project understanding:

```text
Game Knowledge Engine
      |
      v
Planner Agent
      |
      v
Context Engine
      |
      v
Project Memory Engine
      |
      v
Execution Controller
      |
      v
MCP Adapter
      |
      v
Cocos Creator 3.8.8
```

AI Game Studio needs long-term project memory because game development spans
many decisions, files, assets, scenes, systems, and validation results. Without a
controlled memory layer, every session must reload too much context or risk
forgetting why the project is shaped a certain way.

Project memory should reduce repeated context loading while preserving approval,
version awareness, and proof honesty.

## 2. Core Responsibilities

The Project Memory Engine records durable project facts that have been approved,
reviewed, or validated. It must not store unapproved guesses as project truth.

### 2.1 Project Identity Memory

Project Identity Memory records stable baseline information:

- game name
- game type
- engine version
- technical stack
- target platform
- active repository or project path
- owner or approving human
- current milestone

This memory helps Planner Agent and Context Engine identify the active project
without re-reading broad project history by default.

### 2.2 Design Decision Memory

Design Decision Memory records:

- game design decisions
- reason for each change
- historical version changes
- rejected alternatives
- approval status
- related validation result

Design decisions should include why a choice was made, not only what the final
choice is. This prevents future Agents from undoing prior reasoning by accident.

### 2.3 Asset Memory

Asset Memory records approved asset direction:

- character style
- scene style
- UI standard
- animation standard
- naming conventions
- placeholder versus production status
- asset ownership
- import or validation status

Asset Memory supports the Asset Pipeline Engine, but it does not replace Cocos
asset metadata, source files, or runtime validation.

### 2.4 Code Structure Memory

Code Structure Memory records:

- project code structure
- module relationships
- system dependencies
- approved folder boundaries
- key runtime entry points
- shared framework rules
- forbidden coupling

Code Structure Memory helps future implementation plans avoid uncontrolled
refactors, duplicate systems, and hidden dependency drift.

### 2.5 Agent History Memory

Agent History Memory records:

- Planner task history
- QA issue history
- modification records
- approval checkpoints
- blocker history
- validation outcomes
- rollback notes

Agent history must remain audit-oriented. It must not become automatic approval
for future work.

## 3. Architecture Relationships

### Context Engine

The Context Engine retrieves only the memory needed for the current task. It
should not load all project memory by default.

Project Memory Engine provides durable facts. Context Engine selects temporary
task context.

### Planner Agent

Planner Agent queries Project Memory to:

- identify current project state
- avoid repeating old decisions
- detect missing approval
- prepare better task plans
- select relevant capabilities

Planner Agent must not write memory without explicit approval.

### QA Agent

QA Agent uses Project Memory to:

- compare planned behavior against prior decisions
- check repeated issues
- verify validation history
- detect regression risk
- recommend memory updates after approved fixes

QA Agent may recommend memory changes but cannot approve them by itself.

### Game Knowledge Engine

Game Knowledge Engine provides general genre and system knowledge.

Project Memory Engine stores project-specific choices made from that knowledge.

Example:

- Game Knowledge Engine knows tower defense usually needs waves and towers.
- Project Memory Engine records this project uses two enemy lanes and no economy
  in the current slice.

### Asset Pipeline Engine

Asset Pipeline Engine uses Project Memory to keep style, naming, and asset
requirements consistent across sessions.

Project Memory should record approved asset standards, not raw generated images
or unvalidated imports.

### MCP Adapter

MCP Adapter may receive approved project memory only when needed to execute an
approved task.

Project Memory must not grant MCP access. Permission checks and human approval
remain required before Cocos/MCP execution.

## 4. Future Extension Directions

### Memory Retrieval

Memory retrieval should be category-based:

- identity
- design
- asset
- code structure
- Agent history
- validation

Retrieval must be trigger-controlled and scoped to the current task.

### Project Knowledge Graph

A future Project Knowledge Graph may connect:

- scenes
- systems
- scripts
- assets
- prefabs
- characters
- levels
- decisions
- validation records

The graph should support reasoning and conflict detection, not autonomous
execution.

### Version Awareness

Memory entries should track:

- version
- related commit
- source document
- approving human
- last reviewed date
- status: draft, active, superseded, retired

Version awareness prevents stale memory from becoming hidden authority.

### Multi-Agent Shared Memory

Future shared memory may allow Planner, Designer, Coder, QA, and Release Agents
to reference the same approved project facts.

Shared memory requires:

- read permission
- write approval
- conflict detection
- audit record
- rollback or retirement path

No Agent may silently overwrite shared memory.

## 5. Safety Constraints

This document creates:

- no runtime implementation
- no MCP changes
- no autonomous execution
- no Agent implementation
- no Cocos project changes
- no database
- no project-specific memory entries

Project Memory is future architecture only. Human approval remains required for
memory writes, project inspection, MCP execution, Cocos automation, Git actions,
and release actions.
