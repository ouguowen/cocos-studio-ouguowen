# AI Game Studio v3.9 Runtime Validation & Playtest System

This document defines the Runtime Validation & Playtest System architecture for AI Game Studio.

This is architecture documentation only. It does not implement code, modify MCP, modify Cocos Creator projects, create Agent runtime behavior, modify `agents/*`, or change runtime execution logic.

## 1. Runtime Validation Goal

The Runtime Validation & Playtest System gives AI Game Studio a controlled way to reason about game runtime quality after approved implementation work.

Its goal is to let AI-assisted development verify whether a game actually runs, loads the expected scene, exposes the intended gameplay flow, presents usable UI, avoids obvious performance risks, and reports no critical runtime errors.

The system is not autonomous playtesting. It is a validation architecture that requires approved scope, approved project path, and human control before any real project inspection, Cocos/MCP call, browser preview, or runtime operation.

Runtime validation should answer:

- Does the project start?
- Does the expected Scene load?
- Does the intended gameplay path exist?
- Is the UI visible and interactive?
- Are runtime errors present?
- Are performance risks visible?
- Is the result honest enough for QA or release review?

## 2. Automated Test Flow

The future automated validation flow is:

```text
Approved Runtime Validation Request
  -> Start Project
  -> Scene Load Detection
  -> Gameplay Flow Detection
  -> UI Detection
  -> Performance Detection
  -> Error Log Detection
  -> Playtest Report
  -> Human Review
```

### Start Project

The system verifies that the selected project can be opened or launched through an approved execution channel.

Validation checks:

- Approved project path exists.
- Engine version is compatible with the project requirement.
- No unexpected project setting change is required.
- Startup command or preview operation is inside the approved scope.

### Scene Load Detection

The system confirms that the expected Scene loads instead of assuming editor hierarchy equals runtime proof.

Validation checks:

- Expected Scene name or path is active.
- Required root nodes exist.
- Required Camera and Canvas state is valid for the selected game type.
- Missing Reference or broken component errors are captured.

### Gameplay Flow Detection

The system validates the core playable path defined by the current stage.

Examples:

- Prototype: one narrow proof action can be triggered.
- Vertical Slice: the core loop has a visible start, action, result, and report.
- Production: selected feature paths match the approved validation plan.

Gameplay validation must stay scoped. It must not expand into full game acceptance unless that release gate is explicitly approved.

### UI Detection

The system verifies UI visibility and interaction readiness.

Validation checks:

- Required UI nodes are visible.
- Buttons or interactive elements can be detected.
- Text is readable at the expected resolution.
- UI state changes match the approved gameplay path.
- UI does not falsely own gameplay truth.

### Performance Detection

The system captures basic performance risk signals.

Validation checks:

- Startup time is acceptable for the target stage.
- Frame rate does not show obvious collapse during the scoped proof.
- Object creation spikes are identified.
- Repeated `update()` risk is flagged when visible through logs or review.
- Memory or resource warnings are recorded when available.

This is not a full benchmark system. It is an early warning layer for QA and optimization planning.

### Error Log Detection

The system records runtime logs and classifies errors.

Validation checks:

- Browser or preview Console errors.
- TypeScript runtime errors.
- Cocos component lifecycle errors.
- Asset loading errors.
- Missing component or missing reference errors.
- Validation warnings generated by approved QA checks.

Runtime proof must not be claimed when logs are unavailable unless the report clearly states the limitation.

## 3. Playtest Agent Design

The Runtime Validation & Playtest System uses future Agent roles as policy-level responsibilities. This document does not create Agent implementations.

### QA Agent

Responsibilities:

- Own the validation plan.
- Confirm required runtime checks for the current stage.
- Classify validation result as PASS, NEEDS_REPAIR, FAIL, or BLOCKED.
- Produce the final runtime validation report.

Allowed inputs:

- Approved scope.
- Expected Scene.
- Acceptance criteria.
- Previous QA results.
- Project Memory records approved for use.

Expected outputs:

- Runtime validation report.
- Error classification.
- Recommended repair scope.
- Human review summary.

### Gameplay Tester Agent

Responsibilities:

- Exercise the approved gameplay path.
- Verify core loop visibility.
- Detect blocked interactions.
- Report whether the player-facing path matches the design intent.

Allowed inputs:

- Game Design Document excerpt.
- Stage validation target.
- Approved interaction script.
- Current Scene and UI expectations.

Expected outputs:

- Gameplay path result.
- Interaction failures.
- Missing feedback notes.
- Playability risks.

### Performance Agent

Responsibilities:

- Detect obvious runtime performance risks.
- Record frame, object, loading, and resource warning signals when available.
- Recommend whether deeper optimization review is needed.

Allowed inputs:

- Runtime validation result.
- Console logs.
- Performance counters when provided by the approved execution environment.
- Code Review Agent performance findings.

Expected outputs:

- Performance risk report.
- Suspected hotspots.
- Optimization priority proposal.

## 4. Connection With Code Review Agent

Runtime validation and code review form a closed quality loop.

```text
Generated Code
  -> Code Review Agent
  -> Review Findings
  -> Runtime Validation
  -> Playtest Findings
  -> Code Generation Orchestrator
  -> Repair Proposal
```

The Code Review Agent identifies risks before runtime execution, such as lifecycle misuse, unmanaged `update()` logic, unsafe asset references, or Cocos Creator 3.8.8 component issues.

Runtime validation then confirms which risks appear in actual execution. A runtime failure should produce a repair proposal, not automatic code modification.

## 5. Connection With Feedback Learning

Runtime validation results can become learning material only after human confirmation.

Feedback flow:

```text
Runtime Error or Playtest Failure
  -> QA Classification
  -> Root Cause Proposal
  -> Human Confirmation
  -> Feedback Learning Record
```

Examples of reusable learning:

- Scene loads in editor but fails in browser preview.
- UI button is visible but interaction does not update gameplay state.
- Object creation during `update()` causes performance risk.
- Missing prefab reference breaks the approved gameplay path.

Unconfirmed failures must not be stored as permanent knowledge.

## 6. Connection With Project Memory

Project Memory records approved runtime validation facts.

Recommended records:

- Last validated Scene.
- Runtime proof target.
- Browser or preview result.
- Console error summary.
- Gameplay path result.
- UI validation result.
- Performance risk summary.
- Approved blockers and repair decisions.
- Human acceptance state.

Memory write rule:

```text
Runtime Validation Report
  -> Human Review
  -> Approved Memory Write
  -> Project Memory Update
```

Project-specific memory access remains trigger-controlled and non-default. It requires an approved project path and approved read or write scope.

## 7. Human Approval Nodes

Human approval is required before:

- Starting runtime validation on a real Cocos project.
- Opening or inspecting a real project.
- Calling Cocos/MCP tools.
- Running browser preview or engine preview.
- Writing runtime validation results into Project Memory.
- Expanding from a narrow validation path into broad playtesting.
- Generating repair changes from validation findings.
- Treating a validation result as release acceptance.

Approval workflow:

```text
Validation Proposal
  -> Human Scope Approval
  -> Runtime Validation
  -> QA Report
  -> Human Acceptance or Repair Decision
```

No runtime validation result authorizes the next production slice automatically.
