# AI Game Studio v3.2 Feedback Learning Architecture

This document defines the Feedback Learning architecture for AI Game Studio. It
is documentation only. It does not implement a learning system, create runtime
files, create Agent implementations, modify `agents/*`, change MCP, modify
Cocos Creator projects, or execute automatic repair.

## 1. Feedback Learning System Architecture

Feedback Learning turns development problems into reviewed experience that can
improve future planning, review, validation, and architecture decisions.

```text
Development Process
        |
        v
Problem Detection
        |
        v
Feedback Collection
        |
        v
Knowledge Extraction
        |
        v
Experience Memory
        |
        v
Future Decision Optimization
```

Position in AI Game Studio:

- Development Process creates design, code, asset, scene, and validation work.
- Problem Detection identifies issues from review, QA, Cocos validation, and
  human feedback.
- Feedback Collection records the issue, context, severity, decision, and repair
  outcome.
- Knowledge Extraction turns raw feedback into reusable lessons only after
  review.
- Experience Memory stores approved project experience for future retrieval.
- Future Decision Optimization uses confirmed experience to avoid repeated
  mistakes and propose safer architecture.

Feedback Learning is advisory. It must not automatically change project files,
modify code, or create new tasks without human approval.

## 2. Feedback Source Design

### A. Code Review Feedback

Source:

- v3.1 Code Review Agent

Records:

- code issue
- risk level
- affected file or module
- review category
- suggested modification
- final solution
- whether the solution was accepted by a human

Examples:

- repeated logic across systems
- Cocos lifecycle misuse
- object allocation inside `update`
- unclear ownership of gameplay state
- missing validation after generated code

### B. QA Feedback

Source:

- Game Validation Framework

Records:

- bug
- failed test
- performance issue
- user experience issue
- reproduction condition
- validation command or proof target
- final repair result

Examples:

- gameplay loop incomplete
- UI flow unclear
- scene validation failed
- runtime preview mismatch
- asset reference missing

### C. Cocos Engine Feedback

Source:

- Cocos MCP / Engine Validation

Records:

- compile error
- Component error
- Asset error
- Scene error
- Prefab or metadata issue
- Console/runtime error
- validated fix

Examples:

- missing Component binding
- broken prefab reference
- asset import mismatch
- TypeScript decorator mismatch
- scene node path changed without script update

### D. Human Feedback

Records:

- user modification request
- design adjustment
- approval result
- rejection reason
- manual decision
- project priority change

Examples:

- "This feature is too broad for the current slice."
- "Keep this as a validation sandbox only."
- "Do not continue the next proof automatically."
- "Change the architecture, but do not modify the Cocos project."

Human feedback has the highest authority when deciding whether a lesson becomes
stored experience.

## 3. Experience Memory Architecture

Experience Memory stores reviewed lessons in a consistent structure. It should
store reusable context, not raw noisy logs.

Example record:

```text
Experience ID:
  EXP-001

Category:
  Cocos Performance

Problem:
  Large numbers of Enemy instances caused frame drops.

Cause:
  Enemy objects were repeatedly created and destroyed without pooling.

Solution:
  Use PoolManager for repeated Enemy creation and release.

Context:
  Tower Defense Game

Evidence:
  QA performance report and Code Review finding.

Approval:
  Confirmed by human reviewer.
```

Recommended fields:

- experience id
- category
- project type
- problem
- cause
- solution
- context
- affected modules
- evidence source
- validation result
- approval status
- date or version
- reuse guidance

Experience categories:

- Cocos Performance
- Cocos Component Binding
- Scene Architecture
- Asset Pipeline
- UI Consistency
- Gameplay Logic
- Code Architecture
- QA Process
- Human Scope Control

## 4. Knowledge Extraction Pipeline

Raw feedback must be filtered before it becomes reusable knowledge.

```text
Raw Feedback
      |
      v
Problem Classification
      |
      v
Root Cause Analysis
      |
      v
Solution Extraction
      |
      v
Reusable Knowledge
```

Pipeline stages:

- Raw Feedback: collect issue reports, QA notes, Cocos errors, and human
  comments.
- Problem Classification: group by code, asset, scene, gameplay, Cocos engine,
  performance, UI, or process.
- Root Cause Analysis: identify the underlying cause rather than storing only
  symptoms.
- Solution Extraction: record the repair that actually resolved the issue.
- Reusable Knowledge: convert the approved lesson into future planning guidance.

Quality rules:

- Do not store speculation as experience.
- Do not store failed repairs as successful lessons.
- Do not store project-specific secrets or private data.
- Do not let one isolated issue become a universal rule without review.
- Do not overwrite existing experience without audit history.

## 5. Integration With Existing Architecture

v2.10 Project Memory Engine:

- stores approved project memory, decision history, architecture state, and
  validation history.
- Feedback Learning may write only approved experience records into memory.

v2.8 Game Knowledge Engine:

- provides genre rules and reusable design patterns.
- Feedback Learning may add reviewed experience that improves future templates
  or warnings.

v2.11 Game Validation Framework:

- produces QA findings, validation failures, runtime proof results, and bug
  classifications.
- Feedback Learning uses these results as evidence for experience records.

v3.0 Code Generation Orchestration:

- produces task ids, generated files, validation requirements, and Codex repair
  loops.
- Feedback Learning connects issue records back to task planning and retry
  strategy.

v3.1 Code Review Agent:

- produces structured code review reports.
- Feedback Learning extracts recurring code risks, Cocos risks, and repair
  strategies from approved review outcomes.

Combined feedback loop:

```text
Plan
  |
  v
Generate
  |
  v
Review
  |
  v
Validate
  |
  v
Human Confirm
  |
  v
Store Experience
  |
  v
Improve Future Planning
```

The loop improves future decisions without granting autonomous execution.

## 6. Future Optimization Flow

Feedback Learning supports future projects by retrieving confirmed experience
before architecture and task planning.

```text
New Project
    |
    v
Load Historical Experience
    |
    v
Avoid Known Problems
    |
    v
Generate Better Architecture
```

Optimization examples:

- A tower defense project can preload known wave, enemy pooling, and path
  validation lessons.
- A card game can preload deck, hand UI, turn-state, and effect-resolution
  lessons.
- A platformer can preload collision, checkpoint, and animation-state lessons.
- A Cocos project can preload component binding, prefab reference, and asset
  naming lessons.

Future decisions may improve:

- architecture selection
- file structure
- task decomposition
- validation checklist
- performance risk warning
- asset requirement planning
- QA focus

Historical experience must be treated as context, not as automatic approval.

## 7. Human Approval Workflow

Experience storage requires human confirmation.

```text
Feedback Generated
        |
        v
AI Analysis
        |
        v
Human Confirmation
        |
        v
Store Experience
```

Approval is required before:

- storing a new experience record
- updating an existing lesson
- promoting a project-specific fix to reusable knowledge
- using feedback to change project architecture
- using feedback to modify code, assets, scenes, or MCP behavior

Human confirmation packet:

- feedback source
- problem summary
- root cause
- proposed lesson
- evidence
- affected project type
- reuse boundary
- risk of overgeneralization

Safety rule:

- incorrect or unverified experience must not enter the knowledge base
  automatically.
- rejected feedback may be archived as history, but must not be used as guidance
  unless later approved.
