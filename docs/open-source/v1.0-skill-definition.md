# v1.0 Skill Definition

## Endpoint

v1.0 is the point where `cocos-studio-ouguowen` reliably guides a beginner or solo developer from a game idea to a scoped, reviewed, and proof-backed Cocos Creator 3.8.8 MVP path without drifting into unsafe or unapproved implementation.

## Identity

This Skill is an AI Game Studio operating system for Codex + Cocos Creator 3.8.8.

It is not:

- a complete game
- a single game template
- a Cocos Creator manual
- a default MCP automation tool
- a claim that any specific game project is finished

## Core v1.0 Capabilities

The v1.0 Skill should support:

- game idea to scoped MVP planning
- multi-game-type classification before template choice
- architecture and scene/script structure guidance
- pre-write approval before real project modification
- runtime/browser proof when implementation is approved
- QA and release review
- safe Git workflow with explicit staging and no force-push default
- validation and CI protection for Skill rules and documentation links

## Default Behavior

v1.0 should stay lightweight by default.

- Do not open Cocos Creator by default.
- Do not call Cocos MCP by default.
- Do not inspect real Cocos project files by default.
- Treat `next recommended command` as optional advice only, not automatic approval.
- Stop proof-chain work after each approved proof slice unless the user explicitly asks to continue.
- Keep Skill maintenance separate from real game implementation unless the user requests project work and approves the active project path and write scope.

## Success Criteria

v1.0 is successful when:

- a new user can understand what the Skill does and what it does not do
- a game idea can be narrowed into a safe MVP plan
- game type, scope, architecture, ownership, and proof expectations are visible before implementation
- project writes require pre-write approval and explicit file scope
- runtime claims require honest runtime/browser proof or a declared blocker
- QA, release, Git, validation, and CI checks are part of the normal workflow
- the Skill remains fast in normal use because heavy protocols, agents, indexes, and proof docs load only when directly triggered
- repository cleanup can decide what to keep, archive, or delete without guessing the long-term destination

v1.0 is not complete yet merely because this document exists.

## Core vs Extension Modules

Core modules should contain rules that almost every safe Cocos AI workflow needs:

- context loading and operation modes
- scope boundaries and approval rules
- project path and file-scope safety
- game classification and MVP planning
- architecture boundaries
- runtime proof honesty
- QA, release, Git, validation, and CI protection

Extension modules should contain specialized or optional guidance:

- genre-specific templates
- advanced level systems
- economy, monetization, live-ops, social, or compliance details
- animation and presentation deep dives
- multi-agent handoff systems
- provider-specific automation details, only when explicitly approved

If a module is only useful for a narrow scenario, keep it out of the default path and load it by trigger.

## Cleanup Decision Rules

Use these rules when cleaning the repository:

- Keep core routing, safety, approval, proof, validation, and CI docs.
- Keep files required by `scripts/validate_skill_docs.py`.
- Keep files referenced by `core/module-index.md`, unless the cleanup also updates the index and validation expectations.
- Archive historical records that explain past releases, migrations, or decisions.
- Delete only unreferenced duplicates, obsolete generated files, local cache files, or build artifacts.
- Do not delete validation-required or module-index-required files without a paired cleanup of those references and checks.
- Prefer small cleanup patches with validation over broad deletion batches.

## Never Optimize Away

The Skill may become shorter, faster, and easier to navigate, but it must never optimize away:

- safety
- pre-write approval
- explicit project path approval
- approved file scope
- runtime proof honesty
- rollback discipline
- no fake proof
- no hidden continuation from `next recommended command`
- separation between Skill maintenance and real game implementation

## Maintenance Guidance

Future roadmap and cleanup decisions should ask:

1. Does this change move the Skill closer to the v1.0 endpoint?
2. Does it keep the default path lightweight?
3. Does it preserve safety and proof honesty?
4. Does it avoid turning the Skill into a complete game, single template, Cocos manual, or default MCP tool?
5. Does it keep beginner wording clear while preserving enough precision for maintenance?

If the answer is no, the change should be narrowed, moved to an extension, archived as history, or rejected.
