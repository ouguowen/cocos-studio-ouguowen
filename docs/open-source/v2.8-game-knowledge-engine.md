# AI Game Studio v2.8 Game Knowledge Engine Architecture

This document defines the Game Knowledge Engine layer for AI Game Studio. It is
documentation only. It does not create runtime implementation, MCP changes, Cocos
automation, Agent execution, or autonomous execution.

## 1. Purpose

AI Game Studio needs game knowledge beyond Cocos automation because engine tools
can create objects but cannot decide what a good game should contain.

The Game Knowledge Engine provides:

- genre patterns
- gameplay system templates
- design rules
- numerical design guidance
- level design structure
- UI design expectations
- asset requirement translation

It helps Planner Agent and Execution Controller create better plans before any
Cocos operation is proposed.

## 3. Game Genre Knowledge

Tower Defense:

- core gameplay loop: place towers, stop waves, earn resources, upgrade defense
- required systems: map path, wave system, tower system, enemy system, economy,
  UI feedback
- common mechanics: targeting, range, upgrades, wave pacing, boss waves
- typical assets: map, tower sprites, enemy sprites, projectile/effect assets,
  HUD elements

Card Game:

- core gameplay loop: draw cards, choose actions, resolve effects, win battle
- required systems: card library, deck, hand, battle rules, effects, UI
- common mechanics: cost, rarity, turns, combos, status effects
- typical assets: card art, card frames, icons, board, effect visuals

RPG:

- core gameplay loop: explore, fight, progress, collect, upgrade
- required systems: character, inventory, dialogue, quests, combat, progression
- common mechanics: stats, skills, equipment, NPCs, rewards
- typical assets: characters, environments, items, dialogue UI, VFX

Platformer:

- core gameplay loop: move, jump, avoid hazards, collect, reach goal
- required systems: character controller, physics, level hazards, checkpoints
- common mechanics: jumping, moving platforms, enemies, collectibles
- typical assets: tiles, character sprites, obstacles, pickups, UI

Strategy:

- core gameplay loop: plan, allocate resources, command units, win objectives
- required systems: units, map, resources, AI, objectives, UI
- common mechanics: fog of war, upgrades, territory, formations
- typical assets: units, terrain, icons, command UI, effects

Puzzle:

- core gameplay loop: understand rule, manipulate state, solve challenge
- required systems: puzzle grid or objects, rules, win condition, feedback
- common mechanics: matching, pushing, rotation, logic gates, limited moves
- typical assets: tiles, symbols, feedback effects, level UI

Action:

- core gameplay loop: move, attack, dodge, react, defeat enemies
- required systems: player controller, enemies, combat feedback, health, result
- common mechanics: attacks, cooldowns, hit reactions, waves, boss patterns
- typical assets: character, enemies, effects, hit UI, audio cues

## 5. Game System Design Rules

Combat system:

- define actor roles, attack rules, damage rules, feedback, and failure states

Progression system:

- define unlocks, growth path, milestone pacing, and reset rules

Economy system:

- define currency purpose, source, sink, cap, reward cadence, and persistence

Reward system:

- define what is granted, why it matters, when it appears, and how it is shown

Achievement system:

- define target behavior, tracking rules, reward, and visibility

## 6. Numerical Design Rules

Difficulty curve:

- start readable, increase pressure gradually, introduce new variables one at a
  time

Player growth:

- define what improves, how often, and how it affects choices

Enemy scaling:

- define health, speed, damage, quantity, and special behavior ranges

Resource balance:

- define source/sink relationship, scarcity, reward value, and failure recovery

Numbers should be labeled placeholder until design intent and validation exist.

## 7. Level Design Rules

Tutorial level:

- teach one mechanic at a time
- minimize punishment
- make success condition obvious

Normal level:

- combine learned mechanics
- add pacing and optional mastery
- validate difficulty curve

Boss level:

- test mastery
- use clear phases and readable attacks
- provide strong feedback

Difficulty progression:

- increase complexity through mechanics, not only numbers
- avoid sudden unexplained spikes

## 8. UI Design Rules

HUD:

- show current objective, status, resources, and urgent feedback

Menus:

- keep navigation predictable and labels clear

Inventory:

- show item identity, quantity, rarity, and use state

Battle interface:

- prioritize action clarity, cooldown, target, health, and result feedback

Mobile readability:

- preserve touch target size, text contrast, responsive layout, and safe spacing

## 9. Asset Requirement Rules

AI converts gameplay requirements into asset requirements by asking:

- what must be visible
- what must move
- what must give feedback
- what must be reused
- what can remain placeholder
- what needs ownership and approval

Tower Defense required assets:

- scene assets
- character assets
- tower assets
- effect assets
- UI assets

Asset requirements must not trigger external asset import without approval.

## 10. Integration with Agents

Planner Agent queries knowledge to:

- classify game type
- choose template
- identify systems
- decompose tasks
- define approval checkpoints

Task Planning uses templates to:

- create task sequences
- identify dependencies
- map systems to capabilities
- prepare validation requirements

QA Agent validates design by checking:

- selected genre template fits the request
- required systems are scoped
- numbers are labeled placeholder or justified
- UI and asset requirements match gameplay needs
- no unapproved systems are introduced
