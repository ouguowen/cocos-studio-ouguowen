# v5.5 Capability Dataset System

Status: documentation only

Scope: AI Game Studio capability dataset architecture

This document defines the v5.5 Capability Dataset System for AI Game Studio. It
describes how capability metadata can be organized into future game-genre and
resource datasets without forcing AI systems to read the entire documentation
repository for every task.

This phase creates no `capabilities/*.json` files, dataset schema, compiler,
loader, search index, Agent implementation, or execution code. It does not
modify MCP, Agent runtime, Cocos projects, examples, or scripts.

## 1. System Positioning

v5.3 defines capability discovery. v5.4 defines the compact runtime manifest.
v5.5 defines the maintainable source datasets from which a future manifest can
be assembled.

```text
Capability Dataset Sources
  -> Dataset Validation
  -> Dataset Merge
  -> Capability Manifest
  -> Capability Search and Ranking
  -> Selective Context Loading
```

The Capability Dataset System separates authoring from runtime consumption:

- dataset files are human-maintained source records
- the manifest is the compact runtime projection
- canonical Markdown files remain the policy source of truth
- Agents consume selected capability context through existing handoff rules

### Primary goal

The primary goal is to reduce repeated full-document loading.

Instead of searching all docs to discover what a tower-defense, card-game, UI,
animation, or scene task needs, a future router first reads compact dataset
metadata and loads only the selected capability context.

### Non-goals

v5.5 does not define:

- gameplay implementation
- final game design for any genre
- asset production
- Agent behavior changes
- MCP behavior changes
- Cocos Creator project operations
- automatic dataset generation from documents
- automatic approval or permission escalation

## 2. Capability Dataset Structure

A capability dataset is a collection of normalized capability records grouped
by a stable subject, such as a game genre or resource domain.

Each record describes one bounded capability and the minimum metadata required
for search, ranking, dependency resolution, Agent routing, and context loading.

### Required capability fields

Every future dataset record includes:

| Field | Type | Purpose |
|---|---|---|
| `id` | string | Stable capability ID |
| `name` | string | Human-readable capability name inherited from v5.4 |
| `category` | string | Primary capability category |
| `keywords` | string array | Search and intent matching terms |
| `game_genres` | string array | Compatible game genre IDs |
| `required_agents` | string array | Existing Agent Registry IDs required for handoff |
| `context_files` | string array | Canonical repository-relative context files |
| `dependencies` | string array | Required capability IDs |
| `priority` | integer | Final ranking tie-breaker |

The eight fields requested by v5.5 are `id`, `category`, `keywords`,
`game_genres`, `required_agents`, `context_files`, `dependencies`, and
`priority`. The `name` field remains required for compatibility with the v5.4
manifest contract and human diagnostics.

### Conceptual record

This example is architecture documentation only. It is not written to a JSON
dataset in this phase.

```json
{
  "id": "genre.tower-defense.wave-plan",
  "name": "Tower Defense Wave Planning",
  "category": "gameplay",
  "keywords": ["tower defense", "wave", "lane pressure", "spawn schedule"],
  "game_genres": ["tower-defense"],
  "required_agents": ["cocos-game-designer", "cocos-config-designer"],
  "context_files": [
    "design/game-type-templates.md",
    "architecture/level-data-models.md"
  ],
  "dependencies": ["design.classify.game-type"],
  "priority": 80
}
```

The record identifies context and ownership. It does not authorize those Agents
to execute, write files, call MCP, or modify a Cocos project.

## 3. Dataset Field Rules

### `id`

`id` is globally unique across every future file under `capabilities/*.json`.

Recommended forms:

```text
genre.<genre-id>.<action>
resource.<resource-id>.<action>
core.<domain>.<action>
```

Examples:

```text
genre.tower-defense.wave-plan
genre.card-game.turn-resolution-plan
resource.animation.state-design
resource.ui.hud-plan
core.context.scope-review
```

Rules:

- use lowercase ASCII and dot-separated segments
- remain stable across file moves
- do not encode an Agent implementation or MCP provider
- do not encode approval state
- do not reuse a retired ID for a different capability
- dependency references use the exact global ID

### `name`

`name` is a concise display label.

Rules:

- describe one bounded outcome
- remain readable in ranking diagnostics
- do not serve as a dependency key
- do not claim permission or execution authority

### `category`

`category` identifies the primary capability domain.

Initial values may include:

- `planning`
- `gameplay`
- `architecture`
- `resource`
- `quality`
- `validation`
- `release`
- `context`

Genre identity does not belong in `category`; it belongs in `game_genres`.
Resource specialization can use `category: resource` while the capability ID and
dataset filename identify animation, UI, audio, character, or scene scope.

Rules:

- use one primary category
- route cross-domain prerequisites through `dependencies`
- do not use category as a permission level
- reject unknown categories until the future dataset schema is reviewed

### `keywords`

`keywords` provides compact request-matching signals.

Keywords may include:

- genre names and common aliases
- player actions
- content unit names
- requested artifact names
- production terminology
- resource terminology

Rules:

- values must be non-empty strings
- normalized duplicates are invalid
- keyword matching is case-insensitive where appropriate
- exact ID and artifact matches outrank loose keywords
- generic terms such as `game`, `asset`, or `scene` must not dominate ranking
- multilingual aliases may be included when reviewed
- keywords must not silently broaden the user request

### `game_genres`

`game_genres` lists the normalized genres for which the capability is relevant.

Rules:

- values must come from the approved genre vocabulary
- use an empty array only for truly genre-neutral capabilities
- do not use `all` as a shortcut
- hybrid games select multiple explicit genres
- primary and secondary genre order is significant when Project Memory provides
  that distinction
- resource capabilities may list several compatible genres or remain neutral
- genre compatibility is not proof that the capability is required

Examples:

```json
["tower-defense"]
```

```json
["rpg", "story-game"]
```

```json
[]
```

The empty array means no genre filter. It does not mean automatic loading for
every task.

### `required_agents`

`required_agents` references stable IDs from the existing Agent Registry.

Initial valid examples include:

- `cocos-producer`
- `cocos-game-designer`
- `cocos-architect`
- `cocos-dev`
- `cocos-qa`
- `cocos-skill-auditor`
- `cocos-runtime-validator`
- `cocos-ui-programmer`
- `cocos-config-designer`
- `cocos-scene-builder`
- `cocos-release-reviewer`

Rules:

- every value must resolve to a registered Agent ID
- order reflects preferred handoff order, not authority rank
- list only Agents needed for the capability outcome
- Agent availability does not imply approval
- a dataset cannot change an Agent's registered responsibility
- a missing required Agent blocks handoff or requires an approved alternative
- do not create Agent implementations from dataset entries

### `context_files`

`context_files` lists canonical Skill repository documents needed for the
capability.

Rules:

- use normalized repository-relative paths
- reject absolute paths and parent traversal
- do not reference external game projects
- keep the list narrow
- do not duplicate canonical policy in JSON
- do not list every file in a category
- preserve deterministic order
- paths grant no write authority
- task-specific files are added by the router, not embedded as permanent dataset
  context

### `dependencies`

`dependencies` lists capability IDs that must be resolved before handoff.

Rules:

- references must exist globally after all dataset shards are merged
- self-dependencies and cycles are invalid
- duplicates are invalid
- dependencies may cross dataset files
- dependency expansion is bounded
- a dependency does not automatically execute
- a dependency requiring broader scope produces an approval boundary

### `priority`

`priority` is a deterministic integer tie-breaker.

Recommended range:

```text
0 through 100
```

Rules:

- apply only after scope, safety, genre, artifact, keyword, and dependency checks
- do not treat priority as quality, permission, or confidence
- do not use high priority to force a broad capability into every context
- identical ranking results use stable `id` order after priority

## 4. Game Genre Capability Library

The genre capability library describes reusable planning and production
capabilities selected by dominant player loop, not by visual theme.

Initial genre vocabulary:

- `tower-defense`
- `card-game`
- `rpg`
- `platformer`
- `story-game`

These are routing IDs. The existing Game Classifier remains responsible for
determining the dominant loop and identifying hybrids.

### Genre selection contract

```text
Game Request
  -> Dominant Player Loop Classification
  -> Primary Genre ID
  -> Optional Secondary Genre IDs
  -> Genre Capability Dataset Search
  -> Selected Capability Set
```

Rules:

- classify before selecting genre-specific capabilities
- primary genre filters first
- secondary genres add only directly required capabilities
- shared resource capabilities remain separate from genre datasets
- do not load every capability for the selected genre
- do not force one level or config model onto every genre

## 10. Hybrid Genre Rules

Many games combine multiple genres. Dataset routing must preserve a primary and
secondary relationship rather than merging entire libraries.

Example:

```text
Primary: card-game
Secondary: rpg
```

The router should:

1. select capabilities required by the card-game core loop
2. add only requested RPG progression capabilities
3. deduplicate dependencies and context files
4. check the combined context budget
5. reject conflicting state ownership or scope

Forbidden hybrid behavior:

- load all capabilities from both genres
- treat visual theme as a secondary genre
- introduce economy or inventory automatically
- allow a secondary genre to replace the primary loop model
- use hybrid selection to bypass approval or MVP scope

## 11. Resource Capability Library

Resource capabilities are cross-genre capabilities for production assets and
their integration requirements.

Initial resource vocabulary:

- `animation`
- `ui`
- `audio`
- `character`
- `scene`

Resource capability datasets are independent of genre datasets. A request may
combine one genre capability with one or more resource capabilities without
loading unrelated resource domains.

```text
Selected Genre Capability
  + Selected Resource Capability
  -> Combined Dependency Check
  -> Combined Context Set
  -> Agent Handoff
```

Resources express presentation and production needs. They do not own gameplay
truth.

## 17. Future `capabilities/*.json` Data Standard

Future source datasets are stored as JSON shards under an approved
`capabilities/` directory.

Conceptual flat layout:

```text
capabilities/
  core.json
  tower-defense.json
  card-game.json
  rpg.json
  platformer.json
  story-game.json
  resource-animation.json
  resource-ui.json
  resource-audio.json
  resource-character.json
  resource-scene.json
```

No directory or JSON file is created in v5.5.

### File naming rules

- use lowercase kebab-case
- use one stable subject per file
- genre files use the normalized genre ID
- resource files use `resource-<resource-id>.json`
- do not include version numbers in filenames
- do not create one file per capability unless scale requires a reviewed change
- do not put executable JavaScript or comments in JSON files

### Top-level dataset structure

Each future shard uses:

```json
{
  "dataset_id": "genre.tower-defense",
  "dataset_version": "1.0.0",
  "dataset_type": "genre",
  "subject": "tower-defense",
  "capabilities": []
}
```

Resource example:

```json
{
  "dataset_id": "resource.animation",
  "dataset_version": "1.0.0",
  "dataset_type": "resource",
  "subject": "animation",
  "capabilities": []
}
```

Top-level required fields:

| Field | Type | Purpose |
|---|---|---|
| `dataset_id` | string | Stable globally unique dataset identity |
| `dataset_version` | string | Dataset content contract version |
| `dataset_type` | string | `core`, `genre`, or `resource` |
| `subject` | string | Normalized genre, resource, or core subject ID |
| `capabilities` | array | Capability records defined by this document |

### Dataset type rules

`core` contains genre-neutral routing, safety, memory, planning, validation, and
release capabilities.

`genre` contains capabilities whose relevance depends on a dominant or secondary
game loop.

`resource` contains cross-genre presentation, asset, and integration
capabilities.

A record belongs to the file that owns its primary subject. Cross-subject reuse
is expressed through `game_genres` and `dependencies`, not record duplication.

## 18. Dataset Validation Contract

A future dataset validator must validate every shard independently and the
merged dataset globally.

### Per-file validation

- valid JSON object
- required top-level fields present
- recognized `dataset_type`
- filename, dataset ID, type, and subject agree
- semantic dataset version is valid
- `capabilities` is an array
- every record contains required fields
- every field has the expected type
- no duplicate record ID inside the file
- priorities are integers in the accepted range
- paths are repository-relative and normalized

### Global validation

- dataset IDs are unique
- capability IDs are unique across all shards
- all dependencies resolve
- dependency graph contains no cycle
- all Agent IDs exist in Agent Registry
- all genre IDs exist in the genre vocabulary
- all context files exist and remain inside allowed roots
- no duplicate normalized keyword exists within one record
- no conflicting duplicate source owns the same capability
- merged output is deterministic

### Safety validation

- no dataset path points into an external Cocos project
- no dataset record grants write permission
- no dataset record grants MCP permission
- no dataset record alters Agent authority
- no docs-only capability includes runtime implementation as an implicit action
- no missing dependency causes automatic broad context loading

## 19. Dataset Merge and Runtime Projection

The future dataset merge produces the compact v5.4 runtime manifest.

```text
capabilities/*.json
  -> Parse Shards
  -> Validate Local Records
  -> Merge Global IDs
  -> Validate Dependencies and Agents
  -> Normalize and Sort
  -> Project Runtime Fields
  -> capabilities.json
```

The source datasets may contain dataset-level metadata and `game_genres` or
`required_agents`. The runtime manifest may retain those routing fields or
project a smaller search view, provided the information required for safe Agent
handoff remains available.

Merge rules:

- deterministic file order
- deterministic capability order by stable ID
- no last-write-wins conflict handling
- duplicate IDs fail validation
- record arrays are not merged implicitly
- dependencies are resolved only after all shards load
- canonical context file order is preserved
- source provenance is retained for diagnostics

No merge implementation or generated manifest is created in v5.5.

## 20. Dataset Search Flow

The future runtime uses dataset metadata before reading detailed docs.

```text
User Request
  -> Game Genre Classification
  -> Resource Intent Detection
  -> Dataset Shard Selection
  -> Capability Candidate Search
  -> Ranking and Dependency Resolution
  -> Context File Loading
  -> Agent Handoff
```

### Shard selection

Load only:

- `core.json` when core routing metadata is required
- the primary genre shard
- directly requested secondary genre shards
- directly requested resource shards

Do not load:

- every genre shard
- every resource shard
- every referenced Markdown file
- implementation context before an approved implementation request

### Candidate selection

Candidates are filtered by:

- explicit capability ID
- primary and secondary genre
- resource intent
- category
- artifact and operation match
- keywords
- dependency readiness
- current scope and safety boundary

Priority remains the final tie-breaker.

### Context result

The router loads only:

- selected capability context
- selected dependency context
- mandatory lightweight safety context
- task-specific files required by the user's requested output

This keeps context cost proportional to the selected capability set rather than
the total repository size.

## 21. Versioning and Change Governance

Each future dataset shard uses semantic versioning.

Conceptual rules:

- patch: keyword correction, label correction, or non-behavioral metadata repair
- minor: new backward-compatible capability or routing metadata
- major: removed capability, changed dependency contract, changed Agent
  requirement, changed context ownership, or incompatible schema change

Dataset changes require review for:

- routing impact
- context budget impact
- Agent ownership impact
- dependency graph impact
- safety impact
- migration impact

Runtime observations must not mutate datasets automatically. Proposed changes
must be reviewed and written through an approved maintenance scope.

## 23. Human Approval and Safety Boundary

Dataset presence does not equal execution approval.

Dataset matching does not authorize:

- loading unrelated context
- modifying Agent runtime
- calling MCP
- modifying a Cocos project
- importing assets
- creating scenes or prefabs
- installing dependencies
- changing Project Memory
- committing, pushing, tagging, or releasing

The future runtime must preserve:

- original user request
- approved repository and path
- allowed file classes
- forbidden actions
- selected Agent responsibilities
- unresolved dependencies
- context budget
- validation requirements
- human approval state

Stop before handoff when:

- genre classification is materially ambiguous
- the required dataset shard is invalid
- capability IDs conflict
- a dependency is missing or cyclic
- a required Agent ID is unknown
- context paths are invalid
- combined genre and resource context exceeds budget
- a dependency would expand scope
- an existing approval boundary is not satisfied

## 24. Future Implementation Contract

A separate approved implementation phase is required before creating the
`capabilities/` directory, JSON shards, schemas, validators, compilers, or runtime
integration.

That proposal must define:

- JSON Schema ownership and version
- exact directory layout
- allowed genre and resource vocabularies
- Agent Registry reference validation
- global ID registry behavior
- dependency graph limits
- context path allow lists
- merge and sort algorithm
- source provenance format
- manifest projection contract
- compatibility with v5.4 runtime search
- test strategy
- migration and rollback plan

Required future test families include:

- valid genre dataset allow path
- valid resource dataset allow path
- hybrid genre selection path
- unknown genre block path
- unknown Agent block path
- duplicate global ID block path
- missing dependency block path
- dependency cycle block path
- context path traversal block path
- deterministic merge path
- narrow shard loading path
- approval-required handoff path

No future implementation may be promoted until both allow and block paths pass.

## Phase Boundary

v5.5 accepts:

- Capability Dataset structure design
- genre capability library design
- resource capability library design
- `capabilities/*.json` source data specification
- dataset validation and merge contracts
- selective dataset and context loading design
- Agent reference and approval boundaries

v5.5 does not authorize:

- creating the `capabilities/` directory
- creating JSON dataset files
- creating a JSON Schema
- validator, compiler, loader, search, ranking, or runtime code
- MCP modification
- Agent runtime modification
- Cocos project modification
- examples modification
- scripts modification
- automatic dataset generation
- automatic capability execution
- automatic permission escalation

This phase ends with architecture documentation and repository validation only.
