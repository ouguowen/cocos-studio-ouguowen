# AI Game Studio v3.6 Game Template System Architecture

This document defines the Game Template System architecture for AI Game Studio.
It is architecture documentation only. It does not implement code, modify MCP,
modify Cocos Creator projects, create Agent runtime behavior, modify `agents/*`,
or change runtime execution logic.

## 1. Game Template System Goal

AI game development needs game type templates because different genres require
different gameplay loops, system architecture, asset pipelines, validation
targets, and production order.

The Game Template System exists to:

- classify a user idea into a game type
- select an appropriate production template
- prevent one universal workflow from being forced onto every game
- connect genre needs to Agents and Capabilities
- generate a scoped development plan before implementation
- preserve Fast Build Mode by loading only the selected template
- keep human approval between template selection and real project modification

The template system is a planning and orchestration layer. It does not generate
runtime code or create Cocos project files by itself.

## 2. Game Type Template Model

Each game template must define the core loop, required Agents, required
Capabilities, and validation expectations.

Required fields:

- Template ID
- Game Type
- Gameplay Loop
- Required Agents
- Required Capabilities
- Required Systems
- Asset Requirements
- UI Requirements
- Validation Requirements
- Version
- Status

Example:

```text
Template ID:
  template.tower-defense.basic

Game Type:
  Tower Defense

Gameplay Loop:
  prepare -> place tower -> spawn wave -> defend base -> earn reward -> upgrade

Required Agents:
  Planner Agent
  Code Agent
  Asset Agent
  QA Agent

Required Capabilities:
  tower-system-design
  enemy-wave-system
  path-validation
  tower-defense-asset-plan
  gameplay-loop-validation
```

Template rules:

- Template ID must be stable and unique.
- Game Type must describe the genre family.
- Gameplay Loop must define the core repeated player action.
- Required Agents must map to Agent Registry identities or Agent categories.
- Required Capabilities must map to Capability Marketplace entries.
- Templates must not claim implementation is complete.
- Templates must not authorize project modification without approval.

## 4. Template Generation Flow

Template selection starts from user intent and ends with a development plan, not
with automatic implementation.

```text
User Requirement
      |
      v
Planner Agent
      |
      v
Select Template
      |
      v
Load Capabilities
      |
      v
Generate Development Plan
```

Flow details:

1. User Requirement: describes the desired game, platform, constraints, and
   target scope.
2. Planner Agent: classifies game type, identifies risk, and proposes candidate
   templates.
3. Select Template: human reviews and confirms the selected template when it
   affects project direction.
4. Load Capabilities: only directly required capabilities are loaded from the
   Capability Marketplace.
5. Generate Development Plan: Planner Agent creates tasks, dependencies, Agent
   assignments, validation targets, and approval gates.

Development plan output:

- selected template
- MVP scope
- required systems
- task graph
- required Agents
- required Capabilities
- expected files or project areas
- validation plan
- stop conditions

## 5. Template and Capability Relationship

Templates define what capabilities are needed for a game type. Capabilities
define how a specific unit of work is planned, reviewed, or validated.

```text
Game Template
      |
      v
Required Capabilities
      |
      v
Capability Marketplace
      |
      v
Agent Assignment
```

Relationship rules:

- a template may reference many capabilities.
- a capability may be reused by many templates.
- capabilities stay trigger-only unless the selected template needs them.
- beta or deprecated capabilities require explicit human approval.
- missing capabilities must produce a planning blocker, not force ad hoc work.
- template selection does not install or execute capabilities automatically.

Example:

```text
Tower Defense Template
  -> enemy-wave-system
  -> tower-system-design
  -> path-validation
  -> tower-defense-asset-plan
```

## 6. Template and Agent Registry Relationship

Templates use the Agent Registry to identify who should own each stage of work.

```text
Selected Template
      |
      v
Required Agents
      |
      v
Agent Registry
      |
      v
Available Agents
      |
      v
Multi-Agent Handoff
```

Agent mapping examples:

- Planner Agent: template selection, task decomposition, approval packet.
- Design Agent: game loop, system design, player experience.
- Code Agent: code task planning and implementation proposal.
- Asset Agent: asset requirement planning.
- QA Agent: validation and acceptance review.
- Release Agent: release checklist and publication review.

Registry rules:

- required Agents must exist in the Agent Registry or produce a blocker.
- unavailable Agents cannot receive tasks.
- Agent capability must match the selected template requirement.
- Agent availability is not human approval.
- real project modification still requires explicit approved scope.

## 7. Template Version Management

Template status values:

- stable
- beta
- deprecated

Stable templates:

- have clear gameplay loop
- have required systems
- have required Agents
- have required Capabilities
- have validation expectations
- may be recommended for normal approved planning

Beta templates:

- experimental
- require explicit human approval before use
- may have incomplete validation coverage
- must not be default-loaded in Fast Build Mode

Deprecated templates:

- replaced, outdated, or no longer recommended
- must not be selected automatically
- may remain for historical or compatibility reference

Version fields:

- template id
- version
- status
- replacement template if deprecated
- change reason
- validation coverage
- required capability version range

Version rules:

- patch version: wording, examples, or documentation updates
- minor version: added optional system, capability, or validation step
- major version: changed core gameplay loop, required Agents, required
  Capabilities, or permission boundary

## 8. Human Approval Mechanism

Human approval remains required before template-driven project work begins.

Approval is required before:

- selecting a template as the active project direction
- loading beta or deprecated templates
- loading beta or deprecated capabilities through a template
- modifying project files from a template-generated plan
- using Cocos MCP or browser preview for project validation
- writing template-derived decisions into Project Memory
- committing, pushing, tagging, or releasing template-related changes

Human review packet:

- selected template
- game type
- gameplay loop
- required Agents
- required Capabilities
- MVP scope
- task graph
- project path when relevant
- allowed files
- forbidden files
- validation plan
- rollback plan
- stop conditions

Safety rules:

- template selection is not implementation approval.
- capability loading is not project modification approval.
- Agent assignment is not approval.
- human confirmation remains required before real project changes.

## 9. Integration With Existing Architecture

v3.5 Capability Marketplace:

- provides capability metadata, version status, dependencies, loading rules, and
  permission boundaries used by templates.

v3.4 Agent Registry:

- provides Agent identity, role, capability, version, and availability for
  template-required work.

v3.3 Multi-Agent Protocol:

- provides message format, handoff rules, state synchronization, collaboration
  rules, and human approval boundaries.

Integrated flow:

```text
User Requirement
      |
      v
Game Template System
      |
      v
Capability Marketplace
      |
      v
Agent Registry
      |
      v
Multi-Agent Protocol
      |
      v
Human Approval
      |
      v
Approved Development Plan
```

The Game Template System connects genre-specific planning to controlled
capability and Agent routing without creating autonomous execution.
