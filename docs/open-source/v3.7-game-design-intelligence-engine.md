# AI Game Studio v3.7 Game Design Intelligence Engine

This document defines the Game Design Intelligence Engine architecture for AI
Game Studio. It is architecture documentation only. It does not implement code,
modify MCP, modify Cocos Creator projects, create Agent runtime behavior, modify
`agents/*`, or change runtime execution logic.

## 1. Game Design Intelligence Goal

Game Design Intelligence is the layer that helps AI Game Studio understand what
a game idea means before it becomes tasks, code, assets, scenes, or validation
work.

The engine helps AI understand:

- what genre the game belongs to
- what gameplay loop the player repeats
- what systems are required
- what rules govern player action and game state
- what balance constraints affect difficulty and pacing
- what levels, UI, and assets are required
- what validation should prove before implementation continues

The goal is not to invent unlimited features. The goal is to convert a user idea
into a scoped, reviewable, genre-aware production plan.

## 2. Game Design Knowledge Model

The Game Design Knowledge model describes reusable design knowledge that can be
queried by Planner Agent, Game Template System, Capability Marketplace, and QA
flows.

### Game Genre Knowledge

Defines:

- genre identity
- expected player loop
- common systems
- common camera or scene structure
- common UI requirements
- common asset categories
- common validation risks

Examples:

- Tower Defense: path, wave, tower, enemy, objective, reward.
- RPG: character, dialogue, quest, inventory, progression.
- Card Game: deck, hand, turn, effect, battle result.

### Gameplay Rules

Defines:

- player actions
- allowed state changes
- success conditions
- failure conditions
- reward rules
- progression rules

Gameplay rules must define gameplay truth. UI, animation, and assets may present
gameplay, but they must not own final game state.

### System Design Rules

Defines:

- required modules
- ownership boundaries
- dependency direction
- data flow
- scene/script responsibility
- config separation
- validation surface

System rules prevent large, fragile god scripts and keep each game type from
being forced into one universal architecture.

### Balance Rules

Defines:

- difficulty curve
- enemy scaling
- resource income and spending
- player growth
- level pacing
- risk and reward

Balance rules may be placeholders during prototype work, but they must be named
as placeholders and reviewed before production balance claims.

### Level Design Rules

Defines:

- tutorial level shape
- normal level shape
- boss or challenge level shape
- pacing rules
- progression gates
- objective clarity
- failure feedback

Level design must fit the selected game type. A path-based tower defense map, a
platformer level, and an RPG exploration map require different models.

### UI Design Rules

Defines:

- HUD requirements
- menu requirements
- panel hierarchy
- feedback state
- mobile readability
- interaction flow
- result presentation

UI design rules must clarify gameplay state without replacing gameplay logic.

### Asset Requirement Rules

Defines:

- required character assets
- required environment assets
- required UI assets
- required animation assets
- required effect assets
- required audio assets
- naming and consistency expectations

Asset requirements convert design intent into production-ready asset planning
without importing or generating assets automatically.

## 3. Game Type Knowledge

### Tower Defense

Core knowledge:

- loop: prepare -> place towers -> spawn wave -> defend base -> reward -> upgrade
- systems: map path, tower placement, enemy movement, wave spawn, objective,
  reward, result
- balance: wave pacing, enemy health, tower cost, tower damage, reward cadence
- level design: path clarity, spawn points, base location, tower zones
- UI: wave counter, resource display, base health, tower selection
- assets: towers, enemies, projectiles, map tiles, hit effects, HUD icons

### RPG

Core knowledge:

- loop: explore -> interact -> fight or solve -> collect -> upgrade -> progress
- systems: character state, dialogue, quest, inventory, combat or interaction,
  progression
- balance: character growth, item value, enemy difficulty, reward pacing
- level design: exploration routes, NPC placement, quest flow, safe zones
- UI: dialogue panel, inventory, character stats, quest tracker
- assets: characters, portraits, environments, items, skill effects

### Card Game

Core knowledge:

- loop: build deck -> draw -> play card -> resolve effect -> end turn -> result
- systems: card library, deck, hand, turn manager, battle state, effect resolver
- balance: card cost, draw rate, deck size, effect power, turn tempo
- level design: encounter rules, opponent deck, reward progression
- UI: card hand, battle board, resource counter, effect preview
- assets: card art, card frames, icons, battle backgrounds, effects

### Strategy

Core knowledge:

- loop: inspect state -> plan -> allocate resources -> command -> resolve ->
  adapt
- systems: unit/entity management, command queue, resource model, turn or tick
  loop, AI rules, result evaluation
- balance: resource income, unit cost, timing, map control, risk/reward
- level design: terrain, chokepoints, resource nodes, strategic objectives
- UI: dashboard, command panel, map view, status overlays
- assets: units, buildings, terrain, icons, indicators

### Action

Core knowledge:

- loop: move -> attack -> avoid -> use skill -> defeat enemies -> clear
  encounter
- systems: character control, enemy behavior, combat input, skill state, hit
  feedback, objective result
- balance: damage, cooldown, enemy count, arena size, animation timing
- level design: encounter spacing, enemy waves, hazards, cover or movement lanes
- UI: health, skill buttons, feedback markers, result state
- assets: characters, enemies, animation clips, skill effects, hit effects

## 4. Design Reasoning Flow

Design reasoning turns user intent into a scoped production plan.

```text
User Requirement
      |
      v
Game Template
      |
      v
Design Knowledge
      |
      v
Production Plan
```

Reasoning stages:

1. User Requirement: capture game idea, platform, target scope, constraints, and
   user priorities.
2. Game Template: select the closest genre template through v3.6 Game Template
   System.
3. Design Knowledge: load only the genre and system knowledge required by the
   selected template.
4. Production Plan: generate MVP boundary, required systems, task graph,
   required capabilities, validation plan, and approval gates.

Design output:

- selected game type
- core gameplay loop
- required systems
- MVP scope
- required Agents
- required Capabilities
- asset requirements
- level requirements
- UI requirements
- validation requirements
- open risks

The output is a proposal. It is not automatic implementation approval.

## 5. Relationship With Capability Marketplace

The Game Design Intelligence Engine identifies what capabilities are needed.
The Capability Marketplace provides metadata, version status, dependencies, and
loading rules for those capabilities.

```text
Design Knowledge
      |
      v
Required Capability List
      |
      v
Capability Marketplace
      |
      v
Capability Proposal
```

Examples:

- Tower Defense design may request `enemy-wave-system`, `tower-system-design`,
  and `path-validation`.
- RPG design may request `dialogue-flow-planning`, `quest-state-model`, and
  `inventory-architecture`.
- Card Game design may request `card-schema-design`, `turn-flow-planning`, and
  `effect-resolution-validation`.

Rules:

- design reasoning may recommend capabilities.
- capability loading requires the marketplace rules.
- beta or deprecated capabilities require explicit approval.
- capability discovery is not project modification approval.

## 6. Relationship With Agent Registry

The Game Design Intelligence Engine identifies which Agent roles are needed. The
Agent Registry selects available Agents with matching capabilities.

```text
Design Requirement
      |
      v
Required Agent Roles
      |
      v
Agent Registry
      |
      v
Available Agent Candidates
```

Typical role mapping:

- Planner Agent: design reasoning and task decomposition.
- Game Designer Agent: gameplay loop and system design.
- Code Agent: code task planning and implementation proposal.
- Asset Agent: asset requirement planning.
- QA Agent: validation strategy and review.

Rules:

- design intelligence may recommend Agent roles.
- Agent Registry verifies identity, status, version, and capability boundary.
- Agent availability is not human approval.
- missing Agent capability must produce a planning blocker.

## 7. Relationship With Project Memory

Project Memory provides approved project context and historical decisions. Game
Design Intelligence may use this memory only when explicitly triggered and
approved.

Memory may provide:

- game identity
- selected game type
- past design decisions
- accepted or rejected features
- style rules
- asset rules
- validation history
- known project constraints

Memory use rules:

- no default project memory loading in Fast Build Mode
- no real project facts without approved project path and scope
- no memory write without human confirmation
- historical experience is context, not automatic approval

Example:

```text
Project Memory:
  Tower Defense project chose path-based map model.

Design Intelligence:
  Avoids recommending free-roaming map architecture unless user requests a
  redesign.
```

## 8. Human Approval Mechanism

Human approval remains the boundary between design reasoning and project action.

Approval is required before:

- selecting a template as active project direction
- loading project-specific memory
- loading beta or deprecated capabilities
- converting production plan into file changes
- using Cocos MCP
- writing design decisions into Project Memory
- committing, pushing, tagging, or releasing

Approval packet:

- user requirement summary
- selected game type
- selected template
- design reasoning summary
- required systems
- required capabilities
- required Agents
- MVP boundary
- validation plan
- project path when relevant
- allowed files when relevant
- forbidden files
- rollback plan
- stop conditions

Human review prevents AI from turning design reasoning into uncontrolled
development.

## 9. Integration With Existing Architecture

v3.6 Game Template System:

- provides template selection, required systems, required Agents, and required
  Capabilities.

v3.5 Capability Marketplace:

- provides capability metadata, dependencies, version status, loading rules, and
  permission boundaries.

v2.8 Game Knowledge Engine:

- provides base genre knowledge, gameplay templates, system rules, numerical
  design rules, level rules, UI rules, and asset requirement rules.

v2.10 Project Memory:

- provides approved project-specific decisions and validation history when
  explicitly triggered.

Integrated flow:

```text
User Requirement
      |
      v
Game Design Intelligence
      |
      v
Game Template System
      |
      v
Capability Marketplace
      |
      v
Agent Registry / Multi-Agent Protocol
      |
      v
Human Approval
      |
      v
Production Plan
```

The design intelligence layer improves planning quality while preserving
explicit approval and non-default project inspection.
