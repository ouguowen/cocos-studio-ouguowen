# AI Game Studio v3.0 Code Generation Orchestration

This document defines the Code Generation Orchestration architecture for AI Game
Studio. It is documentation only. It does not implement a code generator, create
runtime files, modify `agents/*`, change MCP, modify Cocos Creator project
files, or authorize autonomous execution.

## 1. Code Generation Orchestration Architecture

Code Generation Orchestration coordinates game-development tasks between AI Game
Studio planning, Codex execution, Cocos MCP validation, and QA review.

Target architecture:

```text
AI Game Studio
        |
        v
Task Planner
        |
        v
Code Generation Orchestrator
        |
        v
Codex Executor
        |
        v
Cocos MCP Adapter
        |
        v
Cocos Creator 3.8.8
```

Responsibilities:

- AI Game Studio owns product intent, safety boundary, project context, and
  approval flow.
- Task Planner converts game requirements into scoped development tasks.
- Code Generation Orchestrator sequences approved tasks and sends them to Codex.
- Codex Executor performs approved code or documentation edits inside file scope.
- Cocos MCP Adapter validates Cocos scene, asset, component, prefab, and runtime
  state when approved.
- QA Agent reviews validation, diffs, runtime proof, and blocker state.

The Orchestrator proposes and coordinates. It does not self-approve execution.

## 2. Code Task Planning Design

Game requirements must be decomposed into development tasks before code
generation.

Task fields:

- Task ID
- Task type
- Task dependency
- File target
- Validation requirement
- Approval status
- Rollback expectation

Example user request:

```text
Create tower defense game
```

Converted tasks:

- `T001 GameManager`
- `T002 Tower System`
- `T003 Enemy System`
- `T004 Wave System`

Example task plan:

```text
task_id: T002
task_type: gameplay_system
dependency: T001
file_target: assets/scripts/tower-defense/TowerSystem.ts
validation_requirement: TypeScript check, Cocos component binding review, runtime proof if scene-bound
approval_status: pending
```

Task planning must keep broad game requests from becoming unbounded code
generation.

## 3. Codex Integration Layer

The Codex Integration Layer defines how AI Game Studio may call Codex for
approved work.

Task dispatch:

- send one approved task at a time
- include allowed files and forbidden files
- include expected output
- include validation commands
- include stop conditions

Execution result:

- changed files
- generated files
- validation output
- blockers
- unexpected files
- suggested repair scope

Error feedback:

- failed command
- error message
- suspected cause
- whether retry needs approval
- whether rollback is needed

Retry strategy:

- retry only the smallest failing task
- do not broaden scope without approval
- do not use dangerous Git commands
- preserve user changes
- escalate to human review after repeated failure

Codex is an executor inside approved scope, not an autonomous product owner.

## 4. Cocos MCP Validation Flow

After code generation, approved Cocos validation may follow this path:

```text
Codex
  |
  v
Cocos MCP
  |
  v
Scene validation
  |
  v
Runtime preview
  |
  v
QA Agent
```

Validation may include:

- scene hierarchy inspection
- component binding checks
- prefab reference checks
- asset reference checks
- runtime preview proof
- browser/runtime console review
- unexpected file detection

Cocos MCP validation requires explicit approval, active project path, proof
target, and allowed operation scope.

## 5. Game Template Registry

The Game Template Registry maps game type to architecture, file structure, and
task sequence.

Tower Defense:

- system architecture: GameManager, Map, Tower, Enemy, Wave, Projectile,
  Objective, Result
- file structure: tower-defense scripts, config, prefabs, scenes, UI
- development tasks: map path, tower placement, enemy spawn, wave loop,
  objective proof

Card Game:

- system architecture: Card Library, Deck, Hand, Turn, Battle, Effects, Result
- file structure: card scripts, effect configs, UI panels, battle scene
- development tasks: card schema, hand UI, turn flow, battle resolution

RPG:

- system architecture: Character, Inventory, Dialogue, Quest, Combat,
  Progression
- file structure: character scripts, dialogue data, item configs, scenes, UI
- development tasks: character controller, inventory model, dialogue flow,
  quest state

Platformer:

- system architecture: Player Controller, Level, Hazard, Enemy, Collectible,
  Checkpoint, Result
- file structure: platformer scripts, tile/map data, prefabs, animation states
- development tasks: movement, collision, level flow, checkpoint, result proof

Game type affects:

- system architecture
- file structure
- development task order
- validation requirements
- runtime proof target

## 6. Human Approval Gate

Human approval flow:

```text
AI Planning
      |
      v
Human Approval
      |
      v
Codex Execution
      |
      v
Cocos Validation
```

Approval is required before:

- code generation
- file creation or modification
- Cocos MCP validation
- scene, prefab, asset, metadata, package, build, or settings changes
- runtime preview
- retry that changes scope
- staging, committing, pushing, tagging, or releasing

`next recommended command` is optional advice only and never approval.

## 7. Relationship With Existing Modules

v1.x Agent Architecture:

- provides Agent identity, capability, permission, workflow, and QA governance
- v3.0 uses these boundaries before dispatching tasks to Codex

v2.x Cocos MCP Layer:

- provides adapter, connection proof, capability mapping, task planning,
  execution control, and Cocos automation architecture
- v3.0 uses this layer for validation and future Cocos execution

v2.8 Game Knowledge Engine:

- informs game-type templates, gameplay systems, level needs, UI expectations,
  and asset requirements

v2.9 Asset Pipeline Engine:

- informs asset-related code tasks, prefab needs, animation needs, and validation
  targets

v2.10 Project Memory Engine:

- provides approved project identity, design decisions, asset memory, code
  structure, and Agent history when explicitly triggered

v2.11 Game Validation Framework:

- defines validation domains for code, assets, scenes, gameplay, and UI
- QA Agent uses it to review generated work

## Safety Boundary

This document creates:

- no runtime implementation
- no code generator
- no MCP changes
- no Cocos project changes
- no autonomous execution
- no Agent implementation

Human approval remains required before execution, validation, project changes,
Git actions, and release actions.
