# AI Game Studio v3.1 Code Review Agent Architecture

This document defines the future Code Review Agent architecture for AI Game
Studio. It is documentation only. It does not create runtime files, implement a
Code Review Agent, modify `agents/*`, change MCP, modify Cocos Creator
projects, or execute automatic code repair.

## 1. Code Review Agent Architecture

The Code Review Agent is the quality feedback layer between generated code and
the next Codex revision. It reviews code after generation, produces a structured
report, and routes issues back through the Code Generation Orchestrator.

```text
Codex
  |
  v
Generated Code
  |
  v
Code Review Agent
  |
  v
Review Report
  |
  v
Codex Feedback Loop
```

Position in AI Game Studio:

- Codex creates or modifies code only after approved scope.
- Code Review Agent evaluates the generated code against project architecture,
  Cocos Creator 3.8.8 rules, game design intent, and validation expectations.
- Review Report records pass, warning, or failure status with actionable issues.
- Codex Feedback Loop uses the report to propose a small repair task, not to
  self-modify code without human approval.

The Code Review Agent is a reviewer and feedback producer. It is not an
autonomous fixer and does not bypass approval gates.

## 2. Review Pipeline Architecture

Code review runs after code generation and before merge or release approval.

```text
Code Generated
      |
      v
Static Analysis
      |
      v
Architecture Review
      |
      v
Cocos Engine Review
      |
      v
Performance Review
      |
      v
Game Logic Review
      |
      v
Final Report
```

Pipeline stages:

- Static Analysis: checks TypeScript syntax, local linting expectations, imports,
  duplicate code, and obvious bug patterns.
- Architecture Review: checks module boundaries, data flow, ownership, coupling,
  and whether the implementation matches the approved task.
- Cocos Engine Review: checks Cocos Creator 3.8.8 component lifecycle, node use,
  prefab expectations, asset references, and component binding risks.
- Performance Review: checks update loops, allocation patterns, memory pressure,
  event cleanup, and object lifetime.
- Game Logic Review: checks whether behavior matches the Game Design Document,
  numerical rules, gameplay state ownership, and result conditions.
- Final Report: produces a structured PASS, WARNING, or FAILED result with issue
  ids and repair guidance.

The pipeline may be manual, semi-automated, or future-agent-assisted, but it
must remain scoped and auditable.

## 3. Review Categories

### A. Code Quality Review

Checks:

- naming convention
- modularity
- maintainability
- duplicate code
- function and class responsibility
- error handling
- readability
- testability

Quality questions:

- Does the file do one clear job?
- Are names aligned with the game domain?
- Is shared behavior extracted only when it reduces real complexity?
- Can a future developer understand the change quickly?

### B. Cocos Creator 3.8.8 Review

Checks:

- Component lifecycle: `onLoad`, `onEnable`, `start`, `update`, `onDisable`,
  and `onDestroy` usage.
- Node usage rules: node lookup, parent-child assumptions, active state, and
  transform ownership.
- Prefab usage rules: instantiation, dependency binding, reference stability,
  and prefab ownership.
- Asset reference rules: SpriteFrame, Prefab, AnimationClip, AudioClip, config,
  and resource path assumptions.
- TypeScript rules: decorators, properties, strict typing, imports, and Cocos
  API compatibility.

Cocos-specific review must watch for logic hidden in UI callbacks, animation
events, or unmanaged scene scripts.

### C. Performance Review

Checks:

- `update` overuse
- unnecessary per-frame polling
- object creation inside loops
- memory leak risk
- garbage collection pressure
- listener cleanup
- timer cleanup
- repeated asset loading
- large hierarchy traversal
- excessive dynamic node creation

Performance review should classify findings by current stage. A prototype may
accept small placeholders, but production code must control lifetime, allocation,
and per-frame cost.

### D. Game Architecture Review

Checks:

- system coupling
- data flow design
- module boundaries
- ownership of gameplay truth
- extension ability
- config separation
- scene/script responsibility split
- dependency direction

Architecture review should reject god scripts, hidden global state, and systems
where UI, animation, or scene glue owns final gameplay truth.

### E. Gameplay Review

Checks:

- alignment with the Game Design Document
- numerical logic
- game rule consistency
- objective and result flow
- player feedback requirements
- failure and success states
- progression assumptions

Gameplay review asks whether the code implements the approved rule, not whether
it invents a new feature.

## 4. Review Report Format

Standard review output:

```text
Review Result:

status:
  PASS / WARNING / FAILED

summary:
  Short reviewer summary.

Issues:
  - issue id
  - severity
  - location
  - description
  - suggestion

validation:
  commands_run
  runtime_proof
  blocked_checks

approval:
  human_review_required
  recommended_next_action
```

Issue fields:

- issue id: stable id such as `CR-001`
- severity: LOW, MEDIUM, HIGH, BLOCKER
- location: file, class, function, or line when available
- description: what is wrong
- suggestion: smallest safe repair direction
- category: code quality, Cocos, performance, architecture, gameplay

Example issue:

```text
Issue:

ID:
  CR-001

Severity:
  HIGH

File:
  Enemy.ts

Problem:
  update creates temporary objects every frame.

Suggestion:
  Move reusable data outside update and use object pooling for repeated enemy
  objects.
```

Status rules:

- PASS: no blocking issues and validation expectations are satisfied.
- WARNING: non-blocking issues exist, but the change can proceed after human
  review.
- FAILED: blocking issue, scope mismatch, missing validation, unsafe Cocos
  pattern, or gameplay rule violation.

## 5. Codex Feedback Loop

The Code Review Agent feeds structured issues back into code generation through
the orchestrator.

```text
Code Review Agent
      |
      v
Issue Report
      |
      v
Code Generation Orchestrator
      |
      v
Codex
      |
      v
New Version Code
      |
      v
Review Again
```

Feedback loop rules:

- Review Agent reports issues, but does not directly edit code.
- Code Generation Orchestrator converts accepted issues into scoped repair
  tasks.
- Human approval is required before repair execution.
- Codex receives only the approved repair scope.
- New code must pass through review again before merge.
- Repeated failure must produce a blocker report instead of broadening scope.

Feedback payload:

- original task id
- review status
- issue ids
- repair scope proposal
- allowed files
- forbidden files
- validation commands
- rollback notes

## 6. Integration With Existing Architecture

v1.x Agent Architecture:

- provides Agent identity, permission policy, capability boundaries, handoff
  rules, and human approval gates for the Code Review Agent role.

v2.8 Game Knowledge Engine:

- supplies genre rules, gameplay loops, design constraints, and system templates
  so gameplay review can compare code against intended game design.

v2.10 Project Memory Engine:

- supplies approved project identity, architecture state, decision history,
  asset rules, code structure memory, and previous review results when
  explicitly triggered.

v2.11 Game Validation Framework:

- supplies validation domains for code, assets, scenes, gameplay, UI, and
  performance risk. Code Review Agent uses it as the quality standard.

v3.0 Code Generation Orchestration:

- provides task dispatch, Codex execution, error feedback, retry strategy, and
  validation flow. Code Review Agent plugs into the post-generation review step.

v3.0.5 Game Production Workflow:

- places Code Review Agent inside the Testing, Integration, Optimization, and
  Release phases of the full production lifecycle.

Together, these modules create a closed development loop:

```text
Plan -> Generate -> Review -> Approve Repair -> Validate -> Report
```

The loop remains controlled by explicit scope, validation evidence, and human
approval.

## 7. Human Approval Workflow

Code review must not become automatic code modification.

```text
Code Generated
      |
      v
Review Result
      |
      v
Human Approval
      |
      v
Merge
```

Human approval is required before:

- applying a review suggestion
- changing code in response to review
- broadening repair scope
- changing scene, prefab, asset, metadata, package, build, or settings files
- committing, pushing, tagging, or releasing

Human reviewer receives:

- review status
- issue list
- severity
- affected files
- suggested repair scope
- validation status
- merge recommendation

AI must not modify code without explicit confirmation. Review output is guidance
and evidence, not execution permission.
