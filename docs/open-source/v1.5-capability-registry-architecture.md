# v1.5 Capability Registry Architecture

This document prepares the future Capability Registry architecture for AI Game
Studio. It is documentation only. It does not create runtime code, implement
Agents, execute MCP, automate Cocos, or modify real Cocos project files.

## 1. Capability Registry Architecture

The Capability Registry is the future index of what the Skill, Agents, and
approved workflows can help with. It should support discovery and routing
without granting permission by itself.

The registry should describe:

- capability id
- capability name
- capability owner
- supported task types
- required context
- required permissions
- validation expectation
- lifecycle status
- version

The registry must be trigger-controlled and not loaded by default in Fast Build
Mode.

## 2. Agent Capability Model

Agent capabilities describe what a future Agent role may help with after human
approval.

Agent capability fields:

- agent role
- capability id
- allowed inputs
- expected outputs
- permission requirements
- stop conditions
- validation requirements

Agent capability does not mean Agent autonomy. It does not grant file access,
project inspection, Cocos/MCP use, memory access, or runtime authority.

## 3. Skill Capability Model

Skill capabilities describe what the Skill can guide, review, or validate.

Skill capability fields:

- command or workflow name
- supported game-development area
- required context category
- approval gate
- proof requirement
- rollback expectation
- output format

Skill capabilities should remain separate from runtime implementation. The Skill
may guide a task without executing it.

## 4. Game Development Capability Categories

Capability categories:

- gameplay: core loop, rules, objectives, interaction, and result flow
- level design: level goals, pacing, difficulty, config, and acceptance criteria
- map generation: map model selection, layout constraints, route logic, and map
  validation
- character design: character identity, role, stats intent, and presentation
  boundaries
- animation: state, trigger, exit rule, fallback, and proof requirement
- UI design: screen purpose, controls, feedback, navigation, and data binding
- audio: music, sound effects, trigger rules, ownership, and import boundaries
- Cocos integration: scene, prefab, component, resource, and runtime proof
  guidance
- testing: validation commands, QA review, runtime proof, regression checks, and
  blocker reporting
- optimization: performance, context size, runtime safety, and technical debt
  review

Categories organize discovery. They do not authorize execution.

## 5. Capability Discovery

Capability discovery should answer:

- what capability matches the user request
- whether it belongs to core or extension
- what context is required
- what permission is required
- whether runtime proof is needed
- what output should be returned

Discovery must prefer the smallest safe capability and avoid loading broad
extension context unless directly triggered.

## 6. Capability Matching

Matching inputs:

- user intent
- operation mode
- task type
- approved project path, if any
- requested files
- forbidden files
- required proof type
- context category

Matching output:

- selected capability
- required context
- required approval
- expected output
- stop conditions

If multiple capabilities match, choose the narrowest one that satisfies the task
and safety requirements.

## 7. Permission Control

Capability is not permission.

The control chain is:

```text
Capability Match
        |
        v
Permission Check
        |
        v
Human Approval
        |
        v
Action or Proposal
```

Permission control must protect:

- real Cocos project files
- Cocos/MCP access
- browser preview and runtime console
- memory read/write
- package, CI, release, and protocol files
- staging, commits, pushes, tags, and releases
- broad cleanup or deletion

No capability may bypass human approval.

## 8. Capability Versioning

Each capability should have version metadata:

- version
- status: draft, active, suspended, retired
- owner
- last reviewed date or release marker
- validation coverage
- related docs
- compatibility notes

Versioning prevents stale capabilities from silently controlling future tasks.

## 9. Relationship Between Capability and Context

Capabilities select context; context does not select capabilities by itself.

The Capability Registry should define the minimum context required for each
capability:

- core context for common safety and routing
- task context for current approved work
- optional context for extensions, examples, proof, or deep docs
- memory context only when explicitly approved

Context loading must remain narrow and trigger-controlled.

## 10. Future Runtime Implementation Roadmap

Possible future roadmap:

1. keep capability registry architecture documentation-only
2. define a static capability schema
3. add validation checks for capability metadata
4. add read-only capability discovery command
5. connect capability matching to CLI proposals
6. add permission checks before any write-capable command
7. add Agent capability mapping after Agent governance is validated
8. add Cocos integration capabilities only after separate runtime architecture
   review

## Safety Boundary

This document does not create:

- runtime code
- Agent implementation
- MCP execution
- Cocos automation
- capability database
- real project capability records

The Capability Registry remains future architecture only.
