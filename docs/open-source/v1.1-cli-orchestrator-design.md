# v1.1 CLI Orchestrator Design

This document defines a future CLI Orchestrator architecture for AI Game Studio.
It is documentation only. It does not create Agents, CLI runtime, MCP execution,
Cocos automation, runtime files, or real Cocos project changes.

## 1. CLI Command Architecture

Future commands:

```text
cocos-studio init
cocos-studio design
cocos-studio build
cocos-studio validate
cocos-studio agent status
```

`cocos-studio init` prepares a controlled workspace or project session. It
should verify project identity, repository status, active path, and default
safety boundaries before any work begins.

`cocos-studio design` routes game ideas, feature requests, or content plans into
scoped design outputs. It should classify the task, select the relevant design
context, and produce an approval-ready plan.

`cocos-studio build` prepares an implementation proposal for an approved task.
It should not write by default. It should produce an execution proposal with
allowed files, forbidden files, validation commands, and rollback plan.

`cocos-studio validate` runs approved validation checks such as documentation
validation, npm checks, diff checks, or runtime proof collection when explicitly
authorized.

`cocos-studio agent status` reports Agent registry, capability, permission, and
workflow state. It should be read-only unless a separate approved command changes
Agent governance metadata.

## 2. Command Routing Model

Future routing pipeline:

```text
Human Request
      |
      v
CLI Parser
      |
      v
Task Router
      |
      v
Agent Registry Lookup
      |
      v
Permission Check
      |
      v
Execution Proposal
```

Human Request is the user's intent, including task, constraints, project path,
and desired output.

CLI Parser converts the request into a structured command, arguments, and
initial scope.

Task Router selects the workflow path, such as planning, design, build,
validation, release, or Agent governance.

Agent Registry Lookup checks whether any future Agent role or capability is
relevant. It must not activate Agents by default.

Permission Check compares the requested action against approved scope, protected
files, project path approval, memory access rules, and Cocos/MCP boundaries.

Execution Proposal summarizes what would happen next. It must stop for human
approval before writes, high-impact reads, runtime proof, commits, pushes, tags,
or releases.

## 3. Multi-Agent Orchestration Model

Future Agent roles:

### Planner Agent

- responsibility boundary: scope the task, identify risks, define stop
  conditions, and prepare a plan
- allowed input: user request, lightweight core context, explicitly approved
  repository status
- expected output: scoped plan, forbidden scope, validation needs, approval
  request
- handoff rules: hand off to Designer only after goal and scope are clear

### Designer Agent

- responsibility boundary: define design intent, user-facing behavior,
  acceptance criteria, and constraints
- allowed input: approved Planner output and directly relevant design context
- expected output: design constraints, acceptance criteria, unresolved questions
- handoff rules: hand off to Coder only after design scope is stable and
  implementation approval is requested

### Coder Agent

- responsibility boundary: prepare or perform approved implementation inside
  explicit file scope
- allowed input: approved Planner and Designer outputs, approved files, and
  approved validation commands
- expected output: changed files, implementation notes, validation result,
  blockers
- handoff rules: hand off to QA only after validation and diff scope are ready

### QA Agent

- responsibility boundary: review proof, validation output, diff scope, safety
  boundaries, and release readiness
- allowed input: approved task outputs, validation logs, git status, and diffs
- expected output: PASS, NEEDS_REPAIR, or BLOCKED with concise evidence
- handoff rules: QA does not approve new scope; repair or continuation requires
  human approval

## 4. Agent Communication Format

Future message structure:

```text
task_id:
sender:
receiver:
intent:
context:
requested_action:
validation_result:
approval_status:
stop_conditions:
audit_note:
```

Messages must be task-local, explicit, and compact. They must not grant
permissions beyond the approved scope.

## 5. Human Approval Checkpoints

Approval is required before:

- writing files
- reading real Cocos project files
- accessing project memory
- using Cocos Creator, Cocos MCP, browser preview, or runtime console state
- modifying protected files
- deleting files
- staging, committing, pushing, tagging, or releasing
- expanding scope
- enabling an Agent or capability for execution

Dangerous operations include broad deletes, force pushes, unscoped project
inspection, unapproved Cocos/MCP use, package/build/CI changes, generated asset
changes, runtime behavior changes, and autonomous Agent continuation.

Protected files include:

- `SKILL.md`
- README files
- `package.json`
- `CHANGELOG.md`
- `protocols/*`
- `agents/*`
- runtime files
- real Cocos project files

Autonomous execution is forbidden. The CLI may propose; humans approve.

## 6. Context Loading Strategy

Core Context:

- lightweight default rules, operation modes, scope boundaries, approval rules,
  and default routing

Task Context:

- files and docs directly required by the current command and approved scope

Optional Context:

- extended indexes, proof docs, examples, Agent governance docs, archive docs,
  semantic deep docs, and runtime template references

Memory Context:

- project memory or task history loaded only when explicitly triggered and
  approved

Loading priority:

1. Core Context
2. Task Context
3. Optional Context only when directly triggered
4. Memory Context only when explicitly approved

Isolation rules:

- optional context must not become default context
- memory context must not be read or written without approval
- proof/example context must stay validation sandbox material
- Cocos/MCP state must stay outside default context

## 7. Failure Recovery Model

Failed task handling:

- stop the workflow
- record expected result, actual result, failed step, and blocker
- report unexpected files or forbidden scope

Retry strategy:

- retry only the smallest safe step
- require approval if retry changes scope, files, command type, or project path

Rollback strategy:

- rollback only approved files
- avoid dangerous Git commands
- preserve user changes
- report any rollback limitation honestly

Audit record:

- task id
- command
- approved scope
- participating roles
- validation commands
- result
- blocker or repair
- final status

## 8. Fast Build Mode Preservation

Future CLI orchestration must preserve Fast Build Mode:

- do not load CLI orchestration docs by default
- do not activate Agents by default
- do not load Agent, proof, archive, memory, runtime, or example context unless
  directly triggered
- produce compact execution proposals
- stop at approval checkpoints
- keep validation and safety complete even when output is short

## 9. Explicit Non-Implementation Boundary

This document does NOT create:

- Agents
- CLI runtime
- MCP execution
- Cocos automation
- runtime files

It also does not create example projects, modify protocols, modify `agents/*`,
or authorize autonomous execution.
