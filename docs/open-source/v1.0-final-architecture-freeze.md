# v1.0 Final Architecture Freeze

This document freezes the v1.0 architecture direction for
`cocos-studio-ouguowen`. It is documentation only. It does not modify runtime
behavior, connect Cocos/MCP, create Agents, create examples, or change any real
Cocos project.

## Final Core Boundary

The v1.0 core is the small default operating system for safe Cocos Creator 3.8.8
AI-assisted production. Core includes:

- lightweight context loading
- operation mode routing
- default scope boundary
- pre-write approval
- explicit project path and file-scope control
- runtime proof honesty
- QA and release review
- Git and diff discipline
- validation and CI protection
- game idea to scoped MVP planning
- Cocos Creator 3.8.8 architecture guidance

The core must stay safe, compact, and broadly useful. It must not become a
complete game template, a Cocos manual, a default MCP automation layer, or a
multi-engine framework.

## Extension Boundary

Extensions are non-default and trigger-controlled. Extension material can
include:

- Agent governance and workflow proof
- proof sandbox history
- example packs
- advanced runtime templates
- semantic model deep docs
- extra game-type packs
- evolution and governance extras
- future engine families kept outside the Cocos core

Extensions must not load in Fast Build Mode unless explicitly requested. They
must not modify real projects, read project memory, run Cocos/MCP, or expand
scope without human approval.

## Forbidden Core Additions

Do not add these to v1.0 core:

- autonomous Agent execution
- default Agent loading
- default project memory loading
- default Cocos/MCP execution
- default browser preview
- proof-chain continuation by default
- Godot or other engine-specific rules
- full game implementation claims
- broad runtime template packs
- economy, gacha, shop, inventory, combat, save/load, or live-ops systems as
  default core behavior

Any future addition must prove v1.0 alignment, Fast Build Mode preservation, and
validation coverage before promotion.

## Current Capability Summary

The current Skill can:

- classify game ideas into scoped game-type directions
- guide MVP planning and production readiness
- define architecture boundaries for Cocos Creator 3.8.8 projects
- require pre-write approval before project modification
- separate Skill maintenance from real game implementation
- require honest runtime/browser proof when implementation is approved
- review QA, release, Git, and CI state
- preserve proof and example content as validation sandbox evidence
- define Agent identity, capability, permission, and workflow governance as
  documentation-only policy

The current Skill does not:

- implement Agents
- execute autonomously
- replace developers
- claim a finished game development platform
- claim v1.0 is complete merely because this freeze document exists

## v1.1 Roadmap

Recommended v1.1 work:

- add validation coverage for new Agent governance and freeze docs
- keep Agent governance trigger-only and documentation-first
- evaluate project memory templates only after permission boundaries are stable
- reduce default context further without weakening safety
- clarify extension promotion rules with concrete audit criteria
- keep cleanup patches small and paired with validation/index review
- monitor CI and user-perceived Fast Build Mode speed before expanding scope

## Agent Governance Maturity

Agent governance is at policy-foundation maturity.

Established:

- Agent registry policy
- Agent capability policy
- permission policy
- workflow proof documentation
- workflow validation documentation
- human approval gates
- trigger-only activation
- no autonomous execution boundary

Not established:

- real Agent implementation
- Agent runtime
- Agent registry data files
- Agent execution engine
- autonomous workflow scheduler
- default Agent loading

Agent work should remain documentation-only until a separate approved roadmap,
validation plan, and implementation boundary exist.

## Context Optimization Strategy

Keep the default path small:

- load lightweight context first
- route through the default module index only when needed
- keep extended indexes trigger-only
- keep Agent, proof, archive, example, memory, and runtime-template material out
  of default Fast Build Mode
- preserve compact reporting unless safety or audit needs detail
- treat `next recommended command` as optional advice only

Context optimization must never remove:

- safety
- pre-write approval
- explicit project path approval
- human approval gates
- runtime proof honesty
- rollback discipline
- validation and CI protection
