# AI Game Studio v4.1 Architecture

This document defines the overall AI Game Studio architecture.

This is architecture documentation only. It does not implement code, modify MCP, modify Cocos Creator projects, create Agent runtime behavior, modify `agents/*`, or change runtime execution logic.

## 1. AI Game Studio Positioning

AI Game Studio is an AI-assisted game production operating framework for turning game ideas into scoped, validated, and release-ready Cocos Creator 3.8.8 projects.

It is not a complete game, not a single game template, not a Cocos manual, and not an autonomous game company. Its role is to coordinate design intelligence, production planning, Agent governance, tool integration, validation, and release discipline while preserving human control.

Core positioning:

- AI Game Studio is the orchestration and intelligence layer.
- Codex is the coding and reasoning executor.
- Cocos MCP or other automation providers are execution channels only when explicitly approved.
- Cocos Creator 3.8.8 remains the primary engine target for this Skill.
- Human approval remains required before real project modification, MCP execution, release action, or scope expansion.

## 2. Evolution From Cocos Studio Skill To AI Game Studio

The project started as a Cocos Creator production-control Skill focused on safe Cocos development workflows.

It evolved through these stages:

1. Cocos Studio Skill: production rules, pre-write approval, runtime proof honesty, Git discipline, and Cocos Creator 3.8.8 boundaries.
2. v1.x Architecture Foundation: default scope boundary, validation coverage, Agent governance policy, extension boundary, and Fast Build Mode protection.
3. v2.x Engine and MCP Architecture: Cocos capability mapping, MCP adapter design, task planning, execution controller, knowledge engine, asset pipeline, memory engine, and game validation framework.
4. v3.x Production Intelligence: code generation orchestration, production workflow, code review, feedback learning, multi-Agent collaboration, Agent registry, capability marketplace, template system, design intelligence, production pipeline, and runtime validation.
5. v4.x Release Architecture: release automation and the overall AI Game Studio architecture.

The direction changed from "a useful Cocos Skill" into "a controlled AI Game Studio operating architecture" without removing the original safety boundary.

## 5. MCP Position In The System

MCP is an execution bridge, not the intelligence layer.

MCP position:

```text
AI Game Studio
  -> Agent / Task / Capability Decision
  -> Permission Check
  -> MCP Adapter
  -> External MCP Server
  -> Cocos Creator 3.8.8
```

AI Game Studio should not duplicate MCP server implementation unless a future approved roadmap explicitly creates a separate adapter or client layer.

MCP responsibilities:

- Expose available tools.
- Execute approved tool calls.
- Return structured results.
- Surface errors and logs.

AI Game Studio responsibilities:

- Decide whether a tool should be used.
- Check permission and approval state.
- Translate game intent into tool requests.
- Validate results.
- Record approved outcomes.

## 6. Codex Position In The System

Codex is the reasoning and code execution partner used by AI Game Studio.

Codex position:

```text
Human Request
  -> AI Game Studio Planning
  -> Approved Code Task
  -> Codex Execution
  -> Review / Validation
  -> Human Acceptance
```

Codex may:

- Draft architecture documentation.
- Generate or modify approved files.
- Run validation commands.
- Summarize diffs and risks.
- Prepare commits when explicitly requested.

Codex must not:

- Modify real project files without approved scope.
- Treat recommendations as approval.
- Use Cocos/MCP by default.
- Skip validation or human gates.
- Claim runtime behavior without proof.

## 7. Skill And MCP Relationship

The Skill is the operating policy and routing layer. MCP is an optional execution channel.

Relationship:

- The Skill defines what should happen, when approval is required, and how results must be validated.
- MCP performs tool-level operations only when explicitly authorized.
- The Skill must remain useful even when no MCP provider is available.
- MCP availability must not weaken pre-write approval, project path approval, rollback discipline, or proof honesty.

This separation prevents provider lock-in and preserves a stable open-source architecture.

## 10. Open Source Project Notes

AI Game Studio should remain understandable, auditable, and safe for open-source contributors.

Open-source principles:

- Documentation must distinguish architecture from implementation.
- Safety gates must not be optimized away.
- Default usage must stay lightweight.
- Cocos/MCP access must remain explicit and non-default.
- Project-specific memory must require approval.
- Agent behavior must remain governed by policy before implementation.
- Release claims must match validated behavior.

Contributor guidance:

- Add architecture before runtime.
- Add validation before broad automation.
- Keep core small and move advanced material into trigger-only extension docs.
- Do not add new engine families into the Cocos Skill core.
- Do not claim autonomous production capability without reviewed implementation and proof.
