# v5.4 Capability Manifest Runtime

Status: documentation only

Scope: AI Game Studio capability manifest and runtime routing architecture

This document defines the v5.4 Capability Manifest Runtime design for AI Game
Studio. In this phase, "Runtime" means the architecture contract for future
manifest search, ranking, context loading, and Agent handoff. It does not mean
that runtime code is implemented here.

This phase creates no `capabilities.json`, schema file, parser, search engine,
ranking engine, context loader, cache, Agent implementation, or execution code.
It does not modify MCP, Agent runtime, Cocos projects, examples, or scripts.

## 1. Positioning

v5.3 defines the Capability Registry and Capability Index concepts. v5.4 defines
how those concepts can be represented by a compact manifest and consumed by a
future runtime routing process.

The Capability Manifest Runtime sits between request interpretation and Agent
execution:

```text
User Request
  -> Capability Manifest Runtime
  -> Selected Capability Set
  -> Selected Context Set
  -> Existing Agent Execution Boundary
```

Its responsibility is limited to:

- reading compact capability metadata
- finding candidate capabilities
- ranking candidates
- resolving declared dependencies
- selecting context files
- preparing an execution handoff

It does not own:

- user authorization
- Agent business logic
- MCP behavior
- Cocos Creator operations
- project file writes
- final validation policy

### Primary goal

The primary goal is to reduce the need for AI systems to read all documentation
for every request.

The desired loading path is:

```text
Small manifest
  -> Small candidate set
  -> Small context file set
  -> Task execution
```

The loading cost should scale with the selected capability and dependency graph,
not with the total number of repository documents.

## 2. Capability Manifest File Specification

The recommended future manifest filename is:

```text
capabilities.json
```

This document defines the format concept only. It does not create the file.

### File purpose

`capabilities.json` is a compact, machine-readable routing catalog. It contains
enough metadata to discover capabilities and select canonical context files
without embedding the full content of those files.

The manifest is an index, not a policy database. Canonical Markdown modules
remain the source of truth for behavior, safety rules, approval requirements,
validation, and production process.

### Top-level structure

The minimal top-level form is an object containing a capability array:

```json
{
  "capabilities": [
    {
      "id": "quality.review.code",
      "name": "Code Review",
      "keywords": ["code review", "review changes", "regression"],
      "category": "quality",
      "dependencies": ["context.read.diff-scope"],
      "context_files": [
        "production/review-system.md",
        "protocols/quality-gates.md"
      ],
      "priority": 80
    }
  ]
}
```

The example is descriptive only. It is not an executable registry and does not
authorize any file read, write, tool call, or Agent action by itself.

### Required fields

Every capability entry contains exactly these required routing fields:

| Field | Type | Purpose |
|---|---|---|
| `id` | string | Stable unique capability identity |
| `name` | string | Human-readable capability name |
| `keywords` | string array | Intent and terminology search signals |
| `category` | string | Primary capability domain |
| `dependencies` | string array | Required capability IDs |
| `context_files` | string array | Canonical context paths to load |
| `priority` | integer | Final ranking tie-breaker |

All fields are required even when an array is empty. A predictable record shape
keeps runtime search cheap and avoids special cases.

## 3. Field Semantics

### `id`

`id` is the stable identity used by dependencies, logs, ranking results, and
execution handoff.

Rules:

- must be unique within the manifest
- must use lowercase ASCII
- should use dot-separated `domain.action.subject` form
- must not contain an Agent name or provider-specific MCP identity
- must not change for wording-only updates
- must not imply permission or approval state

Examples:

```text
memory.read.project-context
design.classify.game-type
architecture.plan.runtime-kernel
quality.review.code
validation.verify.skill-docs
```

### `name`

`name` is a short label for human review, diagnostics, and candidate display.

Rules:

- must be concise
- must describe one bounded capability
- must not be used as the stable dependency key
- may change without changing `id` when meaning is preserved
- must not promise execution authority

### `keywords`

`keywords` contains normalized terms that help match a user request.

Keywords may represent:

- user-facing phrases
- command vocabulary
- artifact names
- domain terminology
- operation verbs
- common aliases

Rules:

- entries must be non-empty strings
- matching should be case-insensitive where language rules permit
- duplicate normalized keywords within one record are invalid
- keywords are hints, not the only ranking signal
- broad keywords should not defeat exact artifact or capability matches
- keywords must not silently expand task scope

The manifest may contain multilingual keywords when required by the user base,
but routing must preserve the original request and must not translate a safe
read-only intent into a write intent.

### `category`

`category` groups capabilities for filtering and budget selection.

Initial categories may include:

- `memory`
- `planning`
- `design`
- `architecture`
- `agent`
- `quality`
- `validation`
- `release`
- `package`
- `cocos`
- `mcp`
- `docs`

Rules:

- one capability has one primary category
- category is a filter, not an ownership or permission grant
- cross-category prerequisites belong in `dependencies`
- unknown categories must not be executed without schema review
- category additions require a reviewed manifest version change in a future
  implementation

### `dependencies`

`dependencies` contains capability IDs that must be resolved before the selected
capability is ready for handoff.

Rules:

- each value must refer to an existing `id`
- self-dependency is invalid
- circular dependencies are invalid
- duplicates are invalid
- dependencies must be traversed with a bounded depth
- dependency order must be deterministic
- dependency discovery does not grant permission to execute that dependency

An unresolved dependency produces a blocked candidate. It must not trigger full
repository loading as a fallback.

### `context_files`

`context_files` lists canonical repository-relative files required to understand
the capability.

Rules:

- paths must be normalized repository-relative paths
- paths must not escape the repository root
- each path must point to an allowed canonical context file
- duplicate paths are invalid
- files must be loaded in declared order unless dependency order requires an
  earlier safety module
- a manifest entry must not list the entire docs tree
- a context path is read authorization only when the surrounding task allows it
- a context path never grants write authorization

The runtime must distinguish between:

- capability-owned context files
- dependency context files
- mandatory safety context
- task-provided context

The final context set is deduplicated while preserving deterministic order.

### `priority`

`priority` is an integer used only after stronger ranking signals have been
evaluated.

Recommended conceptual range:

```text
0 through 100
```

Rules:

- higher values win only when candidates are otherwise equally relevant
- priority must not override exact capability ID matches
- priority must not override target-path or artifact compatibility
- priority must not override safety restrictions
- priority must not override missing dependencies
- priority must not authorize Agent execution

Priority is a deterministic tie-breaker, not a confidence score and not a
permission level.

## 4. Manifest Integrity Rules

A future Capability Manifest Runtime must reject or quarantine an invalid
manifest before task routing.

### Structural integrity

The manifest is structurally valid only when:

- the top-level object exists
- `capabilities` is an array
- every entry contains all required fields
- every field has the expected type
- all capability IDs are unique
- all referenced dependencies exist
- all context paths are normalized and inside the repository
- all priorities are valid integers in the accepted range

### Graph integrity

The dependency graph is valid only when:

- no capability depends on itself
- no cycle exists
- every dependency can be resolved
- traversal has a deterministic order
- maximum traversal depth and selected capability count remain bounded

### Context integrity

The context set is valid only when:

- all referenced files exist
- all files belong to allowed context classes
- no path escapes the repository
- no forbidden project path is referenced
- the selected set stays within the active context budget
- canonical policy content is not duplicated into the manifest

### Failure behavior

Invalid manifest state must fail closed for the affected capability.

Allowed outcomes:

- `MANIFEST_READY`
- `MANIFEST_INVALID`
- `CAPABILITY_NOT_FOUND`
- `CAPABILITY_AMBIGUOUS`
- `DEPENDENCY_MISSING`
- `DEPENDENCY_CYCLE`
- `CONTEXT_PATH_INVALID`
- `CONTEXT_BUDGET_EXCEEDED`
- `EXECUTION_APPROVAL_REQUIRED`

Failure must not cause automatic full-doc loading, automatic dependency
installation, or permission escalation.

## 5. Runtime Architecture

The future Capability Manifest Runtime contains six conceptual components.

```text
Manifest Loader
  -> Manifest Validator
  -> Request Matcher
  -> Capability Ranker
  -> Dependency Resolver
  -> Context Set Builder
  -> Agent Handoff
```

### Manifest Loader

Manifest Loader reads the compact manifest from an approved repository location.
It does not read all referenced context files during startup.

Responsibilities:

- locate the approved manifest
- parse structured data
- expose immutable capability records for one routing cycle
- report parse or availability failures

### Manifest Validator

Manifest Validator checks structure, IDs, field types, dependencies, context
paths, and graph integrity before search begins.

It validates metadata consistency only. It does not validate the full semantic
correctness of every referenced architecture document.

### Request Matcher

Request Matcher converts the user request into a routing profile:

- normalized terms
- explicit capability ID when present
- requested artifact
- requested operation
- target path or repository
- domain category
- explicit restrictions
- approval state

The original user request remains authoritative and must accompany the routing
profile.

### Capability Ranker

Capability Ranker scores valid candidates using ordered routing signals. It must
produce explainable ranking evidence rather than a hidden selection.

Recommended ranking order:

```text
Exact capability ID
  -> Scope and target compatibility
  -> Safety compatibility
  -> Exact phrase or artifact match
  -> Category match
  -> Keyword coverage
  -> Specificity
  -> Dependency readiness
  -> Priority
```

The Ranker returns a small candidate set. It does not return every keyword match.

### Dependency Resolver

Dependency Resolver builds a directed acyclic dependency plan for the selected
capability set.

Responsibilities:

- resolve IDs
- reject missing dependencies
- detect cycles
- preserve deterministic order
- stop dependency expansion at configured bounds
- identify dependencies that require separate approval

It must not execute a dependency or broaden the task automatically.

### Context Set Builder

Context Set Builder combines:

- mandatory lightweight core context
- selected capability context files
- dependency context files
- mandatory safety context for the active risk class
- user-provided task context

It then:

- normalizes paths
- removes duplicates
- preserves loading order
- estimates context cost
- enforces the selected context budget
- produces an allow list and an explicit not-loaded list

Context Set Builder is the main mechanism that reduces broad documentation
loading.

### Agent Handoff

Agent Handoff passes the selected capability and context plan to the existing
Agent execution boundary.

The handoff does not implement or alter the Agent. It is a data contract that
should contain:

- original user request
- selected capability IDs
- candidate ranking evidence
- resolved dependency order
- approved context files
- not-loaded context categories
- active context budget
- approved write scope
- forbidden actions
- validation expectations
- human approval state

## 6. Runtime Flow

The required v5.4 runtime flow is:

```text
User Request
  -> Capability Manifest Search
  -> Capability Ranking
  -> Context Loading
  -> Agent Execution
```

### Step 1: User Request

The runtime receives the original request without rewriting its authority or
scope.

It extracts routing signals but preserves:

- requested output
- target path
- forbidden modifications
- validation commands
- approval boundaries
- stop conditions

### Step 2: Capability Manifest Search

Search reads only manifest metadata.

Search behavior:

1. check for an exact capability ID
2. filter by target scope and category
3. match exact phrases and artifact names
4. match normalized keywords
5. remove blocked or structurally invalid candidates
6. return a bounded candidate set

Search must not load the full context of every candidate. Detailed context is
loaded only after ranking selects the final capability set.

### Step 3: Capability Ranking

Ranking selects the smallest capability set that fully covers the request.

Ranking must prefer:

- exact matches over broad matches
- narrow capabilities over umbrella capabilities
- dependency-ready candidates over incomplete candidates
- candidates compatible with explicit restrictions
- deterministic results for the same manifest and request

Priority is applied last.

Ranking output includes selected, rejected, and ambiguous candidates with short
reason codes. Ambiguity must not be resolved by loading all candidates.

### Step 4: Context Loading

Context Loading resolves dependency files, adds mandatory safety context, and
loads only the approved context set.

Loading sequence:

```text
Lightweight Core Context
  -> Dependency Context
  -> Selected Capability Context
  -> Task-Specific Safety Context
  -> User-Provided Context
```

Loading stops when:

- the selected capability is fully supported
- the context budget is reached
- a dependency would expand scope
- a path is invalid
- required approval is missing
- contradictory canonical rules require a blocker decision

The runtime must not silently fall back to reading all docs.

### Step 5: Agent Execution

Agent Execution begins only after context selection and independent permission
checks are complete.

The Capability Manifest Runtime may recommend an Agent-facing capability, but it
does not select an implementation that bypasses the Agent Registry or existing
handoff rules.

Agent Execution must receive the original restrictions. A selected capability
does not authorize:

- a file write outside approved scope
- Agent runtime modification
- an MCP call
- a Cocos project operation
- dependency installation
- commit, push, tag, or release

## 7. Capability Ranking Model

The ranking model should be explainable and deterministic.

### Conceptual score components

A future implementation may use weighted components, but ordering is more
important than any specific numeric formula.

Conceptual components:

- exact ID match
- exact phrase match
- artifact match
- operation match
- category match
- keyword coverage
- scope compatibility
- dependency readiness
- specificity
- manifest priority

Safety compatibility is a gate, not a positive score. An unsafe candidate is
removed rather than merely penalized.

### Tie handling

When two candidates remain tied:

1. prefer the narrower context set
2. prefer fewer unresolved dependencies
3. prefer the more specific artifact match
4. apply manifest priority
5. use stable `id` ordering for deterministic diagnostics

If the candidates represent materially different actions, the result remains
ambiguous and requires clarification or a read-only common context path.

### Multi-capability requests

A request may legitimately require multiple capabilities.

The runtime may select a capability set when:

- each selected capability owns a distinct required outcome
- dependency order is valid
- combined context remains inside budget
- combined permissions stay inside user scope
- no selected capability is redundant

The runtime must not turn every multi-part sentence into an unbounded capability
bundle.

## 8. Context Loading Strategy

The manifest runtime optimizes context through progressive disclosure.

### Startup context

Load only:

- compact Skill identity and safety summary
- operation mode guidance
- `capabilities.json` metadata in a future implementation
- current request

Do not load every manifest-referenced file at startup.

### Selected context

After ranking, load:

- selected capability `context_files`
- dependency `context_files`
- directly triggered safety protocols
- task-local files required for the requested result

### On-demand context

Additional context is loaded one dependency or decision boundary at a time when:

- the selected modules explicitly reference a required canonical rule
- validation reports a missing prerequisite
- the task scope is clarified or expanded by the user
- a safety boundary requires a specific protocol

### Context budget interaction

The Runtime uses the v5.2 budget classes:

- `MIN_CONTEXT`
- `TASK_CONTEXT`
- `GATE_CONTEXT`
- `AUDIT_CONTEXT`

Capability category and requested operation propose the initial budget. Risk,
approval requirements, and validation failures may escalate it. A higher budget
permits more relevant context; it does not authorize broader execution.

### Context release

After task completion:

- release task-specific loaded documents from active context
- retain only the compact execution result required by current task memory
- write Project Memory only through its existing approval process
- do not mutate the manifest from runtime observations automatically

## 9. Caching and Freshness Contract

A future runtime may cache parsed manifest metadata, but caching must not make
the routing result stale or unsafe.

Conceptual cache key inputs:

- manifest content hash
- schema version when introduced
- repository revision or manifest revision
- runtime compatibility version

Cache rules:

- cache metadata, not full documentation by default
- invalidate when manifest content changes
- revalidate dependency and path integrity after invalidation
- do not cache user approval as a permanent capability permission
- do not reuse project-specific context across unrelated projects
- do not treat cached Agent availability as current authority

Caching is optional in the future implementation and is not created in v5.4.

## 10. Security and Approval Boundaries

The manifest is untrusted routing metadata until validated.

### Path safety

- reject absolute paths unless a future schema explicitly defines a safe class
- reject `..` traversal outside the repository
- reject paths into unapproved Cocos projects
- reject scene, prefab, meta, or runtime paths when the active task is docs-only
- do not follow manifest paths as write targets

### Permission safety

- capability presence is not capability permission
- ranking success is not execution approval
- dependency resolution is not dependency execution
- context loading is not write approval
- Agent handoff is not MCP authorization
- MCP availability is not Cocos project authorization

### Human approval

Existing human approval rules remain authoritative. The handoff must preserve
approval state but must not invent it.

Human approval remains required before any existing high-impact boundary,
including Agent runtime changes, MCP changes, real Cocos project writes,
dependency installation, release actions, or scope expansion.

## 11. Observability Contract

A future runtime needs compact diagnostics so routing decisions can be reviewed
without exposing or loading all documentation.

Each routing cycle should record:

- manifest revision
- request classification
- candidate capability IDs
- selected capability IDs
- rejection reason codes
- dependency resolution result
- selected context paths
- context budget result
- approval state
- handoff or blocker result

Diagnostics must not include secrets, unnecessary full document bodies, or an
unbounded copy of the user context.

The minimum result states are:

- `ROUTING_READY`
- `ROUTING_BLOCKED`
- `ROUTING_AMBIGUOUS`
- `CONTEXT_READY`
- `CONTEXT_BLOCKED`
- `HANDOFF_READY`
- `HANDOFF_APPROVAL_REQUIRED`

## 12. Relationship With Existing Architecture

### v3.5 Capability Marketplace

The Capability Marketplace governs capability lifecycle, version, status,
installation proposals, and human approval. v5.4 consumes only approved
capability metadata in a future implementation.

Marketplace availability does not imply manifest inclusion, and manifest
inclusion does not imply execution approval.

Related document: [v3.5 Capability Marketplace Architecture](v3.5-capability-marketplace-architecture.md)

### v5.2 Context Optimization Engine

v5.2 defines context layers, Task Router categories, context budgets, and memory
layering. v5.4 provides the structured metadata and runtime flow that can drive
those decisions.

```text
Capability Manifest Runtime
  -> v5.2 Task Router
  -> Context Budget Manager
  -> Core / Domain / Execution Context
```

Related document: [v5.2 Context Optimization Engine](v5.2-context-optimization-engine.md)

### v5.3 Capability Index System

v5.3 defines Capability Registry records, discovery, matching, ranking order,
dependency rules, and selective module loading. v5.4 converts that architecture
into a concrete manifest file contract and future runtime component model.

```text
v5.3 Capability Registry and Index
  -> v5.4 capabilities.json contract
  -> Manifest Search and Ranking
  -> Context Set Builder
  -> Agent Handoff
```

Related document: [v5.3 Capability Index System](v5.3-capability-index-system.md)

### Runtime Kernel

The Runtime Kernel may consume the v5.4 handoff through its Capability,
Planner, Memory, and Execution Kernels.

```text
Capability Manifest Runtime
  -> Capability Kernel
  -> Planner Kernel
  -> Memory Kernel
  -> Execution Kernel
  -> Agents / Tools / MCP / Cocos Creator when independently approved
```

v5.4 does not implement or modify any Runtime Kernel component.

## 13. Future Implementation Contract

A future implementation phase must open a separate approved scope before
creating `capabilities.json` or runtime code.

That proposal must define:

- manifest repository location
- JSON Schema and versioning policy
- canonical owner
- parser and validation boundary
- search normalization rules
- ranking evidence format
- dependency graph limits
- context budget measurement
- cache invalidation
- Agent handoff schema
- test strategy
- rollback plan
- migration path from existing module routing

Required future test families include:

- valid manifest allow path
- malformed manifest block path
- duplicate ID block path
- missing dependency block path
- dependency cycle block path
- path traversal block path
- ambiguous ranking path
- context budget overflow path
- approval-required handoff path
- deterministic ranking path

No implementation may be promoted until both allow and block paths pass.

## Phase Boundary

v5.4 accepts:

- `capabilities.json` file format design
- required manifest field definitions
- manifest integrity and dependency rules
- request search and capability ranking design
- selective context loading design
- Agent execution handoff design
- cache, diagnostics, and approval contracts

v5.4 does not authorize:

- creating `capabilities.json`
- creating a JSON Schema
- parser, matcher, ranker, resolver, loader, cache, or runtime code
- MCP modification
- Agent runtime modification
- Cocos project modification
- examples modification
- scripts modification
- automatic capability execution
- automatic dependency installation
- automatic permission escalation

This phase ends with architecture documentation and repository validation only.
