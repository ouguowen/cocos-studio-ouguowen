# AI Game Studio v3.4 Agent Registry Architecture

This document defines the Agent Registry architecture for AI Game Studio. It is
architecture documentation only. It does not implement code, modify MCP, modify
Cocos Creator projects, create Agent runtime behavior, modify `agents/*`, or
change runtime execution logic.

## 1. Agent Registry Goal

AI Game Studio needs an Agent Registry so the system can identify which Agent is
responsible for a task, what capability the Agent provides, whether the Agent is
available, and what approval boundary applies before execution.

The registry solves these problems:

- avoid vague Agent ownership
- prevent tasks from being routed to the wrong Agent
- map task requirements to Agent capabilities
- track Agent status during collaboration
- preserve human approval before project modification
- keep Fast Build Mode lightweight by loading only the Agent definitions needed
  by the current task

The Agent Registry is a routing and governance layer. It is not an autonomous
runtime executor and does not create Agents by itself.

## 2. Agent Identity Model

Each registered Agent must have a stable identity record.

Required fields:

- Agent ID
- Agent Name
- Agent Role
- Agent Capability
- Agent Version
- Agent Status
- Responsibility
- Permission Boundary
- Supported Task Types
- Review Requirement

Example identity record:

```text
Agent ID:
  agent.planner

Agent Name:
  Planner Agent

Agent Role:
  Project planning and task decomposition

Agent Capability:
  task-planning, dependency-analysis, approval-packet-generation

Agent Version:
  v1

Agent Status:
  available

Responsibility:
  Convert user intent into scoped task plans and route work to specialist Agents.
```

Identity rules:

- Agent ID must be stable and unique.
- Agent Name should be readable by humans.
- Agent Role must describe the production responsibility.
- Agent Capability must be specific enough for capability discovery.
- Agent Version must change when responsibilities or capability boundaries
  change.
- Agent Status must be explicit before task assignment.

## 3. Capability Discovery

Capability Discovery maps a task to the best available Agent.

Discovery flow:

```text
Task Requirement
      |
      v
Capability Query
      |
      v
Agent Registry
      |
      v
Candidate Agents
      |
      v
Permission Check
      |
      v
Selected Agent
```

Discovery inputs:

- task type
- game type
- required capability
- required tools
- project path approval state
- memory access approval state
- validation requirement
- risk level

Example: UI task

```text
UI task
  |
  v
Search:
  asset-agent
  ui-capability
  qa-ui-validation
```

Possible result:

- Asset Agent for UI asset requirements
- Design Agent for UI layout intent
- QA Agent for UI consistency review

Example: code task

```text
Code task
  |
  v
Search:
  code-agent
  cocos-typescript-capability
  code-review-agent
```

Possible result:

- Code Agent for approved code proposal
- Code Review Agent for generated code review
- QA Agent for validation result review

Capability matching rules:

- prefer the Agent with the narrowest matching capability
- reject Agents that are offline or outside scope
- require human approval before assigning project-modifying tasks
- require validation expectations before assigning implementation tasks
- do not load unrelated Agents into Fast Build Mode

## 4. Agent Lifecycle

Agent lifecycle states:

- registered
- available
- working
- reviewing
- offline

Lifecycle meaning:

- registered: Agent identity and capabilities are known, but the Agent may not
  be available for task assignment.
- available: Agent can receive a task that matches its capability and approval
  boundary.
- working: Agent is handling an approved planning, review, validation, or
  execution-support task.
- reviewing: Agent output is under QA, code review, or human review.
- offline: Agent cannot receive new tasks.

Lifecycle transitions:

```text
registered
    |
    v
available
    |
    v
working
    |
    v
reviewing
    |
    v
available
```

Failure transition:

```text
working -> offline
working -> reviewing
reviewing -> available
reviewing -> offline
```

Lifecycle rules:

- only available Agents may receive new tasks.
- working Agents must report task id, scope, and status.
- reviewing Agents must wait for review result before reassignment.
- offline Agents must not receive tasks.
- failed or blocked Agents must return a blocker report through Planner Agent.

## 5. Agent Communication

Agent Registry connects to v3.3 Multi-Agent Collaboration Protocol by providing
the identity and capability source used during message routing.

Registry-backed communication:

```text
Sender Agent
     |
     v
Agent Registry Lookup
     |
     v
Receiver Agent
     |
     v
Message Protocol
```

The registry answers:

- does the target Agent exist?
- is the target Agent available?
- does the Agent have the requested capability?
- does the Agent require human approval before this task?
- does the task require Cocos/MCP, memory access, or project path approval?

Communication fields supplied by the registry:

- sender Agent ID
- receiver Agent ID
- receiver capability
- receiver status
- permission boundary
- required approval gate
- validation requirement

Registry communication rules:

- a message cannot route to an unregistered Agent.
- a message cannot override an Agent permission boundary.
- a message cannot convert optional advice into approval.
- a message that requires unavailable capability must return a blocker.

## 6. Human Control

The Agent Registry preserves human approval by making high-impact routing
explicit.

Human approval is required before:

- assigning an Agent to modify project files
- assigning an Agent to call Cocos MCP for project-changing work
- assigning an Agent to write project memory
- applying Agent review suggestions
- retrying a failed task with broader scope
- committing, pushing, tagging, or releasing

Human control packet:

- task id
- selected Agent
- selected capability
- project path when relevant
- allowed files
- forbidden files
- expected output
- validation plan
- rollback plan
- stop conditions

Registry safety rule:

- Agent availability is not approval.
- Capability match is not approval.
- Agent handoff is not approval.
- Human confirmation remains required for project changes.

## 7. Integration With Existing Architecture

v2.6 Task Planning:

- provides task id, task type, dependency, required capability, MCP mapping, and
  validation requirement.
- Agent Registry uses these fields to find candidate Agents.

v2.10 Project Memory:

- stores approved Agent history, project decisions, architecture state, and
  validation history when explicitly triggered.
- Agent Registry may reference memory only after memory access is approved.

v3.0 Code Generation:

- uses Agent Registry to route approved code tasks to Code Agent or Codex
  execution support.
- uses registry metadata to determine required review and validation.

v3.3 Multi-Agent Protocol:

- defines message format, task handoff, state synchronization, collaboration
  rules, and human approval.
- Agent Registry provides the identity and capability lookup used by that
  protocol.

Integrated routing loop:

```text
Task Planning
      |
      v
Agent Registry
      |
      v
Capability Discovery
      |
      v
Multi-Agent Protocol
      |
      v
Human Approval
      |
      v
Approved Agent Work
```

The registry makes Agent collaboration discoverable, auditable, and controlled
without creating autonomous execution.
