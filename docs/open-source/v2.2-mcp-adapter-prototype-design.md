# AI Game Studio v2.2 MCP Adapter Prototype Design

This document defines the first prototype architecture of the AI Game Studio MCP
Adapter. It is documentation only. It does not create runtime code, implement an
MCP client, implement Cocos automation, modify core files, or modify real Cocos
project files.

## 1. Prototype Goal

The MCP Adapter is the bridge between AI Game Studio Agents and external MCP
Servers.

It does not replace MCP Servers.

It provides:

- connection management
- tool discovery
- capability mapping
- permission control
- request translation
- response normalization

Architecture:

```text
User
  |
  v
AI Game Studio
  |
  v
Agent Runtime
  |
  v
Capability Registry
  |
  v
MCP Adapter Prototype
  |
  v
External MCP Server
  |
  v
Game Engine
```

The prototype remains a design artifact until a separate implementation plan,
permission model, and validation path are approved.

## 2. MCP Adapter Core Components

### MCP Connection Manager

Responsibilities:

- establish MCP connection
- maintain session state
- detect availability
- reconnect handling

The connection manager must not connect to a real MCP server without explicit
approval in a future implementation phase.

### Tool Discovery Module

Responsibilities:

- discover available MCP tools
- read tool metadata
- register capabilities

Tool discovery should be read-only and must not imply permission to execute
tools.

### Tool Mapping Layer

Responsibilities:

Map:

```text
Agent Intent:
"Create enemy character"
```

into:

```text
MCP Tool:
create_node()
add_component()
set_property()
```

Mapping must stay provider-neutral until a concrete MCP server is approved.

### Permission Gate

Responsibilities:

Classify actions:

Safe:

- query information
- create temporary objects

Approval required:

- modify project assets
- delete resources
- build project

Forbidden:

- uncontrolled destructive operations

The permission gate must run before tool execution and before retry after
failure.

## 3. MCP Request Lifecycle

Future flow:

```text
Agent Request
      |
      v
Task Router
      |
      v
Capability Registry Lookup
      |
      v
Permission Check
      |
      v
MCP Adapter Translation
      |
      v
MCP Tool Call
      |
      v
Result Validation
      |
      v
Agent Feedback
```

The lifecycle stops at the permission check unless the requested operation has
the required human approval.

## 4. Request / Response Format

Abstract request:

```text
task_id:
agent_id:
capability:
target:
parameters:
permission_level:
```

Abstract response:

```text
task_id:
status:
result:
error:
audit_record:
```

Request and response records must be auditable and task-local.

## 5. Cocos Creator 3.8.8 First Mapping Example

Agent:

```text
Create a player character
```

Adapter:

Character requirement:

- Node
- Sprite
- Animation
- Collider

Mapping:

```text
Cocos MCP:
create_node
add_component
set_property
```

This example is a mapping design only. It does not execute Cocos MCP, create a
scene, or modify a project.

## 6. Human Approval Workflow

Checkpoints before:

- asset deletion
- project setting changes
- build release
- real Cocos project modification
- runtime proof collection
- Cocos/MCP execution

Approval flow:

```text
Agent proposal
      |
      v
Human review
      |
      v
Execution
```

Execution remains future architecture. Human approval remains required.

## 7. Error Recovery

Failure cases:

- MCP unavailable
- tool failure
- invalid parameters
- permission denied

Recovery:

- retry
- fallback
- rollback
- audit logging

Retry must not happen automatically for destructive or write operations.
Rollback must preserve user changes and avoid dangerous Git commands.

## 8. Multi Engine Compatibility

The same Adapter architecture can support:

- Cocos MCP
- Unity MCP
- Godot MCP
- Blender MCP

Only engine-specific mappings change.

Each engine adapter must keep its engine rules, files, validation, proof paths,
and runtime assumptions separated.

## 9. Context Passing Strategy

Core Context:

- AI Game Studio rules

Project Context:

- current game information

Engine Context:

- Cocos Creator 3.8.8 capabilities

Task Context:

- current operation

Context passed to MCP must be minimal, approved, and task-local. Do not pass full
Skill docs, proof history, archive docs, or project memory by default.

## 10. Explicit Limitations

This document creates:

- no MCP runtime implementation
- no MCP client code
- no autonomous execution
- no Cocos project modification

Human approval remains required.

## 11. Roadmap

v2.3: MCP connection proof of concept

- validate read-only connection design
- inspect tool metadata
- prove permission gates before tool calls

v2.4: Cocos MCP controlled operation

- approve one safe Cocos operation
- collect normalized result output
- verify rollback and audit logging

v2.5: First Agent driven game creation workflow

- connect Planner output to adapter proposal
- keep Coder and QA behind approval gates
- prove no autonomous execution
