# v1.9 MCP Execution Layer Architecture

This document prepares the future MCP Execution Layer architecture for AI Game
Studio. It is documentation only. It does not implement MCP, create runtime
execution, automate Cocos, create `agents/*` files, modify protocols, create
examples, or authorize autonomous execution.

## 1. MCP Layer Positioning

The MCP layer is a future controlled execution bridge between approved Agent
Runtime requests and external tools.

Relationship:

- Agent Runtime prepares approved task state, capability, context, and
  permission requirements.
- MCP provides tool access after permission checks and human approval.
- Cocos Creator may be accessed through a Cocos MCP provider only after explicit
  project path, scope, and proof target approval.
- External tools may include file, build, validation, browser, or project
  automation providers, each with separate permissions.

The MCP layer must never become default execution. It must sit behind approval,
audit logging, and rollback planning.

## 2. Tool Execution Model

Future execution flow:

```text
Agent request
      |
      v
Permission check
      |
      v
MCP tool selection
      |
      v
Execution proposal
      |
      v
Human approval
      |
      v
Tool execution
```

The execution proposal must include tool name, intent, inputs, allowed paths,
forbidden paths, expected outputs, rollback plan, and stop conditions.

## 3. MCP Capability Categories

Future capability categories:

- file operations: read, write, diff, stage, and status checks
- Cocos scene operations: open scene, inspect hierarchy, create nodes, bind
  components, save scene
- asset operations: inspect, import, bind, validate, or report asset state
- script operations: inspect, edit, validate, or compile approved scripts
- build operations: run approved build or package commands
- test operations: run validation, browser preview, runtime proof, or CI checks

Categories do not grant permission. They only describe possible tool groups.

## 4. Tool Permission System

Permissions:

- read permission: inspect approved files, state, logs, or tool output
- write permission: modify approved files or tool-managed project state
- execute permission: run approved commands or tool actions
- destructive operation approval: required before delete, overwrite, broad
  cleanup, force, reset, rebuild, or irreversible operations

Permission must be checked before tool selection and again before execution.

## 5. Cocos MCP Integration Boundary

Cocos MCP can access only what the user approves:

- approved active Cocos project path
- approved scene, prefab, script, asset, or settings scope
- approved read or write operations
- approved browser/runtime proof target

Cocos MCP cannot access by default:

- real Cocos project files
- `.scene`, `.prefab`, `.anim`, `.meta` files
- external assets
- package, build, settings, or project files
- browser preview or runtime console
- unapproved project paths
- unrelated scenes, prefabs, scripts, or assets

Provider availability must not bypass approval gates.

## 6. Execution Result Format

Tool execution should return:

```text
execution_result:
success:
failure:
logs:
generated_files:
modified_files:
warnings:
validation_result:
rollback_needed:
next_required_approval:
```

Results must distinguish success, failure, warnings, generated files, modified
files, and unresolved blockers.

## 7. Error Recovery

MCP failure handling:

- stop execution
- report tool name and failed operation
- include logs or error output
- list modified or generated files
- identify whether rollback is required

Timeout handling:

- stop or mark the task waiting
- do not retry destructive operations automatically
- request approval before retry if scope or risk changes

Rollback:

- rollback only approved files or tool state
- avoid dangerous Git commands
- preserve user changes
- report rollback limitations honestly

Retry:

- retry only the failed smallest step
- require approval for tool changes, path changes, or write retries

## 8. Security Model

Security requirements:

- sandbox tool access by approved path and capability
- keep audit logs for tool selection, approvals, execution, and results
- require human approval before high-impact or destructive operations
- prevent default access to real projects
- prevent autonomous tool execution
- protect package, CI, release, protocol, runtime, and Agent files
- report unexpected files immediately

The MCP layer is an execution channel, not a source of authority.

## 9. Future Roadmap

Roadmap:

- v2.0 Cocos automation layer: define safe Cocos Creator operation flows, scene
  modification boundaries, proof channels, and generated metadata rules.
- v2.x production execution: add production-grade execution only after CLI,
  Agent Runtime, Permission, Context, Memory, QA, and MCP layers have validation
  and audit coverage.

## 10. Explicit Limitations

This document creates:

- no MCP implementation
- no runtime execution
- no Cocos automation
- no autonomous execution

It also creates no runtime files, no `agents/*` files, no protocol changes, no
examples, and no real Cocos project changes.

Human approval remains required before execution.
