# AI Game Studio v2.5 Cocos MCP Capability Mapping

This document defines the future mapping between AI Game Studio capability
intent and Cocos MCP tool capability groups. It is documentation only. It does
not implement runtime behavior, modify MCP, execute tools, or allow autonomous
execution.

## MCP Capability Registry Mapping

The mapping layer should translate user intent into Agent interpretation, then
into MCP capability groups, and finally into expected results.

Capability groups:

- scene
- node
- component
- prefab
- asset
- animation
- ui builder
- validate

## Scene Capability

- user intent: create, open, save, or inspect a scene
- Agent interpretation: identify scene path, purpose, allowed changes, and
  validation target
- MCP tool mapping: `scene`
- expected result: approved scene operation completed or scene state reported

## Node Capability

- user intent: create, delete, rename, parent, or transform a node
- Agent interpretation: identify node type, parent, transform, naming, and
  forbidden hierarchy changes
- MCP tool mapping: `node`
- expected result: approved node operation completed and hierarchy validated

## Component Capability

- user intent: add, remove, or configure a component
- Agent interpretation: identify component type, target node, properties, and
  binding requirements
- MCP tool mapping: `component`
- expected result: component state updated and inspector/runtime proof collected
  when needed

## Prefab Capability

- user intent: create, modify, instantiate, or inspect a prefab
- Agent interpretation: identify prefab path, source node, dependencies, and
  approval boundary
- MCP tool mapping: `prefab`
- expected result: prefab operation completed without broken references

## Asset Capability

- user intent: query, import, organize, or validate assets
- Agent interpretation: identify source, owner, path, type, usage, and metadata
  risk
- MCP tool mapping: `asset`
- expected result: asset state reported or approved asset operation completed

## Animation Capability

- user intent: create, bind, inspect, or validate animation
- Agent interpretation: identify clip, state, trigger, exit rule, and proof
  requirement
- MCP tool mapping: `animation`
- expected result: animation operation completed while gameplay truth remains
  outside animation callbacks

## UI Builder Capability

- user intent: create Canvas, layout, button, label, panel, or responsive UI
- Agent interpretation: identify UI hierarchy, component needs, text, layout,
  scaling, and runtime visibility target
- MCP tool mapping: `ui builder`
- expected result: approved UI structure created or validated

## Validate Capability

- user intent: verify scene, runtime, project, asset, script, or UI state
- Agent interpretation: identify validation command, proof target, expected
  output, and blocker conditions
- MCP tool mapping: `validate`
- expected result: PASS, NEEDS_REPAIR, BLOCKED, or FAIL with evidence

## Example Mapping

User:

```text
Create player character
```

Agent:

Need:

- `node.create`
- `component.add`
- `prefab.create`

MCP:

- node
- component
- prefab

Result:

```text
Player object created.
```

The example is a mapping illustration only. It does not execute Cocos MCP,
create nodes, attach components, or create prefabs.

## Planner Agent Usage

Planner Agent should:

- classify user intent
- identify required capability groups
- define allowed and forbidden scope
- prepare approval checkpoints
- request only necessary context
- stop before execution

Planner output should select capability mappings but must not execute them.

## Capability Registry Usage

The Capability Registry should:

- store capability group names
- define required permissions
- define expected result format
- define validation requirements
- map Agent intent to MCP capability families

Capability Registry entries must not grant permission by themselves.

## QA Validation Flow

QA should verify:

- selected capability matches user intent
- approved scope was preserved
- MCP mapping was correct
- expected result was produced
- generated or modified files were reported
- no forbidden files or operations appeared
- proof is honest and sufficient

QA result should be PASS, NEEDS_REPAIR, BLOCKED, or FAIL.

## Human Approval Checkpoints

Human approval is required before:

- executing MCP tools
- modifying scenes, prefabs, scripts, assets, metadata, settings, or builds
- deleting resources
- creating release builds
- using browser preview or runtime console
- staging, committing, pushing, tagging, or releasing

`next recommended command` is optional advice only and never approval.

## Safety Boundary

This document creates:

- no runtime implementation
- no MCP modification
- no autonomous execution
- no Cocos project change
- no Agent implementation

It is documentation only.
