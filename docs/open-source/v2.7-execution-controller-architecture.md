# AI Game Studio v2.7 Execution Controller Architecture

This document defines the execution controller layer between Task Planning and
the MCP Adapter. It is documentation only. It does not create runtime
implementation, modify MCP, execute Cocos operations, or allow autonomous
execution.

## 1. Execution Controller Responsibility

The Execution Controller is the future layer that turns an approved Task Plan
into a controlled execution sequence.

Responsibilities:

- receive Task Plan output from Planner Agent
- verify approval status before execution
- select required capability through Capability Registry
- prepare MCP Adapter request
- track task lifecycle state
- stop on missing approval, failure, or unexpected files
- report execution result to QA Agent

The Execution Controller must not execute by default. It may coordinate only
approved tasks.

## 2. Task Lifecycle

Task lifecycle:

```text
created
   |
   v
planned
   |
   v
approved
   |
   v
executing
   |
   v
completed
   |
   v
verified
```

Lifecycle definitions:

- created: user intent is received and assigned a task id
- planned: Planner Agent has produced a Task Plan
- approved: human approval is recorded for the scoped task
- executing: the future controller is coordinating approved execution
- completed: requested operation has returned a result
- verified: QA Agent or validation process has confirmed the result

Any lifecycle transition that broadens scope requires human approval.

## 3. MCP Execution Pipeline

Future pipeline:

```text
Planner Agent
      |
      v
Task Plan
      |
      v
Execution Controller
      |
      v
Capability Registry
      |
      v
MCP Adapter
      |
      v
Cocos MCP
```

The Execution Controller must pass only approved task data to the Capability
Registry and MCP Adapter. It must preserve forbidden scope, proof target, and
rollback expectations.

## 4. Error Handling

MCP failure:

- stop execution
- record MCP server and tool name
- capture logs or error output
- mark task as failed or blocked
- request human review before retry

Cocos operation failure:

- preserve current project state
- record expected and actual result
- list modified or generated files
- propose rollback or repair
- stop before additional Cocos operations

Validation failure:

- report failed validation command or proof target
- keep task completed but not verified
- route result to QA Agent
- require approval before repair

The controller must not hide partial failures.

## 5. QA Agent Feedback Loop

QA Agent receives:

- task id
- approved scope
- execution result
- validation output
- modified or generated files
- warnings
- blockers

QA Agent returns:

- PASS
- NEEDS_REPAIR
- BLOCKED
- FAIL

QA feedback may update the next proposal, but it must not trigger automatic
repair or continuation.

## 6. Human Approval Checkpoints

Human approval is required before:

- moving from planned to approved
- executing MCP tools
- modifying Cocos scenes, prefabs, scripts, assets, metadata, settings, or build
  outputs
- retrying failed write operations
- rollback that touches files
- staging, committing, pushing, tagging, or releasing
- expanding task scope

`next recommended command` is optional advice only and never approval.

## Safety Boundary

This document creates:

- no runtime implementation
- no MCP changes
- no autonomous execution
- no Cocos project changes
- no Agent implementation

It is documentation only.
