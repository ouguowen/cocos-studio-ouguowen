# v5.3 Capability Index System

Status: documentation only

Scope: AI Game Studio capability discovery and context routing architecture

This document defines the v5.3 Capability Index System for AI Game Studio. It
is an architecture document only. It does not create a registry file, parser,
router, Agent implementation, or execution code. It does not modify MCP, Agent
runtime, Cocos projects, examples, or scripts.

## 1. Capability Index Positioning

AI Game Studio contains a growing set of design, production, validation,
release, Agent, tool, and Cocos-related capabilities. A user request normally
needs only a small subset of those capabilities, but an assistant cannot select
that subset efficiently when capability knowledge is scattered across many
documents.

The Capability Index System provides a compact discovery layer between a user
request and module loading. It answers three questions before detailed context
is read:

- What capability does this request need?
- Which modules are required to understand that capability?
- Which dependencies must be present before the task can proceed?

The system has two related concepts:

- **Capability Registry**: the canonical catalog of capability metadata.
- **Capability Index**: the compact, searchable view used for task routing.

The Registry describes capabilities. The Index exposes the minimum metadata
needed to find them. Neither component executes a task or grants permission.

### Primary goal

The primary goal is to reduce the need for Codex to read all documentation for
every request.

Instead of loading the entire Skill repository, the assistant should first read
a small capability index, select one or more relevant capability records, and
then load only the required modules and their declared dependencies.

### Non-goals

v5.3 does not define:

- executable capability registration
- automatic Skill installation
- Agent implementation or Agent runtime behavior
- automatic MCP calls
- Cocos Creator project operations
- permission escalation
- automatic dependency installation
- a replacement for human approval

## 2. Capability Registry

Capability Registry is the logical source of truth for discoverable AI Game
Studio capabilities.

Each record represents one bounded ability, such as project context retrieval,
game classification, architecture planning, code review, runtime validation,
release review, or Cocos scene operation planning.

### Required capability fields

Every Capability Registry entry contains:

| Field | Purpose |
|---|---|
| `capability_id` | Stable and unique capability identity |
| `description` | Short statement of the capability outcome and boundary |
| `trigger_keywords` | Terms or intent signals used during request matching |
| `required_modules` | Canonical modules that must be loaded for this capability |
| `dependencies` | Other capabilities or prerequisites required first |
| `priority` | Tie-breaking weight used after relevance and safety checks |

### Field rules

#### Capability ID

`capability_id` must be stable, unique, readable, and independent of a specific
Agent or MCP provider.

Recommended form:

```text
domain.action.subject
```

Examples:

```text
memory.read.project-context
design.classify.game-type
architecture.plan.runtime-kernel
quality.review.code
validation.verify.runtime-template
release.review.readiness
```

Changing an Agent assignment or tool provider must not require changing the
capability identity.

#### Description

`description` explains the result the capability can produce and the boundary
it must preserve. It should be short enough for index loading and specific
enough to distinguish nearby capabilities.

A description must not claim implementation authority. For example, a
capability may describe Cocos scene planning without implying permission to
modify a real Cocos project.

#### Trigger keywords

`trigger_keywords` are routing hints, not exact command names. They may include:

- user vocabulary
- domain terminology
- requested artifact names
- operation verbs
- validation or release intent

Keyword matching is only one signal. The router must also evaluate target path,
requested output, explicit restrictions, production stage, and current approval
state.

#### Required modules

`required_modules` lists the smallest canonical document set needed to use the
capability safely.

Rules:

- list canonical sources rather than duplicated summaries
- keep the module set narrow
- do not list the entire repository
- separate always-required modules from conditional dependencies
- include safety or approval modules when the capability crosses a write or
  execution boundary

#### Dependencies

`dependencies` describes prerequisites between capabilities or decisions.

Dependency examples:

- architecture planning may depend on project context
- implementation planning may depend on game classification
- runtime execution may depend on write approval
- release review may depend on QA evidence

Dependencies describe ordering and context needs. They do not authorize the
dependent operation.

#### Priority

`priority` resolves competing matches after relevance, scope, safety, and
dependency checks.

Priority must not override:

- an explicit user instruction
- a forbidden path or file class
- human approval requirements
- missing dependencies
- a more specific capability match

Priority is a routing preference, not an execution privilege.

### Conceptual registry record

This example is descriptive only. v5.3 does not create a machine-readable
manifest in this phase.

```yaml
capability_id: quality.review.code
description: Review a scoped code change for defects, regressions, and missing tests.
trigger_keywords:
  - code review
  - review changes
  - regression risk
required_modules:
  - production/review-system.md
  - protocols/quality-gates.md
dependencies:
  - context.read.diff-scope
priority: 80
```

## 3. Capability Index Architecture

Capability Index is a compact projection of the Capability Registry. It is
optimized for discovery and should be much cheaper to load than the detailed
modules it references.

```text
Capability Registry
  -> Compact Capability Index
  -> Candidate Search
  -> Dependency Resolution
  -> Module Selection
```

### Index responsibilities

The Capability Index must support:

- discovery by user intent
- discovery by target artifact
- discovery by domain
- discovery by requested operation
- ranking of overlapping candidates
- lookup of required modules
- lookup of capability dependencies
- detection of unresolved or ambiguous requests

### Index boundaries

The Capability Index must not:

- duplicate full module content
- become a second policy source of truth
- select an Agent implementation directly
- execute tools
- call MCP
- modify a Cocos project
- bypass approval or safety gates

The Index points to canonical modules. Those modules remain authoritative for
domain rules, permissions, validation, and stop conditions.

## 4. Task Routing Flow

The v5.3 task routing flow is:

```text
User Request
  -> Capability Index
  -> Module Selection
  -> Context Loading
  -> Execution
```

Execution in this flow means handing an approved, context-complete task to the
existing execution layer. It does not mean that the Capability Index executes
the task itself.

### Step 1: User Request

The router extracts a request profile:

- intent
- requested outcome
- target artifact or path
- explicit constraints
- validation expectations
- approval state
- production stage when relevant

The original request remains authoritative. Routing must not silently expand
its scope.

### Step 2: Capability Index

The router queries compact capability records using intent, terminology, target
artifact, and operation type.

The result is a ranked candidate set, not an immediate execution decision.

Candidate selection order:

1. remove candidates outside the explicit task scope
2. remove candidates blocked by safety or permission boundaries
3. prefer exact intent and artifact matches
4. prefer narrower capabilities over broad capabilities
5. verify dependencies
6. use priority only as a final tie-breaker

When no safe match exists, the router returns an unresolved capability result.
When multiple equally specific matches remain, the router requests clarification
or selects only the common read-only context.

### Step 3: Module Selection

For each selected capability, Module Selection collects:

- required modules
- unresolved dependencies
- mandatory safety context
- task-specific validation guidance

The selector then removes duplicates and rejects unrelated modules.

Module Selection should produce a small context plan such as:

```text
Selected capability:
  quality.review.code

Load:
  core context summary
  review system
  quality gate
  current diff

Do not load:
  Agent registry
  release automation
  Cocos runtime proof
  unrelated game design modules
```

### Step 4: Context Loading

Context Loading reads only the selected modules and their required dependency
chain. It applies the budget classes defined by the Context Optimization Engine.

Loading must stop when:

- the selected capability is sufficiently supported
- the context budget is reached
- an unexpected dependency would broaden the task
- a required approval is missing
- module metadata conflicts with the user request

If context is insufficient, the router may expand one dependency level at a
time. It must not fall back to full repository loading by default.

### Step 5: Execution

The existing Runtime Kernel, Agents, tools, MCP provider, or manual workflow may
perform execution only when the task independently authorizes that behavior.

The execution handoff contains:

- selected capability IDs
- loaded module set
- resolved dependencies
- approved scope
- forbidden actions
- validation requirement
- human approval state

Capability selection alone never authorizes writes, tool calls, MCP operations,
Agent runtime changes, or Cocos project modification.

## 5. Matching and Ranking Model

The router should evaluate multiple signals rather than rely on raw keyword
frequency.

### Match signals

- explicit capability or command name
- requested artifact type
- action verb
- domain terminology
- target repository or path
- production stage
- explicit forbidden actions
- requested proof or validation
- current Project Memory when applicable

### Ranking order

The conceptual ranking order is:

```text
Scope compatibility
  -> Safety compatibility
  -> Intent relevance
  -> Artifact relevance
  -> Specificity
  -> Dependency readiness
  -> Priority
```

This order prevents a high-priority but broad capability from defeating a
lower-priority exact match.

### Conflict handling

When capabilities overlap:

- combine them only when both are necessary for the requested outcome
- choose the narrower capability when one fully covers the task
- preserve both dependency chains when they own different required decisions
- stop before execution when their permission boundaries conflict
- never resolve ambiguity by loading every candidate module

## 6. Context Reduction Strategy

v5.3 reduces documentation loading through progressive discovery.

### Before v5.3

```text
User Request
  -> Read broad Skill documentation
  -> Search for relevant rules
  -> Plan task
```

This approach scales poorly as the repository grows. The cost of a small task
increases with the total number of documents rather than with task complexity.

### With v5.3

```text
User Request
  -> Read compact Capability Index
  -> Select capability records
  -> Load referenced modules only
  -> Plan task
```

The desired property is:

```text
Context cost ~= selected capability complexity
```

It should not be:

```text
Context cost ~= total repository size
```

### Reduction rules

- keep capability descriptions compact
- point to canonical modules instead of copying their content
- load dependency metadata before dependency documents
- stop dependency traversal at the first sufficient context set
- keep historical architecture documents out of normal execution context unless
  the task explicitly needs architectural history
- release task-specific context after completion
- promote only reviewed results into Project Memory or experience records

### Safety preservation

Context reduction must not remove mandatory safety facts. A small context set
still needs the rules relevant to its risk class.

For example:

- docs-only work loads docs scope and diff review boundaries
- package maintenance loads package validation and repository scope
- Agent runtime work requires explicit approval and Agent safety context
- Cocos project work requires approved project path, write scope, and runtime
  proof rules when applicable

Smaller context means more relevant context, not fewer safeguards.

## 7. Relationship With Existing Systems

### v3.5 Capability Marketplace

The v3.5 Capability Marketplace defines how reusable capabilities are described,
versioned, governed, approved, and associated with Agents.

v5.3 provides the compact discovery path over those capability records:

```text
Capability Marketplace
  -> Capability Registry
  -> Capability Index
  -> Task-specific capability selection
```

The Marketplace governs capability lifecycle. The Capability Index optimizes
lookup and routing. v5.3 does not replace marketplace versioning, status,
permission, validation, or human approval rules.

Related document: [v3.5 Capability Marketplace Architecture](v3.5-capability-marketplace-architecture.md)

### v5.2 Context Optimization Engine

The v5.2 Context Optimization Engine defines Core, Domain, and Execution context
layers, Task Router behavior, context budgets, Skill Manifest design, and memory
layering.

v5.3 makes capability discovery the front door to that context selection:

```text
User Request
  -> Capability Index
  -> v5.2 Task Router
  -> Context Budget Manager
  -> Selected Core / Domain / Execution modules
```

v5.2 answers how much context may be loaded and which layer it belongs to. v5.3
answers which capability needs that context and which modules describe it.

Related document: [v5.2 Context Optimization Engine](v5.2-context-optimization-engine.md)

### v5 Runtime Kernel

The Runtime Kernel consumes the routing result after capability discovery and
context loading. Its Capability Kernel can use selected capability identities,
while its Planner and Execution Kernels use the resolved module and dependency
set.

```text
Capability Index
  -> Context Optimization Engine
  -> Runtime Kernel
  -> Capability Kernel / Planner Kernel / Execution Kernel
  -> Agents / Tools / MCP / Cocos Creator when approved
```

v5.3 remains an architecture-level input to the Runtime Kernel. It does not
implement Runtime Kernel behavior.

## 8. Human Approval and Safety Boundary

Human approval remains mandatory wherever the existing architecture requires
it.

Capability discovery does not equal:

- capability installation
- capability enablement
- task execution
- permission to write files
- permission to load beta or deprecated behavior
- permission to call MCP
- permission to change Agent runtime
- permission to modify a Cocos project

The router must preserve explicit user restrictions in the execution handoff.
Dependencies must not be used to expand scope silently. If a dependency requires
a blocked action, routing stops before execution.

### Required stop conditions

Stop before execution when:

- no capability matches the request safely
- the requested target path is unclear
- required modules are missing or contradictory
- a dependency requires unapproved writes
- Agent runtime or MCP behavior would change without explicit approval
- Cocos project access would be required without an approved project path
- capability selection would broaden the requested artifact scope
- human approval required by an existing protocol is absent

## 9. Future Implementation Contract

A future implementation phase may define a machine-readable registry, index
builder, router, or validation rules, but that work requires a separate approved
scope.

Any implementation proposal must define:

- canonical registry ownership
- schema and versioning
- duplicate capability detection
- dependency cycle detection
- keyword collision handling
- index generation and freshness rules
- context budget integration
- approval-state propagation
- rollback plan
- validation plan

No future index implementation may become a second source of policy truth or an
automatic permission system.

## Phase Boundary

v5.3 accepts:

- Capability Registry concept design
- required metadata field design
- capability matching and ranking design
- task routing flow design
- selective module and context loading design
- relationship design for v3.5, v5.2, and the Runtime Kernel

v5.3 does not authorize:

- executable registry or index files
- parser, router, loader, or runtime code
- MCP modification
- Agent runtime modification
- Cocos project modification
- examples modification
- scripts modification
- automatic capability execution
- automatic permission escalation

This phase ends with architecture documentation and repository validation only.
