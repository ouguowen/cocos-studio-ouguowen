# v5.2 Context Optimization Engine

Status: documentation only

Scope: AI Game Studio Skill architecture design

This document defines the v5.2 Context Optimization Engine for AI Game Studio. It is an architecture document only. It does not modify MCP, does not modify Agent runtime, does not modify any Cocos project, and does not create execution code.

## 1. Current Skill Full-Load Problem Analysis

AI Game Studio has grown from a compact Cocos production Skill into a multi-layer studio architecture. The current repository contains project memory, workflow gates, Agent architecture, validation rules, release rules, example packs, runtime proof guidance, and open-source roadmap documents.

When an assistant loads too much of the Skill at the start of every task, several problems appear.

### Context weight problem

Full Skill loading consumes attention and token budget before the real task starts. Small tasks such as checking package manager policy, writing one open-source architecture document, or reviewing a single script do not need the entire Agent registry, release pipeline, map model, economy model, runtime proof protocol, and Cocos implementation rules at the same time.

The result is slower response time, noisier reasoning, and a higher chance that the assistant will treat every request as a full production audit.

### Routing confusion problem

When all modules are loaded together, unrelated rules compete for priority. A documentation-only architecture request can accidentally inherit real Cocos implementation concerns. A package management task can accidentally pull in Agent handoff logic. A release review can be mixed with first-MVP implementation rules.

The Skill needs a router that decides which context family is relevant before heavy files are loaded.

### Safety dilution problem

More context does not automatically mean safer behavior. If every rule is visible at once, the truly relevant stop condition can be hidden among many unrelated constraints.

Context optimization must preserve safety gates while reducing unrelated context. The correct goal is smaller relevant context, not weaker governance.

### Repeated-load problem

Some facts are stable across many tasks:

- this is a Cocos Creator 3.8.8 production-control Skill
- Skill maintenance is separate from real game project implementation
- MCP is an execution channel, not the identity of the Skill
- Cocos scene, prefab, meta, and real project runtime changes require explicit scope
- `next recommended command` is not automatic approval

These facts should live in compact core context rather than being rediscovered by loading many files repeatedly.

### Documentation growth problem

Open-source architecture documents are now a historical chain. v1, v2, v3, v4, and v5 files describe different layers of AI Game Studio. Loading the whole chain for every request is unnecessary. The system needs a manifest and budget model that can select the smallest useful historical slice.

## 2. Context Optimization Architecture

The Context Optimization Engine is a planning layer that decides what context should be loaded for a task before the assistant performs work.

It sits between the user request and the existing Skill repository:

```text
User Request
  ↓
Context Optimization Engine
  ↓
Task Router
  ↓
Context Budget Manager
  ↓
Skill Manifest
  ↓
Selected Context Set
  ↓
AI Game Studio Runtime Kernel
```

The engine has four responsibilities:

- classify the task
- choose the relevant context layer
- enforce a context budget
- preserve mandatory safety context

It does not execute tasks. It does not call MCP. It does not create Agent runtime behavior. It does not edit Cocos files. It only decides what the assistant should know before acting.

### Architecture components

Task Router classifies the request by intent, risk, file scope, and production stage.

Context Budget Manager limits how much context can be loaded for the selected task class.

Skill Manifest describes available modules, their purpose, trigger conditions, risk level, and dependencies.

Memory Layer Selector chooses the right memory scope: stable Skill identity, project memory, task memory, or historical experience.

Safety Context Injector adds mandatory stop rules when the request touches risky boundaries.

## 3. Skill Layered Design

The Skill should be loaded through three layers:

- Core Layer
- Domain Layer
- Execution Layer

Each layer has a different purpose and different loading cost.

### Core Layer

Core Layer is the smallest always-available context. It describes identity, scope, safety boundaries, and routing vocabulary.

Core Layer should include:

- Skill identity
- operation mode selection
- context loading policy
- path scope rules
- write approval rules
- core safety boundaries
- package or repository maintenance boundaries when relevant

Core Layer answers:

- What is this Skill?
- What is this task allowed to touch?
- Is this documentation, package management, game implementation, QA, or release work?
- What must never happen by default?

Core Layer should stay compact enough to be loaded at the start of most tasks.

### Domain Layer

Domain Layer contains task-specific knowledge. It is loaded only after Task Router identifies the relevant domain.

Domain examples:

- project memory
- game classification
- level data model
- map model
- numerical design
- economy design
- animation presentation
- Agent architecture
- capability marketplace
- release automation
- runtime validation
- open-source architecture history

Domain Layer answers:

- Which subject area is this task about?
- Which design rules or architecture records are relevant?
- Which historical version documents matter?
- Which domain-specific risks exist?

Domain Layer should not be loaded wholesale. Each task should select one or a small number of domain modules.

### Execution Layer

Execution Layer contains concrete operation rules, validation commands, proof requirements, and tool use boundaries.

Execution Layer examples:

- docs validation
- package check
- generated artifact check
- runtime template TypeScript validation
- Cocos preview proof protocol
- QA review checklist
- release review checklist
- diff review protocol

Execution Layer answers:

- What command or proof is required?
- Which files are expected to change?
- Which files are forbidden?
- How should success, failure, or blocker status be reported?

Execution Layer is loaded only when execution, validation, or review is needed. Documentation planning can often stay in Core Layer plus a small Domain Layer.

## 4. Task Router Design

Task Router selects the context path for a user request.

### Router inputs

Task Router reads:

- user intent
- target artifact
- target path
- requested validation
- production stage
- risk boundary
- explicit forbidden actions
- existing workspace state

### Router categories

The initial routing categories are:

| Category | Example request | Context path |
|---|---|---|
| Documentation architecture | create v5.2 architecture doc | Core Layer + open-source architecture Domain Layer |
| Skill package maintenance | fix package manager, run check | Core Layer + package Execution Layer |
| Agent architecture design | design Agent registry model | Core Layer + Agent Domain Layer |
| Real Cocos implementation | create scene or runtime script | Core Layer + Cocos Domain Layer + pre-write Execution Layer |
| Runtime proof | validate browser preview | Core Layer + runtime proof Execution Layer |
| QA or release | review build readiness | Core Layer + QA or release Domain Layer + gate Execution Layer |
| Memory update | record project truth | Core Layer + memory Domain Layer |

### Router decisions

Task Router must decide:

- `DOCS_ONLY`
- `PACKAGE_MAINTENANCE`
- `SKILL_ARCHITECTURE`
- `AGENT_DESIGN_ONLY`
- `AGENT_RUNTIME_CHANGE`
- `COCOS_PROJECT_WORK`
- `MCP_PROVIDER_WORK`
- `VALIDATION_ONLY`
- `QA_GATE`
- `RELEASE_GATE`
- `STOP_FOR_SCOPE`

Only some decisions allow file writes. Documentation architecture allows Markdown file creation inside the approved docs path. Package maintenance allows package and validation-script changes only when requested. Cocos project work and Agent runtime changes require explicit approval and are not implied by architecture tasks.

### Router stop conditions

Task Router must stop when:

- target path is unclear
- documentation work begins to imply runtime implementation
- MCP behavior would change without approval
- Agent runtime would change without approval
- Cocos project files would be touched without an approved project scope
- validation would write generated files into repository outputs unexpectedly
- context budget would require broad repository loading without a real reason

## 5. Context Budget Management

Context Budget Manager controls how much information can be loaded for each task.

### Budget classes

`MIN_CONTEXT` is for small checks and single-file documentation tasks.

Allowed:

- Skill identity summary
- operation mode
- direct target file
- direct validation command
- current git status

`TASK_CONTEXT` is for normal Skill maintenance or architecture work.

Allowed:

- Core Layer
- one or two Domain Layer files
- target file or neighboring version documents
- validation command references

`GATE_CONTEXT` is for approval, QA, release, runtime proof, or risky execution.

Allowed:

- Core Layer
- selected gate protocol
- proof protocol
- diff review protocol
- relevant target files

`AUDIT_CONTEXT` is for full Skill review, validation script review, release governance, or safety audit.

Allowed:

- broad module scan
- multiple protocol files
- cross-document link checks
- validation script inspection

### Budget rules

The engine should prefer the smallest class that can safely answer the request.

Budget escalation is allowed when:

- validation fails
- scope expands
- a risky file class appears
- current context contradicts the requested action
- an integration boundary is unclear

Budget escalation should be explicit. The assistant should know why more context is needed.

### Budget anti-patterns

Avoid:

- loading every Agent file for documentation-only work
- loading runtime proof rules for package manager fixes
- loading release documents for early architecture design
- loading Cocos implementation rules for open-source roadmap notes
- treating a validation command as permission to inspect unrelated projects

## 6. Skill Manifest Design

Skill Manifest is a machine-readable index concept for routing context. This phase defines the design only and does not create a manifest file or parser.

### Manifest purpose

The manifest should allow Task Router to answer:

- which modules exist
- what each module is for
- when it should load
- what dependencies it has
- whether it is safety-critical
- whether it can trigger file writes
- whether it belongs to Core, Domain, or Execution Layer

### Manifest fields

A future manifest entry should include:

```text
id: stable module id
path: repository path
layer: core | domain | execution
domain: memory | agent | runtime | package | release | cocos | mcp | docs
summary: short purpose
load_when: trigger conditions
requires: prerequisite modules
forbids: actions that remain blocked
risk_level: low | medium | high
write_scope: none | docs | package | generated | project
validation: optional validation command or proof
```

### Manifest rules

The manifest must not become a second source of truth for production policy. It should route to canonical documents, not duplicate them.

The manifest must not automatically authorize execution. It can say which context is relevant, but approval rules still come from the appropriate protocol.

The manifest must be cheap to read. It should help avoid full repository loading.

## 7. Memory Layering Strategy

Context optimization needs memory layering because not all memory has the same stability or cost.

### Stable Skill Memory

Stable Skill Memory contains identity and permanent safety boundaries:

- Skill purpose
- Cocos Creator 3.8.8 focus
- provider-neutral MCP stance
- separation between Skill maintenance and real game implementation
- mandatory approval for risky file classes
- operation mode vocabulary

Stable Skill Memory belongs in Core Layer.

### Project Memory

Project Memory contains current project truth:

- current milestone
- game type
- MVP status
- approved scope
- active assumptions
- open questions
- accepted risks

Project Memory should load only when the task depends on a specific game project or milestone decision.

### Task Memory

Task Memory contains current-turn working facts:

- requested file
- allowed changes
- forbidden changes
- validation commands
- observed dirty worktree
- blocker status

Task Memory should be compact and should expire when the task ends.

### Experience Memory

Experience Memory contains lessons from prior runs:

- common validation failures
- package manager traps
- context overload examples
- successful routing patterns
- repeated safety blockers

Experience Memory can improve routing, but it must not override current user instructions or canonical safety rules.

### Memory promotion

Information should be promoted only when it is reviewed:

- task observation becomes task memory immediately
- task memory becomes project memory only after owner review
- repeated task memory becomes experience memory only when it is useful across tasks
- experience memory becomes Skill rule only through controlled evolution

## 8. Relationship With Existing Architecture

### v2.10 Project Memory Engine

The v2.10 Project Memory Engine defines stable project truth. v5.2 uses it as the project-specific memory source, but does not load it for every task.

When a request depends on milestone truth, game type, approved scope, or active assumptions, Context Optimization Engine routes into Project Memory.

Related document: [v2.10 Project Memory Engine](v2.10-project-memory-engine.md)

### v3 Agent Architecture

v3 Agent Architecture defines role ownership, handoff, review, and collaboration patterns. v5.2 does not change Agent runtime behavior.

Context Optimization Engine loads Agent context only when the task involves Agent design, handoff, role responsibility, collaboration protocol, or Agent review. It should not load Agent runtime context for package management or ordinary docs tasks.

Related documents:

- [v3.1 Code Review Agent Architecture](v3.1-code-review-agent-architecture.md)
- [v3.3 Multi-Agent Collaboration Protocol](v3.3-multi-agent-collaboration-protocol.md)
- [v3.4 Agent Registry Architecture](v3.4-agent-registry-architecture.md)

### v4 AI Game Studio Architecture

v4 AI Game Studio Architecture defines the studio-level system view. v5.2 becomes the context-selection layer that lets the studio load only what a task needs.

The v4 system answers what AI Game Studio is. The v5.2 system answers what part of AI Game Studio should be active for this task.

Related document: [v4.1 AI Game Studio Architecture](v4.1-ai-game-studio-architecture.md)

### v5 Runtime Kernel

The v5 Runtime Kernel is the central runtime coordination concept. v5.2 supports it by deciding which context enters the Runtime Kernel before planning, capability matching, execution tracking, or validation begins.

Runtime Kernel coordinates task state. Context Optimization Engine prepares the smallest safe context for that coordination.

The relationship is:

```text
User Request
  ↓
Context Optimization Engine
  ↓
Runtime Kernel
  ↓
Memory Kernel / Planner Kernel / Capability Kernel / Execution Kernel
  ↓
Agents / Tools / MCP / Cocos Creator when approved
```

v5.2 does not implement Runtime Kernel code. It defines the context policy Runtime Kernel should eventually consume.

## Phase Boundary

v5.2 accepts:

- architecture documentation
- context layer design
- routing design
- budget design
- manifest design
- memory layering design
- integration explanation

v5.2 does not authorize:

- MCP modification
- Agent runtime modification
- Cocos project modification
- execution code creation
- automatic context loader implementation
- package manager changes
- validation script changes
- real game implementation

Any future implementation must open a separate approved scope, name expected files, preserve the existing safety gates, and pass validation before code is changed.
