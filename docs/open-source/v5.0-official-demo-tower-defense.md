# AI Game Studio Official Demo

Project name: **Tower Defense AI Demo**

This document defines the official demo design for AI Game Studio.

The goal is to prove that AI Game Studio can practically assist Cocos Creator game development through a complete, controlled workflow. This is documentation only. It does not create Cocos project files, modify MCP, implement Agents, modify `agents/*`, or change runtime execution logic.

## Demo Goal

Tower Defense AI Demo demonstrates how AI Game Studio can guide a developer from a one-sentence game idea to a scoped Cocos Creator production plan.

The demo should show:

- Game Design Intelligence converts a user idea into a design.
- Production Pipeline converts the design into tasks and modules.
- Codex can implement approved code tasks.
- Asset Pipeline manages asset requirements.
- Cocos MCP Integration can connect approved tasks to Cocos Creator.
- Runtime Validation checks the actual playable result.
- Release Pipeline prepares build and release outputs.

The demo is not a claim of autonomous game creation. Human approval remains required before project creation, file modification, Cocos/MCP execution, runtime validation, or release actions.

## 1. Game Design Intelligence

Input:

```text
Create a simple tower defense game where enemies move along a road and towers attack them.
```

Expected output:

- Game design document.
- Core gameplay loop.
- Numerical rules.
- Level plan.

### Game Design Document

The Game Design Document should define:

- Game title: Tower Defense AI Demo.
- Game type: tower defense.
- Target engine: Cocos Creator 3.8.8.
- Player goal: stop enemies before they reach the base.
- Core loop: build towers, start wave, defeat enemies, earn resources, upgrade defense.
- MVP scope: one map, one enemy type, one tower type, one wave sequence, one result state.
- Non-goals: no gacha, no shop, no save/load, no multiplayer, no live ops.

### Core Gameplay

Core gameplay should include:

- A visible path from spawn point to base.
- Enemies moving along the path.
- Towers detecting enemies in range.
- Towers firing at enemies.
- Enemy defeat and base reach outcomes.
- A simple objective state.
- A clear win or fail result for the demo level.

### Numerical Rules

Initial numerical rules should define placeholder values with purpose:

- Enemy health: enough to survive several tower hits.
- Enemy speed: slow enough to observe path movement.
- Tower range: large enough to validate target detection.
- Tower attack interval: readable for runtime proof.
- Wave count: small enough for demo validation.
- Starting resource: enough to place the first tower.

All numbers are demo placeholders until reviewed through numerical design.

### Level Planning

The first demo level should include:

- One scene.
- One road or path.
- One spawn point.
- One base point.
- Two to five enemies.
- One to three tower slots.
- One battle UI panel.
- One result display.

## 2. Production Pipeline

Production Pipeline turns the Game Design Document into production-ready structure and task planning.

Expected outputs:

- Project structure.
- Module division.
- Development tasks.

### Project Structure

The planned Cocos Creator project structure should separate:

- Scenes.
- Scripts.
- Prefabs.
- UI.
- Config.
- Art assets.
- Audio assets.
- Validation reports.

Example planning structure:

```text
assets/
  scenes/
  scripts/
    tower-defense/
  prefabs/
    towers/
    enemies/
    projectiles/
  ui/
  config/
    tower-defense/
  audio/
```

This document does not create those folders. It only defines the demo design.

### Module Division

The demo should be divided into:

- GameManager: owns game state and result.
- Player System: owns player resources and build permissions.
- Tower System: owns tower placement, target detection, and attack timing.
- Enemy System: owns enemy spawning, movement, health, and base reach.
- Wave System: owns wave configuration and spawn timing.
- UI System: owns HUD, resource display, wave status, and result display.
- Validation System: owns proof markers and runtime validation expectations.

### Development Tasks

Example task plan:

| Task ID | Task | Output |
| --- | --- | --- |
| TD-001 | Create GameManager plan | Game state and result contract |
| TD-002 | Create Player System plan | Resource and placement rules |
| TD-003 | Create Tower System plan | Tower targeting and attack contract |
| TD-004 | Create Enemy System plan | Enemy movement and health contract |
| TD-005 | Create Wave System plan | Wave data and spawn flow |
| TD-006 | Create UI System plan | HUD and result UI contract |
| TD-007 | Create runtime validation plan | Launch, gameplay, UI, error checks |

Each implementation task requires pre-write approval before Codex changes real project files.

## 3. Code Generation

Code Generation is performed through Codex only after a task has approved scope, target files, validation requirements, and rollback plan.

Codex-generated systems may include:

### Player System

Responsibilities:

- Store player resource amount.
- Validate tower placement cost.
- Spend resources when a tower is placed.
- Report resource changes to UI.

### Tower System

Responsibilities:

- Represent tower stats.
- Detect enemies in range.
- Select target.
- Fire projectile or apply damage.
- Expose proof logs for attack behavior when runtime validation is approved.

### Enemy System

Responsibilities:

- Spawn enemies from wave data.
- Move enemies along path points.
- Track health.
- Report defeated or base-reached state.

### Wave System

Responsibilities:

- Load wave configuration.
- Spawn enemies in controlled timing.
- Report wave start, wave complete, and all waves complete.

### UI System

Responsibilities:

- Display resources.
- Display wave state.
- Display base health or objective state.
- Display result state.
- Provide start-wave interaction if included in scope.

Codex must not generate code without approval, must not modify `.scene`, `.prefab`, `.meta`, package settings, or runtime project files unless explicitly approved, and must not claim runtime success without proof.

## 5. Cocos MCP Integration

Cocos MCP Integration explains how AI Game Studio connects approved plans to Cocos Creator.

Flow:

```text
Human-approved task
  -> AI Game Studio permission check
  -> MCP Adapter
  -> External Cocos MCP Server
  -> Cocos Creator 3.8.8
```

Possible approved MCP actions:

- Create or open project.
- Create scene.
- Create nodes.
- Add components.
- Bind prefabs.
- Query scene hierarchy.
- Validate references.
- Run preview.

Restricted MCP actions:

- Delete assets without confirmation.
- Modify project settings without approval.
- Raw-edit `.scene`, `.prefab`, `.anim`, or `.meta` files.
- Execute broad automation without scoped validation.

AI Game Studio does not replace MCP. It governs when and how MCP may be used.

## 6. Runtime Validation

Runtime Validation proves that the demo works in a real execution path after approved implementation.

Validation targets:

- Game startup.
- Battle flow.
- UI interaction.

### Game Startup

Checks:

- Project opens or preview starts.
- Expected scene loads.
- Camera and Canvas are visible.
- No critical console errors appear.

### Battle Flow

Checks:

- Wave starts.
- Enemies spawn.
- Enemies move along the road.
- Towers attack enemies.
- Enemy defeat or base reach is recorded.
- Result state appears.

### UI Interaction

Checks:

- Start wave button works if included.
- Resource display updates.
- Wave display updates.
- Result screen appears.
- UI does not falsely own gameplay truth.

Validation result format:

```text
Runtime Validation:
Status: PASS / NEEDS_REPAIR / BLOCKED
Scene:
Gameplay proof:
UI proof:
Console errors:
Known blockers:
```

## 7. Release Pipeline

The Release Pipeline prepares demo output only after validation passes.

Expected outputs:

- Build.
- Test Report.
- Release.

### Build

Build planning should define:

- Target platform.
- Build type.
- Version.
- Required assets.
- Known limitations.

### Test Report

The test report should include:

- Runtime validation result.
- Gameplay validation result.
- UI validation result.
- Performance notes.
- Console error summary.
- Open blockers.

### Release

Release preparation should include:

- Demo version.
- Release notes.
- Screenshot requirements.

No build, upload, tag, or release action is automatic.
