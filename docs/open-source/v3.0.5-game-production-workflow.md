# AI Game Studio v3.0.5 Game Production Workflow Architecture

This document defines the full game production lifecycle architecture for AI Game
Studio, from idea to release. It is documentation only. It does not create
runtime files, implement Agents, modify `agents/*`, change MCP, modify Cocos
Creator projects, or execute autonomous game development.

## 1. Game Production Lifecycle Architecture

AI Game Studio coordinates the complete game production flow through staged,
human-approved work.

```text
Idea
  |
  v
Game Design
  |
  v
Prototype
  |
  v
Production
  |
  v
Integration
  |
  v
Testing
  |
  v
Optimization
  |
  v
Release
```

Lifecycle stages:

| Stage | Input Data | Output Data | Agent Used | Tools Used |
| --- | --- | --- | --- | --- |
| Idea | user intent, genre, platform, constraints | game concept, target audience, risk list | Planner Agent, Game Designer Agent | Game Knowledge Engine, Project Memory |
| Game Design | game concept, genre template, constraints | GDD, MVP boundary, system list, asset list | Game Designer Agent, Planner Agent | Game Knowledge Engine, Asset Pipeline Engine |
| Prototype | GDD, MVP scope, first proof target | prototype plan, validation target, proof result | Planner Agent, Code Generation Orchestrator, QA Agent | Codex, Cocos MCP Adapter when approved |
| Production | approved MVP plan, task graph, asset plan | implemented systems, assets, scenes, configs | Code Generation Orchestrator, Asset Pipeline Agent | Codex, Asset Pipeline Engine, Cocos MCP Adapter |
| Integration | code tasks, asset tasks, scene plan | integrated scene, prefab links, UI flow | Code Generation Orchestrator, QA Agent | Cocos MCP Adapter, Game Validation Framework |
| Testing | integrated build, validation rules | bug reports, QA status, risk list | QA Agent | Game Validation Framework, runtime preview when approved |
| Optimization | performance reports, QA feedback | optimized tasks, reduced risk, release checklist | Planner Agent, QA Agent | Game Validation Framework, Project Memory |
| Release | validated game state, release checklist | release decision, release notes, archive record | Release Agent, QA Agent | Git workflow, CI, release review |

Each stage must define its inputs, outputs, responsible Agent, and validation
gate before moving forward.

## 2. Production Phase Architecture

AI Game Studio manages production as a controlled chain of design, planning,
generation, asset work, integration, and validation.

```text
Game Design Document
        |
        v
Task Planning
        |
        v
Code Generation
        |
        v
Asset Production
        |
        v
Integration
        |
        v
Validation
```

Production phase responsibilities:

- Game Design Document: defines the game loop, genre, systems, scenes, assets,
  UI, controls, and MVP boundary.
- Task Planning: converts the design into a dependency-aware task graph.
- Code Generation: dispatches approved code tasks to Codex through the Code
  Generation Orchestrator.
- Asset Production: converts design needs into production-ready asset tasks.
- Integration: connects scripts, assets, scenes, prefabs, UI, and config data.
- Validation: verifies code, scene, gameplay, UI, assets, and runtime behavior
  against the approved target.

No production phase may skip human approval when it changes real project files.

## 3. Agent Responsibility Mapping

AI Game Studio assigns clear responsibilities to each Agent role. Agents exchange
structured context, task results, validation records, and blocker reports.

Game Design Phase:

- Game Designer Agent owns game loop, genre fit, player experience, feature
  boundaries, and design constraints.
- Output handed to Planner Agent: GDD, MVP scope, system list, asset list,
  validation targets.

Programming Phase:

- Code Generation Orchestrator owns code task sequencing, Codex dispatch,
  expected files, validation requirements, retry boundaries, and diff review.
- Output handed to QA Agent: changed files, validation logs, runtime proof,
  blockers, rollback notes.

Art Phase:

- Asset Pipeline Agent owns asset requirements, naming rules, style consistency,
  import expectations, prefab needs, and asset validation rules.
- Output handed to Integration and QA: asset task list, generated or approved
  assets, binding expectations, style notes.

Testing Phase:

- QA Agent owns code review, scene validation, asset consistency, UI flow,
  gameplay loop checks, performance risks, and release readiness notes.
- Output handed to Planner Agent: pass/fail result, blockers, repair tasks,
  release risk.

Project Management:

- Planner Agent owns task decomposition, dependency order, approval checkpoints,
  status tracking, and next-slice planning.
- Output handed to human reviewer: proposal, scope, risks, validation plan, and
  optional next recommended command.

Agent handoff rules:

- Every handoff must include task id, owner, scope, allowed files, forbidden
  files, expected output, validation result, and blockers.
- Agent output is advisory until human approval authorizes execution.
- A failed handoff returns to Planner Agent for repair planning.

## 4. Game Type Workflow Template

Game type controls system design, technical architecture, task breakdown, and
asset needs. AI Game Studio must not force one universal workflow onto all games.

Tower Defense:

- System design: map path, tower placement, enemy waves, projectiles, objective,
  result flow.
- Technical architecture: wave config, tower scripts, enemy controller, path
  data, level scene, HUD.
- Task split: map model, tower system, enemy system, wave system, objective
  proof, result proof.
- Asset needs: towers, enemies, projectiles, map tiles, effects, HUD icons.

RPG:

- System design: character state, dialogue, inventory, quests, progression,
  combat or exploration.
- Technical architecture: character modules, item config, dialogue data, quest
  state, scene transitions, UI panels.
- Task split: character model, dialogue system, inventory model, quest flow,
  progression validation.
- Asset needs: characters, portraits, environments, item icons, dialogue UI,
  animation states.

Card Game:

- System design: card library, deck, hand, turn flow, battle resolution, effects.
- Technical architecture: card config, deck manager, hand UI, battle controller,
  effect resolver.
- Task split: card schema, deck system, hand display, turn system, effect proof.
- Asset needs: card art, card frames, icons, battle UI, effect visuals.

Platformer:

- System design: player movement, collision, hazards, enemies, collectibles,
  checkpoints, level completion.
- Technical architecture: player controller, level data, collision layers,
  enemy prefabs, checkpoint state.
- Task split: movement proof, collision proof, enemy proof, collectible proof,
  checkpoint proof.
- Asset needs: character sprites, tiles, hazards, enemies, collectibles, level
  UI, animations.

Simulation:

- System design: resource loops, actors, time progression, tasks, upgrades,
  state feedback.
- Technical architecture: simulation state, tick controller, actor systems,
  resource config, UI dashboard.
- Task split: state model, tick loop, actor behavior, resource flow, UI
  validation.
- Asset needs: actors, buildings, icons, panels, progress indicators, feedback
  effects.

## 5. Data Flow Architecture

Production data should move from idea to validated result through explicit,
auditable artifacts.

```text
Game Idea
    |
    v
Game Design Document
    |
    v
Project Memory
    |
    v
Task Graph
    |
    v
Code Tasks
    |
    v
Asset Tasks
    |
    v
Validation Results
```

System connections:

- v2.10 Project Memory Engine stores approved project identity, design
  decisions, architecture state, asset conventions, task history, and validation
  history.
- v2.8 Game Knowledge Engine provides genre rules, gameplay templates, system
  patterns, level design rules, and UI expectations.
- v2.9 Asset Pipeline Engine converts gameplay and visual requirements into
  asset categories, naming rules, import expectations, and validation checks.
- v3.0 Code Generation Orchestration converts task graph entries into approved
  Codex execution requests and validation-ready outputs.

Data flow rules:

- Design decisions must be traceable to tasks.
- Tasks must be traceable to changed files or generated assets.
- Validation results must be traceable to task ids.
- Project Memory may be read only when explicitly triggered and approved.
- Real project facts must not enter memory without approved project path and
  scope.

## 6. Human Approval Workflow

Human approval remains the control point between AI planning and execution.

```text
AI Planning
    |
    v
Human Approval
    |
    v
Execution
    |
    v
Validation
    |
    v
Report
```

Approval checkpoints:

- before writing project files
- before creating or changing scenes, prefabs, scripts, assets, metadata,
  package files, build settings, or project settings
- before using Cocos MCP or browser preview
- before retrying with broader scope
- before committing, pushing, tagging, or releasing

The approval packet must include:

- project path
- files to change
- forbidden files
- expected output
- validation plan
- rollback plan
- stop conditions

AI must not modify a project without explicit confirmation of path and scope.

## 7. Integration With Existing Architecture

The full AI Game Studio architecture combines v1.x Agent governance, v2.x Cocos
execution architecture, and v3.x production orchestration.

v1.x Agent Architecture:

- defines Agent identity, capability control, permission policy, handoff rules,
  workflow proof, and approval gates.
- provides the governance layer for Planner, Designer, Code Generation, Asset,
  QA, and Release roles.

v2.x Cocos MCP Layer:

- defines MCP adapter, connection proof, capability mapping, execution
  controller, Cocos automation architecture, and validation boundary.
- provides the future execution and validation path for Cocos Creator 3.8.8.

v2.8 Game Knowledge Engine:

- supplies genre templates, system rules, numerical design guidance, level
  design patterns, UI rules, and asset requirement reasoning.

v2.9 Asset Pipeline Engine:

- supplies asset requirement analysis, naming convention, engine integration
  rules, style consistency, and asset validation.

v2.10 Project Memory Engine:

- supplies long-term project identity, design decisions, asset memory, code
  structure memory, and Agent history when explicitly triggered.

v2.11 Game Validation Framework:

- supplies validation domains for code, assets, scenes, gameplay, UI,
  performance risk, and QA feedback.

v3.0 Code Generation Orchestration:

- supplies approved code task planning, Codex dispatch, retry strategy, Cocos
  validation flow, and human approval gates.

Together, these modules form an AI Game Studio that can plan, coordinate,
validate, and release game work while keeping real project execution under human
control.
