# AI Game Studio v2.0.5 Cocos Creator 3.8.8 Capability Map

## Purpose

This document defines the capability boundary between AI Game Studio and Cocos
Creator 3.8.8.

This is a research and architecture document only. It does not implement MCP,
run Cocos automation, execute Agents, or modify Cocos projects.

## 1. Cocos Creator 3.8.8 Architecture Overview

Editor layer:

- provides project, scene, asset, prefab, inspector, extension, preview, and
  build workflows
- owns serialized editing of scenes, prefabs, animations, and metadata
- must be treated as the safe surface for Cocos asset modification

Runtime layer:

- runs TypeScript components, lifecycle methods, scene logic, UI, animation, and
  gameplay behavior
- proves user-visible behavior only through runtime/browser/editor preview
  evidence when required

Asset system:

- manages imported files, generated metadata, resource paths, asset UUIDs,
  bundles, dependencies, and import status
- must not be bypassed by raw edits to serialized asset references

Scene system:

- stores node hierarchy, transforms, components, cameras, Canvas, UI, and
  runtime entry points
- must be modified through editor-safe workflows when serialized scene data is
  involved

Component system:

- attaches engine components and user scripts to nodes
- exposes configurable properties through the editor and runtime component model

Extension system:

- allows editor tooling and automation surfaces
- must remain permission-controlled and provider-neutral in AI Game Studio

## 2. Cocos Project Capability Map

### Scene Capability

Scene capability includes:

- create scene
- open scene
- save scene
- inspect scene hierarchy

Scene work requires explicit project path, scene path, operation scope, and
proof target when runtime behavior matters.

### Node Capability

Node capability includes:

- create node
- delete node
- rename node
- transform modification
- parent-child relationship

Node operations must preserve hierarchy intent and stop on unexpected references
or forbidden scene scope.

### Component Capability

Component capability includes:

- add component
- remove component
- configure component properties

Examples:

- Sprite
- Label
- Animation
- Collider
- Script

Component changes must be validated through hierarchy, inspector state, script
binding, and runtime proof when behavior changes.

### Prefab Capability

Prefab capability includes:

- create prefab
- modify prefab
- instantiate prefab
- prefab dependency management

Prefab work must preserve references and avoid raw text edits to serialized
prefab files.

### Asset Capability

Asset capability includes:

- asset discovery
- asset import
- asset metadata
- texture
- audio
- animation
- material

Asset operations require source, owner, purpose, approved import path, metadata
awareness, and rollback strategy.

### UI Capability

UI capability includes:

- Canvas
- Widget
- Layout
- Button
- Label
- Panel
- UI scaling

UI automation should preserve visible layout, readable text, responsive
behavior, and runtime visibility proof when required.

### Animation Capability

Animation capability includes:

- Animation clips
- Animation controller
- State machine
- Character animation workflow

Animation must express state and feedback without owning gameplay truth.

### Script Capability

Script capability includes:

- TypeScript script generation
- component binding
- lifecycle methods

Examples:

- `onLoad`
- `start`
- `update`

Script work requires approved file scope, validation, and runtime proof when
behavior changes.

### Build Capability

Build capability includes:

- build process
- platform targets
- validation

Build work requires explicit platform, output, configuration, and release scope
approval.

## 3. AI Game Studio Capability Boundary

AI can request:

- scene creation
- node operations
- asset queries
- code generation
- validation

AI cannot directly do:

- modify engine core
- bypass permissions
- destroy project data
- raw-edit serialized Cocos assets without approved editor-safe workflow
- claim runtime proof without actual approved proof
- execute Cocos/MCP operations by default

AI Game Studio proposes and validates. Human approval remains the authority for
project modification and high-impact operations.

## 5. Permission Model

Planner:

- read-only by default
- prepares scope, risks, context needs, and approval checkpoints

Designer:

- scene, UI, and resource planning
- does not modify project files by default

Coder:

- script modification only after explicit file-scope approval
- does not use Cocos/MCP by default

QA:

- testing and validation
- reviews proof, diffs, logs, and blockers

System:

- restricted operations
- cannot perform destructive, release, build, or project configuration changes
  without approval

## 6. Human Approval Checkpoints

Automatic, when already inside approved scope:

- create files
- generate scripts
- inspect assets

Approval required:

- delete assets
- project configuration changes
- release build
- Cocos/MCP execution
- scene, prefab, animation, or metadata writes
- package, build, settings, or CI changes
- destructive operations

Automatic actions still require the current task to have an approved scope. No
operation is automatic outside its approved boundary.
