# v1.6 Planner Agent Prototype

This document prepares the future Planner Agent prototype architecture for AI
Game Studio. It is documentation only. It does not implement an Agent, create
runtime code, execute MCP, automate Cocos, or modify real Cocos project files.

## 1. Planner Agent Role Definition

The Planner Agent is the first proposed Agent role because planning is the
safest place to introduce Agent assistance.

Responsibilities:

- clarify user intent
- identify task type
- define approved and forbidden scope
- select required capability
- request only necessary context
- prepare pre-write proposals
- identify validation commands
- define stop conditions
- prepare handoff material

The Planner Agent does not approve its own plan, execute implementation, write
files by default, inspect real Cocos projects by default, or run Cocos/MCP.

## 2. Planner Input/Output Model

Allowed inputs:

- current user request
- lightweight core context
- explicitly named files
- approved git status or diff output
- approved project path, if provided
- approved memory snippets, if memory access is triggered

Expected outputs:

- task summary
- task type
- recommended operation mode
- files to create or modify
- forbidden files
- required context
- required capability
- approval checkpoint
- validation commands
- rollback plan
- stop conditions

## 3. User Request Analysis Flow

Analysis flow:

```text
User Request
      |
      v
Intent Detection
      |
      v
Scope Boundary Check
      |
      v
Task Type Classification
      |
      v
Risk Classification
      |
      v
Planner Output
```

The Planner must stop if the request is ambiguous, touches forbidden files,
requires Cocos/MCP without approval, or expands beyond the approved task.

## 4. Task Decomposition Strategy

The Planner decomposes work into small approval-safe slices:

- one documentation patch
- one policy patch
- one validation patch
- one implementation story
- one QA review
- one commit/push step

It should not combine unrelated architecture, runtime, validation, release, and
cleanup changes into one plan.

Each slice should define:

- goal
- scope
- allowed files
- forbidden files
- validation
- rollback
- completion signal

## 5. Capability Matching Process

Capability matching selects the narrowest capability that satisfies the task.

Matching inputs:

- user intent
- task type
- game-development category
- approved files
- required proof
- required context
- risk level

Matching outputs:

- selected capability
- reason for selection
- required permission
- required validation
- stop conditions

Capability matching must not grant permission by itself.

## 6. Context Request Process

The Planner requests context in layers:

1. Core context
2. Task context
3. Optional context only when directly triggered
4. Memory context only after explicit approval

The Planner should explain why each non-core context source is needed. It should
avoid loading proof, archive, example, Agent, runtime, or memory context by
default.

## 7. Memory Retrieval Process

Memory retrieval is optional and trigger-controlled.

The Planner may request memory only when:

- the task needs project-specific continuity
- the user has approved the active project path
- the memory category is named
- read scope is approved

Memory retrieval output should include:

- memory source
- relevance
- version or review marker
- conflict risk
- whether the memory can be used safely

## 8. Agent Handoff Preparation

The Planner prepares handoff material for future Designer, Coder, or QA roles.

Handoff package:

```text
task_id:
from:
to:
intent:
approved_scope:
forbidden_scope:
required_context:
required_permission:
expected_output:
validation_required:
stop_conditions:
audit_note:
```

Handoff does not activate the next Agent automatically. Human approval remains
required.

## 9. Human Approval Checkpoints

Approval is required before:

- writing files
- reading real project files
- reading or writing memory
- using Cocos Creator, Cocos MCP, browser preview, or runtime console
- changing approved scope
- handing off to implementation
- staging, committing, pushing, tagging, or releasing

The Planner can propose approval checkpoints, but cannot approve them.

## 10. Planning Result Format

Recommended Planner output:

```text
PLANNER_RESULT
task:
task_type:
scope:
allowed_files:
forbidden_files:
required_context:
selected_capability:
approval_required:
validation_commands:
rollback:
stop_conditions:
next_recommended_command:
```

`next_recommended_command` is optional advice only and never automatic
continuation.

## 11. Failure Handling

If planning fails, return:

- BLOCKED or NEEDS_CLARIFICATION
- reason
- missing approval, context, path, or scope
- safest next question or repair step
- files that must remain untouched

The Planner must not guess through missing safety information.

## 12. Future Runtime Implementation Roadmap

Possible roadmap:

1. keep Planner Agent prototype documentation-only
2. add validation coverage for Planner output requirements
3. create read-only Planner CLI proposal command
4. add capability matching without execution
5. add permission checks
6. add memory request gate
7. add handoff package generation
8. add Cocos/MCP integration only after separate runtime and approval review

## Safety Boundary

This document does not create:

- Agent implementation
- runtime code
- MCP execution
- Cocos automation
- planner CLI command
- real project memory
- real Cocos project changes

The Planner Agent remains future architecture only.
