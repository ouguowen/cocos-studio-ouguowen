# v2.0 Cocos Automation Architecture

This document prepares the future Cocos Automation architecture for AI Game
Studio. It is documentation only. It does not implement Cocos automation, MCP
runtime, Agent runtime, autonomous execution, or real project modification.

## 1. Cocos Automation Layer Positioning

The Cocos Automation Layer is the future engine-operation layer behind Agent
Runtime and MCP Execution. It should translate approved AI Game Studio tasks into
controlled Cocos Creator Editor operations.

Architecture:

```text
Agent
  |
  v
Runtime
  |
  v
MCP
  |
  v
Cocos Automation
  |
  v
Cocos Creator
```

Relationship:

- Agent Runtime prepares task state, capability, context, permission, and proof
  expectations.
- MCP Execution Layer selects and invokes approved tools.
- Cocos Automation Layer performs approved Cocos-specific editor operations.
- Cocos Creator Editor remains the source of scene, prefab, asset, and runtime
  truth.

The Cocos Automation Layer must never bypass human approval, path scope, runtime
proof honesty, or rollback discipline.

## 2. Cocos Operation Capability Model

### Project Management

Capabilities:

- create project
- inspect project
- validate project structure

Project operations require explicit active project path approval and must not
create or inspect projects by default.

### Scene Automation

Capabilities:

- create scene
- modify scene
- manage nodes
- transform operations

Scene operations must use editor-safe workflows and must not raw-edit serialized
Cocos scene files.

### Asset Automation

Capabilities:

- import assets
- organize assets
- resource validation

Asset operations require source, owner, purpose, import scope, and rollback
expectations.

### Component Automation

Capabilities:

- attach components
- configure properties
- manage prefabs

Component operations must preserve bindings and report missing scripts,
components, or prefab references.

### Script Automation

Capabilities:

- create scripts
- modify scripts
- bind scripts

Script changes require approved file scope, validation commands, and runtime
proof when behavior changes.

### Animation Automation

Capabilities:

- animation clips
- state management
- character animation workflow

Animation automation must keep gameplay truth outside animation callbacks and
must require proof when animation affects visible runtime feedback.

### UI Automation

Capabilities:

- Canvas
- UI nodes
- layouts
- buttons
- panels

UI automation must preserve layout clarity, responsive visibility, and runtime
proof when browser or preview visibility matters.

### Build Automation

Capabilities:

- build
- validation
- platform packaging

Build automation requires explicit platform, output, config, and rollback
approval.

The proposal stage must list operations, files, scene paths, asset paths,
validation commands, proof target, rollback plan, and stop conditions before
execution.

## 4. Cocos MCP Command Boundary

Allowed after approval:

- controlled scene editing
- asset management
- script generation
- project inspection

Restricted:

- destructive operations
- deleting projects
- uncontrolled file changes
- broad asset imports
- raw text edits to Cocos serialized assets
- unapproved package, build, settings, or project changes
- cross-project inspection

Provider availability is not permission.

## 6. Large World / Map Automation

Large world automation may include:

- large scene handling
- chunk loading
- procedural generation
- memory optimization

Large map work requires map model selection, streaming or chunking strategy,
performance constraints, and validation proof before implementation.

## 7. Character and Animation Pipeline

Future pipeline areas:

- character asset generation
- skeleton workflow
- animation import
- state machine

Character and animation automation must separate character design, asset import,
animation state, and gameplay logic.

## 8. UI Design Automation

Future UI automation may include:

- UI style consistency
- layout generation
- resolution adaptation
- component reuse

UI automation should preserve design system rules, responsive layout, readable
text, and proof of runtime visibility.

## 9. Testing and Validation

Validation may include:

- build validation
- runtime testing
- performance checks
- asset consistency

Testing must report PASS, NEEDS_REPAIR, BLOCKED, or FAIL with evidence. Runtime
claims require approved proof.

## 10. Safety and Permission Model

Human approval checkpoints:

- active project path approval
- operation list approval
- file and asset scope approval
- scene or prefab scope approval
- Cocos/MCP provider approval
- build/platform approval
- runtime proof approval
- Git action approval

Operation logging should record:

- user request
- selected capability
- tool/provider
- approved path
- modified files
- generated files
- validation output
- warnings
- rollback status

Rollback strategy:

- rollback only approved changes
- avoid dangerous Git commands
- preserve user changes
- report partial rollback honestly

## 12. Explicit Limitations

This document creates:

- no Cocos automation implementation
- no MCP runtime
- no Agent runtime
- no autonomous execution
- no real project modification

It also creates no runtime files, no `agents/*` files, no protocol changes, no
examples, and no Cocos project files.
