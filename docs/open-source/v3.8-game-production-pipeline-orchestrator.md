# AI Game Studio v3.8 Game Production Pipeline Orchestrator

This document defines the Game Production Pipeline Orchestrator architecture for AI Game Studio.

This is architecture documentation only. It does not implement code, modify MCP, modify Cocos Creator projects, create Agent runtime behavior, modify `agents/*`, or change runtime execution logic.

## 1. Production Pipeline Goal

The Game Production Pipeline Orchestrator coordinates the full game production lifecycle from idea to release while keeping every stage scoped, validated, and human-approved.

Its goals are:

- Convert game ideas into staged production work.
- Keep idea, design, prototype, vertical slice, production, testing, optimization, and release work separated.
- Assign responsible Agents per stage through the Agent Registry.
- Request only the Capabilities needed for the current stage.
- Record approved decisions, validation results, and blockers into Project Memory.
- Require a Quality Gate before moving to the next stage.
- Preserve human approval before project modification, MCP execution, release actions, or scope expansion.

The orchestrator is a planning and governance layer, not an autonomous execution runtime.

## 2. Pipeline Stages

The production pipeline uses the following lifecycle:

```text
Idea
  -> Game Design
  -> Prototype
  -> Vertical Slice
  -> Production
  -> Testing
  -> Optimization
  -> Release
```

Each stage has a clear input, output, responsible Agent relationship, Capability requirement, Project Memory record, and Quality Gate.

## 3. Stage Inputs and Outputs

| Stage | Input | Output | Primary Agent Relationship | Required Gate |
| --- | --- | --- | --- | --- |
| Idea | User requirement, target platform, constraints, audience | Concept brief, genre hypothesis, initial risk list | Planner Agent, Game Designer Agent | Idea Gate |
| Game Design | User requirement, concept brief, selected genre | Game Design Document, MVP boundary, game loop, system list | Game Designer Agent, Planner Agent | Design Gate |
| Prototype | Game Design Document, MVP boundary, first proof target | Playable Demo, proof report, blocker list | Code Agent, Asset Agent, QA Agent | Prototype Gate |
| Vertical Slice | Playable Demo, approved slice scope | Integrated slice with core loop, UI direction, art direction, validation report | Planner Agent, Code Agent, Asset Agent, QA Agent | Vertical Slice Gate |
| Production | Vertical slice, task graph, approved roadmap | Implemented systems, scenes, assets, configs | Code Agent, Asset Agent, Planner Agent | Production Gate |
| Testing | Integrated features, validation requirements | QA report, bugs, review findings, acceptance status | QA Agent, Code Review Agent | Testing Gate |
| Optimization | QA report, performance findings, feedback history | Optimization plan, accepted fixes, updated validation result | QA Agent, Planner Agent, Code Agent | Optimization Gate |
| Release | Accepted build scope, release checklist, QA pass | Release notes, tag or release proposal, archive record | Release Agent, QA Agent, Planner Agent | Release Gate |

Example stage conversions:

- Game Design: user requirement -> Game Design Document.
- Prototype: Game Design Document -> Playable Demo.
- Testing: integrated feature build -> QA report.
- Release: accepted release scope -> release proposal.

## 4. Agent Participation Relationship

The orchestrator does not create Agents. It coordinates future Agent responsibilities through the Agent Registry and Multi-Agent Collaboration Protocol.

Recommended Agent participation:

- Planner Agent: owns stage planning, task graph creation, dependency tracking, and stage transition proposals.
- Game Designer Agent: owns game loop, system design, level design intent, and Game Design Document preparation.
- Code Agent: owns code task proposals and implementation planning after human approval.
- Asset Agent: owns asset requirement analysis, style constraints, and production-ready asset planning.
- QA Agent: owns validation requirements, stage gate checks, defect reports, and acceptance status.
- Code Review Agent: owns generated code review and feedback to the Code Generation Orchestrator.
- Release Agent: owns release readiness checks, release notes, tagging proposal, and release hygiene.

Handoff rule:

```text
Stage Output
  -> Agent Handoff Message
  -> Capability Requirement
  -> Human Approval
  -> Next Stage Proposal
```

No Agent may self-approve a high-impact transition. Human approval remains the final control point.

## 5. Capability Call Relationship

The pipeline requests Capabilities through the Capability Marketplace only when the current stage needs them.

Capability routing model:

```text
Pipeline Stage
  -> Capability Need
  -> Capability Marketplace
  -> Capability Version and Dependency Check
  -> Agent Registry Lookup
  -> Human Approval
  -> Execution Proposal
```

Example Capability usage:

- Idea: genre classification, scope estimation, risk analysis.
- Game Design: game template selection, gameplay loop design, MVP boundary planning.
- Prototype: code generation planning, minimal asset requirements, runtime proof planning.
- Vertical Slice: UI direction, core loop integration, scene assembly planning.
- Production: system implementation planning, asset pipeline planning, validation scheduling.
- Testing: scene validation, gameplay validation, code review, performance review.
- Optimization: performance analysis, feedback learning extraction, risk repair planning.
- Release: release checklist, CI verification, changelog and tag proposal.

Capabilities remain trigger-only and non-default. Loading a Capability must not imply Cocos/MCP access, project inspection, or runtime execution.

## 6. Project Memory Record Relationship

The Project Memory Engine records only approved project facts and reviewed stage results.

Recommended Project Memory records:

- Project identity: game name, genre, engine version, target platform.
- Design decisions: selected template, Game Design Document decisions, MVP scope.
- Task state: approved task graph, dependencies, stage status.
- Asset decisions: art style, UI rules, animation requirements.
- Validation history: proof results, QA reports, blockers, repairs.
- Release decisions: accepted release scope, release notes, tag records.
- Feedback learning: confirmed issues, root causes, accepted lessons.

Memory write rule:

```text
Pipeline Result
  -> Human Confirmation
  -> Project Memory Write Proposal
  -> Approved Memory Record
```

No project-specific fact may be stored without an approved project path, approved scope, and human confirmation.

## 7. Quality Gate Mechanism

Every stage must pass a Quality Gate before the orchestrator can propose the next stage.

Each Quality Gate includes:

- Entry criteria.
- Required outputs.
- Validation checks.
- Responsible Agent.
- Human approval state.
- Known blockers.
- Rollback or repair plan.

Recommended stage gates:

| Gate | Required Validation |
| --- | --- |
| Idea Gate | Concept is understandable, scope is not inflated, target genre is identified. |
| Design Gate | Game Design Document exists, MVP boundary is explicit, core loop is testable. |
| Prototype Gate | Prototype goal is narrow, proof result is recorded, blockers are known. |
| Vertical Slice Gate | Core loop, UI direction, art direction, and validation report are aligned. |
| Production Gate | Task graph is approved, dependencies are clear, risky work is isolated. |
| Testing Gate | QA report exists, known failures are classified, no false PASS is claimed. |
| Optimization Gate | Optimization work is measured, regressions are checked, changes are approved. |
| Release Gate | CI/release checks pass, release notes are accurate, no unsupported claims are made. |

Gate result format:

```text
Gate:
Status: PASS / NEEDS_REPAIR / BLOCKED
Stage:
Required outputs:
Validation result:
Human approval:
Blockers:
Next recommendation:
```

The next recommendation is optional advice only. It is not automatic approval to continue.

## 8. Human Approval Nodes

Human approval is required before:

- Advancing from one pipeline stage to the next.
- Modifying real project files.
- Calling Cocos/MCP tools.
- Inspecting real project files.
- Loading project-specific memory.
- Writing project-specific memory.
- Expanding scope beyond the approved stage.
- Committing, pushing, tagging, or creating a release.

Approval workflow:

```text
AI Proposal
  -> Human Review
  -> Approved Scope
  -> Execution or Validation
  -> Report
```

The orchestrator must stop after each approved stage or proof slice unless the user explicitly asks to continue.

## 9. Integration With Existing Architecture

The v3.8 orchestrator connects these architecture layers:

- v3.7 Game Design Intelligence: interprets user intent, game genre knowledge, gameplay rules, and design reasoning.
- v3.6 Game Template System: selects game type templates and required systems.
- v3.5 Capability Marketplace: provides Capability metadata, dependencies, version states, and loading rules.
- v3.4 Agent Registry: selects available Agents with matching roles and Capabilities.
- v2.10 Project Memory Engine: records approved project state, decisions, and validation history.

Integrated flow:

```text
User Requirement
  -> Game Design Intelligence
  -> Game Template System
  -> Game Production Pipeline Orchestrator
  -> Capability Marketplace
  -> Agent Registry
  -> Human Approval
  -> Stage Work Proposal
  -> Quality Gate
  -> Project Memory
```

This keeps the production lifecycle coherent without turning architecture documents into runtime automation.
