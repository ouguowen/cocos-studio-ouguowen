# v1.7 QA Agent Architecture

This document prepares the future QA Agent architecture for AI Game Studio. It
is documentation only. It does not implement an Agent, create runtime code,
execute MCP, automate Cocos, or modify real Cocos project files.

## 1. QA Agent Role Definition

The QA Agent is a future review and validation role. It should evaluate whether
an approved task meets scope, safety, proof, quality, and regression
expectations.

Responsibilities:

- review acceptance criteria
- check changed files against approved scope
- review validation output
- identify missing proof
- report blockers
- prepare bug reports
- give feedback to Planner, Designer, or Coder roles

The QA Agent cannot approve new scope, execute implementation, run Cocos/MCP by
default, or claim runtime proof without actual approved evidence.

## 2. Quality Evaluation Model

Quality evaluation should inspect:

- scope correctness
- forbidden file safety
- validation result
- runtime proof requirement
- user-facing behavior
- code maintainability
- asset consistency
- UI consistency
- performance risk
- rollback readiness

QA result states:

- PASS
- NEEDS_REPAIR
- BLOCKED
- FAIL

Each result must include concise evidence.

## 3. Gameplay Testing Strategy

Gameplay testing should focus on approved behavior only.

Checklist:

- expected player action exists
- expected feedback is visible
- objective or result state updates correctly
- no unapproved systems are introduced
- no hidden economy, inventory, gacha, save/load, combat, or progression scope is
  added
- runtime proof is captured when required

Gameplay testing must not expand into unapproved feature design.

## 4. Code Review Strategy

Code review should check:

- approved files only
- small and reversible changes
- no unrelated refactor
- no giant god scripts
- no unmanaged update loops
- no gameplay truth hidden in UI or animation callbacks
- no package, build, protocol, or settings changes unless approved
- no dangerous Git workflow

The QA Agent should report concrete findings and avoid rewriting code itself.

## 5. Asset Consistency Checking

Asset checks should verify:

- asset paths are approved
- imported assets have owners and purpose
- placeholder assets are labeled
- generated companion files are approved
- no external assets are introduced without approval
- Cocos `.meta` files are handled through approved rules

Asset checking does not replace Cocos import validation or runtime proof.

## 6. UI Consistency Checking

UI checks should verify:

- visible labels match the approved feature
- controls fit their containers
- feedback states are clear
- navigation is predictable
- text does not overlap
- UI state does not own gameplay truth
- browser/runtime visibility is proven when required

UI consistency should be judged against approved scope, not a new design
direction.

## 7. Performance Validation Strategy

Performance validation should identify risks:

- excessive per-frame work
- unmanaged dynamic loading
- memory leaks
- large unbounded lists
- asset loading spikes
- unnecessary context or runtime dependencies

For documentation-only or policy tasks, performance validation is normally not
applicable. For runtime tasks, performance proof requirements must be approved
before execution.

## 8. Bug Report Format

Recommended QA bug report:

```text
BUG_REPORT
title:
severity:
scope:
expected:
actual:
evidence:
files:
reproduction_steps:
suspected_cause:
repair_scope:
requires_approval:
```

Bug reports should be specific, reproducible, and limited to the approved task.

## 9. Planner Feedback Loop

QA should feed back to the Planner when:

- scope was unclear
- approval gates were missing
- validation commands were incomplete
- forbidden files appeared
- proof requirements were underspecified
- task decomposition was too broad

Planner feedback should improve the next proposal, not authorize automatic
continuation.

## 10. Human Approval Checkpoints

Human approval remains required before:

- repair work
- scope expansion
- reading real Cocos project files
- using Cocos Creator, Cocos MCP, browser preview, or runtime console
- changing assets, scenes, prefabs, metadata, package, build, CI, release, or
  protocol files
- staging, committing, pushing, tagging, or releasing

QA may recommend, but humans approve.

## 11. Future Runtime Roadmap

Possible future roadmap:

1. keep QA Agent architecture documentation-only
2. add validation coverage for QA result format
3. add read-only QA checklist command
4. add diff-scope checker
5. add validation-output summarizer
6. add runtime proof review after separate runtime architecture approval
7. add Cocos/MCP proof collection only after explicit engine automation review

## Safety Boundary

This document does not create:

- QA Agent implementation
- runtime code
- MCP execution
- Cocos automation
- bug tracking database
- real project test execution
- real Cocos project changes

The QA Agent remains future architecture only.
