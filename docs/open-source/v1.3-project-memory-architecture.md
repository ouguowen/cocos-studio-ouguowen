# v1.3 Project Memory Architecture

This document prepares the future project memory architecture for AI Game
Studio. It is documentation only. It does not create runtime code, databases,
Agent implementation, MCP integration, Cocos automation, or real Cocos project
changes.

## 1. AI Game Studio Memory Architecture

AI Game Studio memory should help the Skill remember approved project facts
without loading an entire project history into default context.

Memory is trigger-controlled:

- not loaded by default
- not written without approval
- scoped to an approved project path or repository
- separated from Skill maintenance docs
- reviewed when conflicts appear

Memory should support planning, design, implementation review, QA, and release
without becoming an uncontrolled source of truth.

## 2. Project Memory Model

Project memory records stable facts about a specific approved project.

Possible fields:

- project id
- project name
- approved project path
- owner
- active game type
- current milestone
- current stage
- current approved scope
- known constraints
- validation status
- last reviewed version

Project memory must not be created or updated unless the user explicitly
approves project memory access and the active project path.

## 3. Long-Term Memory

Long-term memory stores durable decisions and facts:

- game identity
- target platform
- selected engine version
- architecture decisions
- naming conventions
- approved folder structure
- accepted production rules
- release milestones

Long-term memory changes should be rare, reviewed, and versioned.

## 4. Task Memory

Task memory stores short-lived information for the current approved task:

- task id
- user request
- approved files
- forbidden files
- expected output
- validation commands
- blockers
- final status

Task memory should expire or archive after the task closes. It must not authorize
future work by itself.

## 5. Asset Memory

Asset memory records approved asset information:

- asset path
- asset type
- owner
- source
- usage
- import status
- placeholder or production status
- risk notes

Asset memory must not replace source-control truth or Cocos asset validation.
Generated or imported assets still need approval and proof.

## 6. Scene Memory

Scene memory records approved scene-level facts:

- scene path
- scene purpose
- required root nodes
- required components
- important bindings
- camera and canvas expectations
- validation proof status

Scene memory must not be treated as proof that the current Cocos scene is valid.
Runtime or editor proof is still required when behavior matters.

## 7. Character Memory

Character memory records approved character definitions:

- character id
- character name
- role
- gameplay purpose
- animation states
- UI representation
- asset references
- current approval status

Character memory should separate design identity from runtime implementation.

## 8. Animation Memory

Animation memory records approved presentation rules:

- animation id
- owner
- trigger
- state
- exit rule
- fallback behavior
- proof requirement

Animation memory must not own gameplay truth. Gameplay state must remain in
runtime systems, not animation callbacks alone.

## 9. UI Design Memory

UI design memory records approved interface decisions:

- screen or panel name
- purpose
- required controls
- data inputs
- feedback states
- navigation behavior
- accessibility notes
- validation status

UI memory should help preserve UX decisions without forcing UI implementation
before approval.

## 10. Memory Retrieval Strategy

Memory retrieval should be narrow:

1. load core Skill context
2. identify task type
3. ask whether memory is needed
4. load only the memory category relevant to the approved task
5. cite the memory source in the response

Do not load all memory categories by default. Do not infer memory access from a
`next recommended command`.

## 11. Context Optimization

Memory should reduce context weight, not increase it.

Rules:

- keep memory entries short
- prefer summaries over full transcripts
- store stable facts, not every conversation turn
- archive old task memory
- keep proof logs separate from default memory
- load memory by trigger and category

Fast Build Mode must stay lightweight.

## 12. Memory Conflict Resolution

When memory conflicts with files, user instructions, validation results, or
runtime proof:

1. stop and report the conflict
2. identify the conflicting sources
3. prefer current user instruction when safe
4. prefer current repository state over stale memory
5. prefer runtime/browser proof over remembered claims
6. update or retire memory only after approval

Never silently overwrite memory.

## 13. Version Tracking

Memory entries should include version markers:

- created date or release marker
- last reviewed date
- related commit hash when available
- approving human
- source artifact
- status: draft, active, superseded, retired

Version tracking helps prevent stale memory from becoming hidden authority.

## 14. Future Runtime Implementation Roadmap

Possible future implementation path:

1. keep memory architecture documentation-only
2. define memory templates and validation rules
3. add read-only memory inspection commands
4. add explicit memory write approval
5. add conflict detection
6. add CLI memory commands
7. integrate with Agent planning only after permission policy and audit coverage
8. add Cocos project memory only after separate Cocos runtime architecture review

## Safety Boundary

This document does not create:

- runtime code
- database storage
- Agent implementation
- MCP execution
- Cocos automation
- real project memory entries
- real Cocos project changes

Memory remains explicit, scoped, human-approved, and trigger-controlled.
