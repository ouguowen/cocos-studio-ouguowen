# v1.2 CLI Prototype Design

This document prepares the v1.2 CLI prototype design for AI Game Studio. It is
documentation only. It does not create runtime code, implement Agents, execute
MCP, automate Cocos, or modify real Cocos project files.

## 1. CLI Architecture

The future CLI should act as a controlled orchestration shell around the Skill,
not as an autonomous game builder.

Core responsibilities:

- parse user intent into structured commands
- route tasks to the correct planning, design, build, validation, or Agent
  status path
- check permissions before any read, write, runtime, Cocos/MCP, Git, or memory
  action
- produce execution proposals before high-impact actions
- preserve audit records for decisions, approvals, validation, and failures

The CLI prototype should start as a thin command interface. It should not embed
large Skill context, runtime logic, or Agent execution engines in v1.2.

## 2. Command Design

Future commands:

```text
cocos-studio init
cocos-studio design
cocos-studio build
cocos-studio validate
cocos-studio agent status
```

`cocos-studio init`:

- verifies the working directory
- identifies the Skill repository or approved Cocos project path
- checks Git status
- reports active mode and safety boundary
- does not create files unless explicitly approved

`cocos-studio design`:

- routes a user idea into scoped planning or design output
- loads only core and task-specific context
- returns design constraints, acceptance criteria, and next approval point

`cocos-studio build`:

- prepares a pre-write execution proposal
- lists files to create or modify
- lists forbidden files
- lists validation commands and rollback plan
- stops before writes until human approval

`cocos-studio validate`:

- runs approved validation commands
- reports PASS, NEEDS_REPAIR, or BLOCKED
- does not expand scope based on validation results

`cocos-studio agent status`:

- reports registered Agent policy state, capability state, and permission
  boundaries
- remains read-only by default
- does not activate Agents

## 3. Planner Agent Responsibility Definition

The first future Agent candidate is the Planner Agent.

Planner Agent responsibility:

- clarify user intent
- identify active project or repository boundary
- define allowed files and forbidden files
- select required context
- prepare pre-write proposals
- identify validation commands
- define stop conditions

Planner Agent cannot:

- write files by default
- inspect real Cocos projects without approval
- run Cocos/MCP by default
- execute implementation
- approve its own plan
- continue to Designer, Coder, or QA without explicit workflow approval

## 4. CLI Request Routing Flow

Future flow:

```text
Human Request
      |
      v
CLI Command Parser
      |
      v
Task Type Detection
      |
      v
Context Selection
      |
      v
Permission Check
      |
      v
Planner Proposal
      |
      v
Human Approval
      |
      v
Approved Execution or Validation
```

The routing flow must stop before any high-impact action. Approval is required
before file writes, real project inspection, memory updates, Cocos/MCP use,
runtime proof, Git staging, commits, pushes, tags, or releases.

## 5. Agent Message Format

Future Agent messages should use a compact structured format:

```text
task_id:
command:
sender:
receiver:
intent:
approved_scope:
forbidden_scope:
context_loaded:
requested_action:
permission_status:
validation_result:
stop_conditions:
audit_note:
```

Messages are task-local. They do not grant future permission, broaden scope, or
authorize autonomous continuation.

## 6. Human Approval Checkpoints

Human approval is required before:

- creating or modifying files
- deleting files
- reading project-specific memory
- writing project-specific memory
- inspecting real Cocos project files
- using Cocos Creator, Cocos MCP, browser preview, or runtime console
- modifying package, build, CI, settings, release, protocol, or Agent files
- staging, committing, pushing, tagging, or releasing
- activating an Agent workflow

Forbidden autonomous operations:

- self-approving execution
- continuing from one command to the next without approval
- using `next recommended command` as implicit approval
- changing scope after validation failure
- running Cocos/MCP because a provider is available

## 7. Context Loading Strategy

Core Context:

- minimal Skill boundary, operation mode, approval, and validation rules

Task Context:

- files directly named by the command or needed to complete the approved task

Optional Context:

- extended index, proof docs, examples, archive records, Agent governance docs,
  semantic deep docs, and runtime templates

Memory Context:

- project memory loaded only when explicitly triggered and approved

Priority:

1. Core Context
2. Task Context
3. Optional Context only when directly triggered
4. Memory Context only after explicit approval

The CLI must never load optional or memory context by default in Fast Build Mode.

## 8. Error Recovery

When an error occurs:

- stop the command
- report the failed stage
- report expected and actual result
- list changed files and unexpected files
- identify whether approval is required for repair
- propose the smallest rollback or retry path

Retry rules:

- retry only the failing stage
- do not broaden scope without approval
- do not use dangerous Git commands
- preserve user changes
- keep audit notes for all failures and repairs

## 9. Future Implementation Roadmap

Recommended v1.2 implementation path:

1. Define CLI command schema as documentation and tests.
2. Add a non-runtime CLI stub only after separate approval.
3. Add read-only `init`, `validate`, and `agent status` prototypes first.
4. Add Planner Agent proposal generation as documentation-backed output.
5. Add permission checks before any write-capable command.
6. Add audit log output for every command.
7. Add Cocos/MCP integration only after a separate runtime architecture review.

## Safety Boundary

This document does not implement:

- runtime code
- CLI executable behavior
- Agent execution
- MCP execution
- Cocos automation
- real project modification

Fast Build Mode remains preserved: this document is optional context and should
not load by default.
