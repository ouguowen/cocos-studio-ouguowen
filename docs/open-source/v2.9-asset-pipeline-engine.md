# AI Game Studio v2.9 Asset Pipeline Engine Architecture

This document defines the Asset Pipeline Engine layer for AI Game Studio. It is
documentation only. It does not create runtime implementation, MCP changes,
Cocos automation, Agent execution, examples, or real Cocos project changes.

## 1. Purpose

AI Game Studio needs an asset pipeline layer because game production assets are
not just generated images. A playable project needs assets that can be named,
organized, imported, referenced, animated, validated, and reused in Cocos
Creator.

Difference between AI generating images and production-ready game assets:

- AI generated images may be visually useful but still lack naming, resolution,
  slicing, animation states, prefab binding, and engine metadata.
- Production-ready game assets must fit the game's style, pipeline, file
  structure, Cocos import rules, component references, and validation needs.

Assets must satisfy:

- consistency
- naming rules
- resolution rules
- engine compatibility
- prefab requirements

## 2. Asset Pipeline Position

Architecture:

```text
Game Knowledge Engine
      |
      v
Asset Requirement Analysis
      |
      v
Asset Pipeline Engine
      |
      v
Asset Validation
      |
      v
Cocos MCP Adapter
      |
      v
Cocos Creator 3.8.8
```

The Asset Pipeline Engine translates game design needs into asset requirements
and validation expectations before any future Cocos import or binding operation.

## 3. Asset Requirement Analysis

Asset Requirement Analysis converts gameplay requirements into concrete asset
needs.

Example:

Tower Defense:

Gameplay:

- Enemy system

Required assets:

- enemy character
- enemy animation
- enemy prefab
- enemy effects

Tower system:

Required assets:

- tower model/sprite
- attack animation
- projectile effect
- upgrade icon

The analysis should identify what must be visible, animated, interactive,
reused, imported, or validated.

## 6. Asset Naming Convention

Recommended folders:

- `textures/`
- `characters/`
- `enemies/`
- `ui/`
- `effects/`
- `audio/`

Examples:

- `enemy_robot_idle.png`
- `enemy_robot_attack.png`
- `enemy_robot.prefab`

Naming should express asset role, subject, state, and type clearly enough for
humans, Agents, and future tooling to validate references.

## 7. Cocos Creator 3.8.8 Integration Rules

Asset import workflow should define:

- source path
- target Cocos path
- asset type
- owner
- usage
- placeholder or production status
- import validation

Prefab creation requirements:

- approved node structure
- required components
- required asset bindings
- prefab path
- dependency validation

SpriteFrame binding:

- texture imported as Cocos asset
- SpriteFrame available
- target Sprite component approved
- broken reference check

Animation clip binding:

- clip imported or created through editor-safe workflow
- target Animation component approved
- states and triggers named
- fallback behavior defined

Component references:

- all asset references must be approved
- script/component bindings must be validated
- generated metadata must remain in approved scope

## 8. Asset Validation Rules

Validation should detect:

- missing assets
- incorrect naming
- wrong resolution
- broken references
- inconsistent style

Validation should also report:

- unapproved external assets
- unexpected generated files
- missing metadata
- unused required assets
- placeholder assets used as production assets

## 9. Agent Integration

Future integration flow:

```text
Game Knowledge Engine
      |
      v
Asset Requirement Generator
      |
      v
Asset Pipeline Engine
      |
      v
Planner Agent
      |
      v
Execution Controller
      |
      v
Cocos MCP
```

Planner Agent should use asset requirements to create scoped proposals. The
Execution Controller should not import or bind assets without approval.
