# AI Game Studio v4.0 Release Automation System

This document defines the Release Automation System architecture for AI Game Studio.

This is architecture documentation only. It does not implement code, modify MCP, modify Cocos Creator projects, create Agent runtime behavior, modify `agents/*`, or change runtime execution logic.

## 1. Release Automation Goal

The Release Automation System gives AI Game Studio a controlled way to manage the game release process after production, testing, and runtime validation have passed.

Its goal is to help AI-assisted teams prepare builds, version records, release checklists, store materials, and final approval reports without bypassing human control.

Release automation should answer:

- Is the selected release scope approved?
- Are required builds identified?
- Is the version number correct?
- Are release blockers resolved or documented?
- Are store materials ready?
- Has runtime validation passed?
- Has a human approved the release action?

The Release Automation System is a release governance layer, not an autonomous publishing runtime.

## 2. Build Pipeline

The future build pipeline coordinates target-specific release preparation.

```text
Release Scope
  -> Build Target Selection
  -> Platform Build Plan
  -> Validation
  -> Release Candidate Report
  -> Human Approval
```

### Android Build

Android build preparation should define:

- Package identifier.
- Version code and version name.
- Signing requirement.
- Target device class.
- Asset compression requirement.
- Performance validation requirement.
- Store package output expectation.

Approval is required before generating release builds, changing project settings, modifying signing configuration, or uploading store artifacts.

### iOS Build

iOS build preparation should define:

- Bundle identifier.
- Version and build number.
- Signing and provisioning requirement.
- Device compatibility target.
- App Store metadata requirement.
- Runtime validation requirement.
- Export package expectation.

Approval is required before changing signing, build settings, provisioning, or App Store delivery materials.

### Web Build

Web build preparation should define:

- Web platform target.
- Compression and loading strategy.
- Hosting target.
- Browser compatibility requirement.
- Asset loading validation.
- Runtime preview validation.
- Deployment artifact expectation.

Approval is required before deployment, hosting configuration changes, or public release.

## 3. Version Management

Version management keeps the release state traceable.

Recommended version record:

- Product version.
- Build number.
- Target platform.
- Release channel.
- Commit hash.
- Release date.
- Runtime validation result.
- Known limitations.
- Human approval status.

Version flow:

```text
Release Candidate
  -> Version Proposal
  -> Changelog Review
  -> Build Target Mapping
  -> Human Approval
  -> Version Record
```

Version changes must not be automatic. Any package metadata, project settings, store metadata, or release tag changes require explicit approval.

## 4. Release Checklist

The release checklist verifies that the release candidate is safe to package or publish.

### Performance Check

Checks:

- Startup performance.
- Runtime frame stability.
- Memory warning signals.
- Loading time risk.
- Object creation or GC pressure risk.
- Platform-specific performance blockers.

### Asset Check

Checks:

- Required release assets exist.
- Asset names follow project rules.
- No broken references.
- Texture and audio sizes are acceptable.
- Store screenshots and icons are planned.
- No unapproved placeholder assets are presented as final.

### Error Check

Checks:

- Runtime validation has no critical errors.
- Console logs are reviewed.
- Known warnings are documented.
- Build errors are resolved.
- Missing references are absent or explicitly accepted.

### Version Check

Checks:

- Version number is correct.
- Build number is unique.
- Commit hash is recorded.
- Changelog matches the release scope.
- Release notes do not claim unsupported functionality.
- Target platform list is accurate.

Checklist result format:

```text
Release Checklist:
Status: PASS / NEEDS_REPAIR / BLOCKED
Target platform:
Version:
Build result:
Runtime validation:
Known blockers:
Human approval:
```

## 5. Store Material Generation

Store Material Generation prepares release-facing text and media requirements. It does not publish anything by itself.

### Game Summary

The game summary should include:

- Game title.
- Genre.
- Core gameplay loop.
- Target audience.
- Supported platform.
- Current release scope.

The summary must not overclaim features, automation, AI generation, or production completeness.

### Update Log

The update log should include:

- New features in the release.
- Bug fixes.
- Performance improvements.
- Known limitations.
- Compatibility notes.

Only validated and approved changes may appear in release notes.

### Screenshot Requirements

Screenshot planning should define:

- Required platforms and aspect ratios.
- Main gameplay screenshot.
- UI screenshot.
- Feature highlight screenshot.
- Store hero image requirement.
- Localization needs, if any.

Screenshots must reflect actual validated runtime behavior.

### Promotional Copy

Promotional copy should include:

- Short description.
- Long description.
- Feature bullets.
- Tagline candidates.
- Store keyword ideas.

Promotional copy must stay consistent with the approved release scope and must not claim unfinished systems.

## 6. Connection With Production Pipeline

The Release Automation System receives a candidate only after the Production Pipeline reaches the Release stage.

```text
Game Production Pipeline
  -> Testing
  -> Optimization
  -> Release Gate
  -> Release Automation System
  -> Human Release Approval
```

The Production Pipeline provides:

- Approved release scope.
- Stage validation results.
- Known limitations.
- Project Memory records.
- Final QA status.

Release Automation returns:

- Release checklist result.
- Build target plan.
- Version record.
- Store material checklist.
- Release readiness report.

## 7. Connection With Runtime Validation

Runtime Validation is a required input before release approval.

```text
Runtime Validation Report
  -> Release Checklist
  -> Release Readiness Decision
```

The Release Automation System consumes:

- Scene load result.
- Gameplay flow result.
- UI validation result.
- Performance risk summary.
- Error log summary.
- Human acceptance state.

A release cannot be marked ready if runtime validation is missing, stale, blocked, or inconsistent with the release claim.

## 8. Human Approval Flow

Human approval is required before:

- Creating a release build.
- Changing version metadata.
- Changing project build settings.
- Signing or exporting a platform build.
- Publishing store materials.
- Creating a release tag.
- Uploading artifacts.
- Declaring a release accepted.

Approval flow:

```text
Release Proposal
  -> Checklist Review
  -> Build Target Review
  -> Store Material Review
  -> Human Approval
  -> Release Action
  -> Release Report
```

The system may recommend the next release action, but a recommendation is optional advice only and never automatic approval.
