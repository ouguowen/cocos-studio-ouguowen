# AI Game Studio v2.6 Task Planning Prototype

This document defines the task planning layer between Planner Agent and MCP
Capability Registry. It is documentation only. It does not create runtime
implementation, change MCP, execute tools, create Agents, or allow autonomous
execution.

## 1. User Intent Parsing

The planning layer starts by turning a user request into structured intent.

Example:

```text
Create tower defense game
```

Parsed intent:

- game type: tower defense
- target: playable project plan or implementation proposal
- required systems: map, towers, enemies, waves, UI, validation
- approval needs: scope, project path, file boundaries, runtime proof
- risk: broad game creation request, requires decomposition

The Planner Agent must not jump directly from broad intent to implementation.
It must produce a task plan and approval checkpoint.

## 2. Task Plan Schema

Task plan fields:

- task id
- task type
- dependency
- required capability
- MCP mapping
- validation requirement

Example schema:

```text
task_id:
task_type:
dependency:
required_capability:
mcp_mapping:
validation_requirement:
approval_required:
stop_conditions:
```

The task plan is a proposal, not permission to execute.

## 3. Planner Agent Output Format

Example user request:

```text
Create player character
```

Planner output:

- design task: define character role, visual direction, and acceptance criteria
- asset task: identify required sprite, animation, collider, and prefab assets
- node task: create or select character root node
- component task: add Sprite, Animation, Collider, and script components
- prefab task: create reusable player character prefab
- animation task: define idle, move, hit, and death state requirements

Recommended output format:

```text
PLANNER_TASK_PLAN
user_request:
scope:
tasks:
dependencies:
required_capabilities:
mcp_mappings:
validation:
approval_checkpoint:
```

## 4. Execution Flow

Future flow:

```text
User
  |
  v
Planner Agent
  |
  v
Task Plan
  |
  v
Capability Registry
  |
  v
MCP Adapter
  |
  v
Cocos MCP
  |
  v
QA Agent
```

The flow must stop at approval checkpoints before MCP execution, Cocos project
modification, file writes, or runtime proof collection.

## 5. Human Approval Checkpoints

Human approval is required before:

- converting a plan into implementation
- reading or modifying real Cocos project files
- using Cocos MCP
- creating scenes, nodes, prefabs, assets, scripts, or animations
- changing project settings
- running browser preview or runtime proof
- staging, committing, pushing, tagging, or releasing

The Planner may recommend a next step, but recommendation is not approval.

## Safety Boundary

This document creates:

- no runtime implementation
- no MCP changes
- no autonomous execution
- no Cocos project changes
- no Agent implementation

It is documentation only.
