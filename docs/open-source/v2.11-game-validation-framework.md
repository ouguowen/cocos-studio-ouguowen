# AI Game Studio v2.11 Game Validation Framework Architecture

This document defines the Game Validation Framework architecture for AI Game
Studio. It is documentation only. It does not create runtime files, implement
Agents, implement MCP, modify Cocos projects, or authorize autonomous execution.

## 1. Positioning

The Game Validation Framework is the unified quality validation layer for AI
Game Studio game production.

It exists because AI-assisted game production can generate code, assets, scenes,
UI, and gameplay plans quickly, but speed without a shared validation standard
creates drift. A game can appear complete while still having broken references,
inconsistent assets, missing gameplay loops, unsafe code structure, poor UI
adaptation, or unproven runtime behavior.

The framework should provide common validation language across:

- planning
- design
- development
- asset production
- scene assembly
- gameplay proof
- QA review
- release readiness

Validation is not implementation. It checks whether approved work is complete,
safe, and honest.

## 2. Validation Domains

### 2.1 Code Validation

Code Validation checks whether implementation follows TypeScript and Cocos
Creator 3.8.8 development expectations.

Capabilities:

- TypeScript conventions
- Cocos Creator 3.8.8 development rules
- module dependency checks
- potential bug checks
- performance risk checks

Code validation should inspect:

- file scope matches approval
- no unrelated refactors
- no uncontrolled global state
- no giant god scripts
- no unmanaged `update()` loops
- no gameplay truth hidden only in UI or animation callbacks
- no unsafe dependency direction
- validation commands pass

### 2.2 Asset Validation

Asset Validation checks whether assets are production-usable, consistent, and
safe to import or reference.

Capabilities:

- image size rules
- file naming rules
- resource reference checks
- character style consistency
- UI asset consistency

Asset validation should inspect:

- missing assets
- wrong folder
- wrong naming
- wrong resolution
- unapproved external source
- broken Cocos metadata or references
- placeholder asset accidentally used as production art
- inconsistent character, scene, UI, effect, or icon style

### 2.3 Scene Validation

Scene Validation checks whether Cocos scene and prefab composition matches the
approved design and runtime needs.

Capabilities:

- Node hierarchy checks
- Prefab completeness
- Component configuration checks
- Missing Reference detection

Scene validation should inspect:

- required root nodes
- Camera and Canvas where needed
- required controllers
- bound script properties
- prefab dependencies
- unexpected missing references
- unexpected generated files
- hierarchy depth or naming drift

Scene validation must use editor-safe inspection in future automation and must
not rely on raw text editing of serialized Cocos assets.

### 2.4 Gameplay Validation

Gameplay Validation checks whether the approved playable behavior actually
exists.

Capabilities:

- core gameplay existence
- game loop completeness
- numerical reasonableness
- level flow checks

Gameplay validation should inspect:

- player action exists
- system response exists
- objective can progress
- result state is reachable
- level flow starts and ends
- numbers are placeholder-labeled or justified
- rewards, economy, combat, or progression systems are not added without
  approval
- runtime proof exists when behavior matters

### 2.5 UI Validation

UI Validation checks whether the interface is usable, consistent, and visible.

Capabilities:

- layout checks
- resolution adaptation
- interaction flow
- style consistency

UI validation should inspect:

- text fits container
- controls are readable
- touch targets are usable
- layout adapts to target resolution
- visual state matches gameplay state
- interaction flow is predictable
- browser/runtime visibility is proven when required

## 3. Relationship With Existing Systems

### QA Agent

QA Agent is the primary consumer of the Game Validation Framework. It uses the
framework to produce PASS, NEEDS_REPAIR, BLOCKED, or FAIL with evidence.

### Project Memory Engine

Project Memory Engine stores approved validation history, known issues, prior
decisions, and regression risks. It does not replace current validation.

### Game Knowledge Engine

Game Knowledge Engine defines genre and system expectations. Game Validation
Framework checks whether the produced game work satisfies those expectations.

### Asset Pipeline Engine

Asset Pipeline Engine produces asset requirements and asset validation targets.
Game Validation Framework checks asset completeness, consistency, references,
and production readiness.

### Planner Agent

Planner Agent uses the framework to define validation requirements before work
starts. It should include validation scope in every task plan.

### Execution Controller

Execution Controller uses validation checkpoints to decide whether a task can
move from completed to verified. It must stop if validation fails.

## 4. Validation Pipeline

Pipeline:

```text
Requirement
      |
      v
Design
      |
      v
Development
      |
      v
Asset Generation
      |
      v
Scene Assembly
      |
      v
Validation
      |
      v
Feedback Repair
```

Pipeline rules:

- validation requirements should be planned before implementation
- each stage should produce reviewable output
- failed validation should stop automatic continuation
- repair scope must be narrow and approved
- validation history should be recorded when project memory is approved

## 5. Future Extensions

Automated Playtest:

- simulate approved gameplay flows
- detect blocked objectives
- verify result states

AI Player Simulation:

- test common player paths
- identify confusing interactions
- inspect difficulty spikes

Visual Consistency Check:

- compare style, palette, UI spacing, asset naming, and resolution
- detect visual mismatch between screens or asset groups

Performance Benchmark:

- measure frame time
- inspect memory growth
- detect excessive dynamic loading
- compare target platform constraints

All future extensions require separate approval and validation before runtime
implementation.

## 6. Safety Constraints

This document creates:

- no runtime implementation
- no MCP changes
- no autonomous execution
- no Agent implementation
- no Cocos project changes
- no validation runtime
- no automated playtest implementation

The Game Validation Framework is future architecture only. Human approval remains
required for runtime execution, MCP access, Cocos automation, memory writes, Git
actions, and release actions.
